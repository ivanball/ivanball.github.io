23. ADC Identity Module (Users, Profiles, GDPR Export/Erasure)

What this chapter covers. This is the Identity bounded context of MMCA.ADC, end to end and across every layer: the User aggregate that is the conference's notion of a person; the use cases that mutate it (register, log in, refresh/revoke tokens, change password, change preferences, delete + erase); the read paths (admin user list, GDPR data export, the cross-module attendee-id query); the EF persistence and seeding; the REST/OAuth/contracts/gRPC surface that lets the module run as its own microservice; and the Blazor profile/user-management UI. It is the leaf of ADC's module graph: IdentityModule declares no Dependencies, so it is a leaf in the module dependency graph (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.API/IdentityModule.cs:13), yet almost every other module ultimately depends on the identity it issues. Tokens authorize Conference and Engagement calls, and the UserId it mints is the scalar foreign key that bookmarks, notifications, and speaker links all point at. The module is a textbook vertical slice of the whole architecture, so this chapter doubles as a worked example of the patterns taught in the primer. [Rubric §3, Clean Architecture] [Rubric §5, Vertical Slice] [Rubric §4, DDD]

A note on how the actual authentication mechanics are split. The Identity module owns the User aggregate and the ADC-specific application workflow, but the reusable auth machinery it leans on now lives almost entirely in MMCA.Common: JWT minting/validation (ITokenService), password hashing (IPasswordHasher), login-lockout (ILoginProtectionService), the shared login/register/refresh/revoke orchestration (AuthenticationServiceBase<TUser>), the AuthControllerBase and OAuthControllerBase REST shapes, and JWKS publishing. This chapter focuses on the ADC-specific half: the domain model, the ADC hooks and use cases, and how Common's auth ports are consumed. When you follow a link into group 8 or group 12, you are looking at an injected port or an inherited base class, not Identity code.

The User aggregate, one entity, many hats

Everything orbits User (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.Domain/Users/User.cs:17), a sealed AuditableAggregateRootEntity<UserIdentifierType> (UserIdentifierType = int) that also implements IAnonymizable and Common's IAuthUser. It carries far more than credentials: the canonical email + first/last name (each tagged [Pii] for the privacy machinery, ADR-005, User.cs:20,24,28), the password hash/salt as byte[], the Role string, the refresh-token lifecycle, optional MAUI device metadata, the optional external-OAuth provider/key pair, the optional LinkedSpeakerId that bridges into the Conference context (BR-207/208/209, User.cs:53), and the per-user PreferredCulture/PreferredTheme UI preferences (User.cs:83,86, ADR-027/ADR-028). Two construction paths exist as Result-returning factories: User.Create(...) for local sign-up (User.cs:137) and User.CreateExternal(...) for OAuth-first accounts with empty credentials (User.cs:179); both validate through UserInvariants and Result.Combine, so an invalid user can never be constructed (factory-returns-Result).

The aggregate's behavior is expressed as intention-revealing mutators rather than public setters: ChangePassword (which raises UserPasswordChanged, User.cs:297), UpdateRefreshToken/RevokeRefreshToken, LinkSpeaker/UnlinkSpeaker, LinkExternalProvider, UpdateDeviceMetadata, UpdatePreferences (the allowlist-validated culture/theme setter, User.cs:262), and an override of Delete() that revokes the refresh token and raises UserDeleted (User.cs:329, event at User.cs:335). The most subtle member is Anonymize() (User.cs:352): it overwrites every PII field in place with placeholders, including a deleted-{Id}@anonymized.invalid email crafted to keep the unique-email invariant (BR-200) holding across many erased accounts, and it is idempotent (a second call short-circuits to a no-op success by comparing the current email to the placeholder, User.cs:364). This is the entity-level half of the soft-delete-versus-erasure split (ADR-005) [Rubric §30, Compliance/Privacy]: Delete() flips IsDeleted for lifecycle, Anonymize() scrubs the data for a right-to-be-forgotten request, and they are deliberately separate operations.

Write use cases, the CQRS slices

Each mutating workflow is a CQRS slice (command/query record + handler), wrapped by the FeatureGate to Logging to Caching to Validating to Transactional decorator pipeline taught in group 5. The richest collaborator is AuthenticationService (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.Application/Users/AuthenticationService.cs:22), and it is worth being precise about where its logic lives, because it moved. The shared login/register/refresh/revoke workflow, the validate-first ordering, the dual-fetch pattern, ADR-029 lockout and rate limiting, and BR-205/206 refresh-token rotation with reuse detection, now all live in Common's AuthenticationServiceBase<TUser>; the ADC subclass supplies only the ADC-specific hooks (AuthenticationService.cs:30). Those hooks are worth internalizing because they show how the generic base is specialized:

  • The untracked lookup. FindUntrackedByEmailAsync (AuthenticationService.cs:34) fetches the user untracked so the base can verify the password cheaply, and EmailExistsAsync (AuthenticationService.cs:46) runs with ignoreQueryFilters: true so a soft-deleted (erased) account's email cannot be re-registered. Soft-deleted accounts are invisible to the normal query filter, so a login against one returns the same generic failure as a missing account, no account-enumeration leak [Rubric §11, Security].
  • The Attendee default and the speaker_id claim. CreateUser (AuthenticationService.cs:50) fixes the default role to UserRole.Attendee (BR-45), and CreateAccessToken (AuthenticationService.cs:60) folds in the private SpeakerClaims helper (AuthenticationService.cs:164) so a linked speaker's id rides the JWT as a speaker_id claim (BR-209).
  • The post-commit registration event. OnUserRegisteredAsync (AuthenticationService.cs:70) publishes UserRegistered through IIntegrationEventPublisher after the base has committed, so the EF-generated UserId is real (AuthenticationService.cs:72-74). ExternalLoginAsync (AuthenticationService.cs:88) is the one workflow the subclass implements outright: find by provider+key, else link the external provider onto an existing local account by email match, else create a fresh external Attendee, then publish the same UserRegistered for brand-new accounts (AuthenticationService.cs:146-151).

The other write slices are smaller and self-contained: ChangePasswordHandler verifies the current password before hashing the new one (ChangePasswordHandler.cs:29); DeleteUserHandler enforces an owner-or-Organizer check (DeleteUserHandler.cs:25), then calls user.Delete() and user.Anonymize() in one transaction to honor the PRIVACY.md §5 "erase within 30 days" promise (DeleteUserHandler.cs:40,47); their commands (ChangePasswordCommand, DeleteUserCommand) carry the acting user's id and role in the record so authorization is testable without mocking ambient context.

Per-user preferences: culture and theme that follow the account

A returning user's language and theme are durable, per-account state rather than a device-local cookie. Two slim CQRS slices back this, both dispatched by AuthController (not the user-management controller): ChangePreferencesHandler loads the User and calls UpdatePreferences (User.cs:262), persisting PreferredCulture and PreferredTheme (User.cs:83,86); a null field on the ChangePreferencesRequest is coalesced to the stored value (ChangePreferencesHandler.cs:27-29) so the app-bar culture switcher and the theme toggle can each save their own field without clobbering the other. The command (ChangePreferencesCommand) is ICacheInvalidating keyed on the user-cache prefix, so the next read is fresh. GetUserPreferencesHandler is the read side, projecting the two columns into a UserPreferencesResponse (GetUserPreferencesHandler.cs:22) that the UI reads at login to re-apply a choice across devices. These are the entity-side of the en-US/es internationalization (ADR-027) and the light/dark theme (ADR-028) work [Rubric §27, Internationalization]: culture and theme become a stored, data-subject preference on the aggregate. UpdatePreferences validates both against an allowlist before persisting (UserInvariants' EnsurePreferredCultureIsValid/EnsurePreferredThemeIsValid, User.cs:265-266), and Identity's edge error messages are localized through IdentityErrorResources, a resource-anchor type whose .resx siblings are keyed by domain error Code (for example "User.Email.Empty") and resolved by Common's IErrorLocalizer once the host calls AddErrorResources<IdentityErrorResources>(). The AddUserPreferences EF migration that adds the two columns exists in the Identity migrations project (MMCA.ADC/Source/Hosting/MMCA.ADC.Migrations.SqlServer.Identity/Migrations/20260627221640_AddUserPreferences.cs) and UserConfiguration maps both (UserConfiguration.cs:100,104).

Read use cases, list, export, and the cross-module attendee query

Reads are pure query handlers (no SaveChanges). GetUsersHandler backs the organizer-only admin grid (BR-51), returning a filtered/sorted/paged PagedCollectionResult<UserListDTO> projected to the deliberately slim UserListDTO. It pushes COUNT, ORDER BY, OFFSET/FETCH, and the projection down to SQL through the read repository's IQueryable, deliberately never materializing PasswordHash, PasswordSalt, RefreshToken, or the device fields (GetUsersHandler.cs:29-52), with a server-side page-size cap of 500 (BR-11, GetUsersHandler.cs:25). ExportUserDataHandler implements the GDPR/CCPA data-subject access request (PRIVACY.md §7): owner-or-Organizer authorization (ExportUserDataHandler.cs:23), then a projection of the user's Identity-owned PII into UserDataExportDTO that deliberately omits secrets (password hash/salt, refresh token, provider key) while including device fingerprints, because those are personal data [Rubric §30, Compliance/Privacy]. The two DTOs diverging on purpose, export-everything versus list-the-minimum, is privacy-by-design enforced at the type level.

A third read is the cross-module seam: IAttendeeQueryService (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.Shared/Users/IAttendeeQueryService.cs:8) declares "give me every active Attendee's user id," and lives in Identity.Shared so the Notification module can call it without referencing Identity's domain or infrastructure [Rubric §1, SOLID] (Dependency Inversion). In-process it is satisfied by AttendeeQueryService (a read-repository projection, AttendeeQueryService.cs:11); when Identity is disabled in a host, IdentityModule.RegisterDisabledStubs (IdentityModule.cs:19) registers the no-op DisabledAttendeeQueryService so the container still resolves. This is the same "shared contract + swappable implementation" shape that makes service extraction reversible.

Eventual consistency: how Identity and Conference stay in sync

The User to Speaker link (BR-207/209) is the module's headline distributed-systems story, and it is fully asynchronous across the database-per-service boundary (ADR-006): there are no cross-database foreign keys, only scalar ids reconciled by integration events [Rubric §6, CQRS & Event-Driven] [Rubric §7, Microservices Readiness]. After a new account commits, AuthenticationService.OnUserRegisteredAsync publishes UserRegistered after the transaction, so the EF-generated UserId is real (AuthenticationService.cs:70-76). That event flows through the outbox (ADR-003) to Conference's UserRegisteredHandler, which tries to auto-link a speaker by email match. Conference then publishes SpeakerLinkedToUser back, and Identity's SpeakerLinkedToUserHandler (and its SpeakerUnlinkedFromUser sibling) sets/clears User.LinkedSpeakerId through one consistent code path, calling user.LinkSpeaker(...) (SpeakerLinkedToUserHandler.cs:48). The practical consequence: the JWT returned at registration does not yet carry a speaker_id claim, it appears on the next token refresh after the link round-trips. The handlers are idempotent (they skip if already pointing at the same speaker, SpeakerLinkedToUserHandler.cs:43) because broker delivery is at-least-once.

The REST + OAuth API surface

Three controllers expose the module, all inheriting Common base classes. AuthController (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.API/Controllers/AuthController.cs:24) extends AuthControllerBase for login/register/refresh/revoke, overriding RegisterAsync to capture the client IP for rate limiting (BR-213, AuthController.cs:41-52) and adding PUT /auth/password (dispatches ChangePasswordCommand, AuthController.cs:72) plus the PUT/GET /auth/preferences pair (the per-user culture/theme endpoints described above, AuthController.cs:98,124), all straight through the decorator pipeline; login advertises a 429 for lockout (BR-212, AuthController.cs:55). UsersController (UsersController.cs:28) hosts the organizer list, the {userId}/export GDPR endpoint, and DELETE {userId}, each inheriting ApiControllerBase.HandleFailure so a Result.Failure becomes the right RFC-9457 status [Rubric §9, API & Contract Design]; the list is gated by a [HasPermission(IdentityPermissions.UsersRead)] attribute (UsersController.cs:37). UserClaimsController echoes the caller's JWT claims back, useful for the UI to read role/speaker_id.

OAuthController (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.API/Controllers/OAuthController.cs:20) handles the Google/GitHub flows, and here too the security-careful machinery lives in the base class. It is a thin subclass of OAuthControllerBase (Common) that supplies only the auth/oauth routing and API version; the base implements the challenge to provider-callback to complete to single-use-code-exchange flow where the token pair is stashed server-side and the browser carries only an opaque, first-use-burned code, so access/refresh tokens never appear in a redirect URL, browser history, the Referer header, or upstream access logs [Rubric §11, Security] [Rubric §26, Front-End Security]. The provider schemes themselves are wired by ExternalAuthExtensions (AddExternalAuthProviders, in MMCA.Common.API), gated per provider on OAuth:<Provider>:ClientId in each service host's Program.cs. The actual find-or-create/link logic lives in AuthenticationService.ExternalLoginAsync (AuthenticationService.cs:88).

Persistence and seeding

The Identity slice owns its own SQL database (ADC_Identity, ADR-006). UserConfiguration (UserConfiguration.cs:12) inherits Common's EntityTypeConfigurationSQLServer<User, UserIdentifierType> (engine-agnostic entities). Note the mapping particulars: Email is a converted scalar via HasConversion (an Email value object to/from a flat nvarchar, UserConfiguration.cs:20-26), not an owned nested type; the hashes fall through to EF's default varbinary(max) for byte[] (UserConfiguration.cs:41,43); email carries a unique index (UserConfiguration.cs:112); and both the LinkedSpeakerId 1:1 link and the (LoginProvider, ProviderKey) pair get filtered unique indexes that permit many nulls (UserConfiguration.cs:117,121). ModuleApplicationDbContext is the module's design-time/seeding context handle. Two seeders cooperate: the infrastructure-level IdentityModuleDbSeeder does the actual idempotent insert of the bootstrap Organizer/Attendee accounts, and the API-level IdentityModuleSeeder is an IModuleSeeder that the module invokes during startup, building and delegating to the infrastructure seeder (IdentityModuleSeeder.cs:22) once an IPasswordHasher is available to hash the seed passwords.

Composition and service extraction

IdentityModule is the IModule discovered by the ModuleLoader in topological order; its Register (IdentityModule.cs:23) delegates to AddIdentityModule(...), whose per-layer DependencyInjection helpers (Application, Infrastructure, API, UI, Contracts, all extension(IServiceCollection) blocks) register handlers, validators, EF configs, and the in-process AttendeeQueryService. Because the module talks only to abstractions, it lifts out cleanly into MMCA.ADC.Identity.Service, the standalone host that boots only this module. Its AttendeesGrpcService (MMCA.ADC/Source/Services/MMCA.ADC.Identity.Service/Grpc/AttendeesGrpcService.cs:19) exposes IAttendeeQueryService over the wire (ADR-007), and the companion MMCA.ADC.Identity.Contracts project ships the typed-client side: AttendeeQueryServiceGrpcAdapter implements the same C# interface on top of the generated gRPC client (AttendeeQueryServiceGrpcAdapter.cs:14), and the contracts DependencyInjection.AddIdentityAttendeeClient uses Replace (not TryAdd, DependencyInjection.cs:47) to swap whatever IAttendeeQueryService is registered, real impl or disabled stub, for the gRPC adapter at the composition root after ModuleLoader.DiscoverAndRegister. Identity is also the JWKS authority: it signs JWTs with RS256 and publishes its public key at /.well-known/jwks.json, which the other services fetch through the YARP Gateway to validate tokens with no shared secret (ADR-004/008) [Rubric §7, Microservices Readiness]. The same application code runs in-process or as a service with no rewrite because every collaboration is behind a port resolved at the edge.

The UI side

Profile and user-management screens are authored once as Razor components in IdentityUIModule (MMCA.ADC/Source/Modules/Identity/MMCA.ADC.Identity.UI/IdentityUIModule.cs:13), an IUIModule that contributes "My Profile" (all users) and "Users" (Organizer-only) nav items (IdentityUIModule.cs:15-19) via IdentityRoutePaths. The Profile page (Profile.razor.cs:13) lets a user change their password and delete their own account; the UserList page is the organizer grid. Both talk to the Gateway through UserService (the HTTP adapter behind IUserUIService, UserService.cs:14), which derives from Common's AuthenticatedServiceBase to attach the bearer token and a Polly retry policy. Because these are shared Razor Class Libraries, the same pages render across Web, Android, iOS, macOS, and Windows (write-once UI) [Rubric §18, UI Architecture].

In short: Identity is a small module by type-count but a complete demonstration of the codebase's spine, a DDD aggregate with Result factories and domain events, ADC-specific hooks over a shared Common auth workflow, CQRS slices behind a decorator pipeline, soft-delete plus GDPR anonymization, event-driven cross-context linking over an outbox, JWKS-based cross-service auth, gRPC-backed extraction, and a write-once Blazor UI, all anchored on the one entity the whole conference revolves around.

ChangePreferencesRequest

    • Device metadata (:61-88): seven optional fields (DeviceId, DeviceFormFactor, DevicePlatform, DeviceModel, DeviceManufacturer, DeviceName, DeviceType), each HasMaxLength(UserInvariants.DeviceFieldMaxLength) and non-Unicode; these are MAUI-only (BR-201/BR-202).

    • UI preferences (:99-106): PreferredCulture and PreferredTheme, each a nullable 10-char non-Unicode column; null means "follow the request or OS default" (ADR-027 for culture, ADR-028 for theme).
    • Ignored computed properties (:108-110): FullName and IsExternalLogin are builder.Ignore(...), so neither gets a column.
    • Four indexes (:112-123): unique on Email (fast, unambiguous login lookup); a filtered unique index on RefreshToken WHERE [RefreshToken] IS NOT NULL (one live token per user without colliding on many nulls); a filtered unique index on LinkedSpeakerId WHERE … IS NOT NULL (enforcing the 1:1 User to Speaker link); and a filtered composite unique index on (LoginProvider, ProviderKey) where both are non-null (one external identity per provider key pair).
  • Why it's built this way: filtered unique indexes let a nullable credential column stay unique only among the rows that have a value, which is exactly the "at most one active refresh token / one linked speaker / one external identity" rule expressed in SQL. Reading lengths from UserInvariants is the [Rubric §16, Maintainability] single-source-of-truth move. The PreferredCulture/PreferredTheme columns are what make a user's i18n and theme choice durable server-side ([Rubric §27, Internationalization]).
  • Where it's used: discovered by the Identity module's persistence assembly scan and applied to the SQLServerDbContext model; its columns and indexes back the migrations in the Identity migrations project.

[Rubric §11, Security] is the dominant category (this is the credential path), joined by [Rubric §2, Design Patterns] and [Rubric §1, SOLID] (open-closed via a template base). The base class fixes the algorithm of each flow (validate first, ADR-029 lockout and rate limits, the dual-fetch read pattern, BR-205/BR-206 refresh rotation with reuse detection) and calls back into protected abstract/virtual hooks for the parts that differ per app. This subclass fills those hooks; it does not re-implement LoginAsync/RegisterAsync/RefreshTokenAsync/RevokeTokenAsync.   • Where it's used: called from the Notification service host's Program.cs (the consumer of attendee ids) once Identity runs as its own service.
