10. Notifications (Push + In-App Inbox + Email)

What this chapter covers. This group is the codebase's notification subsystem: the delivery channels that let the backend reach a user out-of-band, and the seams that keep those channels domain-agnostic and independently extractable. There are four cooperating mechanisms, and they split along a durability line:

  1. Push, a broadcast recorded for audit and delivered in real time to connected browsers over a SignalR WebSocket hub.
  2. In-app inbox, a durable, per-user record so a recipient who was offline (or who closed the tab) can still see what they missed, with read/unread state and an unread-count badge.
  3. Email, a thin SMTP sender port for system/admin mail.
  4. Live channels, an ephemeral group fan-out over the same hub (poll results, new questions on the conference-day live layer) that is deliberately not persisted: a client that is not connected and subscribed at publish time simply never sees the event (ADR-039).

Channels 1 to 3 are the "durable" story; channel 4 is the "transient" story, and the two sit behind different application ports on purpose. The whole subsystem is split across the layers you met in the primer: the Domain owns two small aggregates (PushNotification and UserNotification) plus the PushNotificationCreated event and the PushNotificationStatus enum; the Application layer defines the ports (IPushNotificationSender, ILiveChannelPublisher, IEmailSender, INotificationRecipientProvider) and the CQRS slices that orchestrate them; the Infrastructure layer supplies the adapters (SignalRPushNotificationSender, SignalRLiveChannelPublisher, SmtpEmailSender, NotificationHub) plus no-op fallbacks; the API layer exposes two REST controllers; and the ADC application contributes the app-specific recipient resolution (AttendeeNotificationRecipientProvider), a gRPC transport for live channels (LiveChannelPublisherGrpcAdapter, LiveChannelGrpcService), and the thin module host (NotificationModule). It is a compact but complete vertical slice of the Clean Architecture / CQRS / port-and-adapter story [Rubric §3, Clean Architecture] [Rubric §5, Vertical Slice].

The push and inbox pair is governed by ADR-024 (two-channel user notifications): one application use case (SendPushNotificationHandler, MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/PushNotifications/UseCases/Send/SendPushNotificationHandler.cs:17) writes the durable per-user UserNotification inbox (read/unread) and fires a transient SignalR push, with the transport behind IPushNotificationSender (a no-op NullPushNotificationSender default, swapped for the SignalR sender by AddPushNotifications with an optional Redis backplane) and the audience behind INotificationRecipientProvider. Push failure is non-fatal because the inbox is the source of truth, and in ADC this whole subsystem runs as a dedicated MMCA.ADC.Notification.Service built on exactly these seams.

How "send a push notification" works end-to-end

Sending is the most instructive flow, because it touches every layer and three different consistency concerns (durable audit, per-user inbox, best-effort real-time delivery). It is driven by SendPushNotificationHandler (MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/PushNotifications/UseCases/Send/SendPushNotificationHandler.cs:17), which an organizer reaches by POST-ing to NotificationsController (MMCA.Common/Source/Presentation/MMCA.Common.API/Controllers/Notifications/NotificationsController.cs:31):

  1. Resolve recipients. The handler does not know who the audience is, it asks the injected INotificationRecipientProvider for the user IDs (SendPushNotificationHandler.cs:30-31). This is the key seam: MMCA.Common ships a no-op NullNotificationRecipientProvider (returns an empty list), and the consuming app overrides it. ADC registers AttendeeNotificationRecipientProvider (MMCA.ADC/Source/Modules/Notification/MMCA.ADC.Notification.Application/AttendeeNotificationRecipientProvider.cs:10), which delegates to the Identity module's IAttendeeQueryService to mean "every attendee" (AttendeeNotificationRecipientProvider.cs:14-16). An empty recipient set short-circuits to a Validation failure (SendPushNotificationHandler.cs:33-39), so the framework stays domain-agnostic while the app decides scope.
  2. Persist the audit aggregate. PushNotification.Create(...) (MMCA.Common/Source/Core/MMCA.Common.Domain/Notifications/PushNotifications/PushNotification.cs:56) validates title/body via PushNotificationInvariants (PushNotification.cs:62-64) and returns a Result<PushNotification> (the Result pattern, factory-method-returns-Result). It is saved through the IUnitOfWork repository with Status = Pending and a snapshot RecipientCount (SendPushNotificationHandler.cs:42-55).
  3. Fan out the inbox rows. For every recipient the handler creates a UserNotification row (SendPushNotificationHandler.cs:58-63), the durable per-user inbox entry that survives a missed real-time delivery. This is the "inbox" channel: storage first, delivery second.
  4. Deliver in real time (best-effort). The handler calls IPushNotificationSender.SendToUsersAsync (SendPushNotificationHandler.cs:70-74). The delivery is wrapped in a try/catch that swallows any exception by design (the CA1031 suppression at :79-81): a broker/SignalR hiccup must not lose the already-persisted inbox rows. On success it calls notification.MarkAsSent() (:76), on failure notification.MarkAsFailed() (:83), then saves the status once more (:87). So PushNotificationStatus is an observed-delivery audit field, not a gate. [Rubric §29, Resilience], delivery is intentionally non-fatal; durability lives in the database, not the WebSocket.
  5. Return the DTO. The saved aggregate is mapped by PushNotificationDTOMapper (a Mapperly source-generated mapper, ADR-001; MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/PushNotifications/DTOs/PushNotificationDTOMapper.cs:17) and returned as 201 Created (NotificationsController.cs:56).

Notice the ordering: audit and inbox are persisted before delivery is attempted, and delivery failure only updates a status flag. That is the design's answer to "what if the user is offline?", they read it later from the inbox, and the organizer sees a Sent/Failed history. The PushNotificationCreated domain event is raised inside Create (PushNotification.cs:75), but delivery does not wait on it (see the closing note).

The SignalR hub and its two delivery adapters

The real-time surface centers on NotificationHub (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Hubs/NotificationHub.cs:17), a single [Authorize]d SignalR Hub (:16) that serves both transient channels. The hub stays thin, but it is no longer method-free: it publishes the client-facing method-name constants (ReceiveNotificationMethod = "ReceiveNotification", :20, and ReceiveChannelEventMethod = "ReceiveChannelEvent", :23) and it hosts two callable hub methods, JoinChannelAsync (:44) and LeaveChannelAsync (:55), which add/remove the calling connection to a SignalR group keyed by a channel string (for example event:1 or session:123). Each join/leave validates the channel key against a configured regex (PushNotificationSettings.ChannelKeyPattern, cached per host, :61-71) and throws a HubException on a bad key, so a client cannot subscribe to arbitrary group names. Per-user connection mapping is handled by SignalR's IUserIdProvider (a ClaimBasedUserIdProvider registered by AddPushNotifications).

Two adapters push out-of-band through IHubContext<NotificationHub>, neither holding a hub connection itself. SignalRPushNotificationSender (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/SignalRPushNotificationSender.cs:13) addresses Clients.User(id) / Clients.Users(batch) / Clients.All and invokes ReceiveNotificationMethod; large audiences are chunked into batches of 100 user IDs (:15, :42-59) so one send does not overwhelm the connection manager [Rubric §12, Performance & Scalability]. SignalRLiveChannelPublisher (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/SignalRLiveChannelPublisher.cs:12) instead addresses Clients.Group(channelKey) and invokes ReceiveChannelEventMethod (:15-19). The browser side of both (the NotificationHubService, the channel listeners, and the bell UI) lives in group 15, Common UI framework.

The live-channel path (ADR-039), transient by design

Live channels are the ephemeral counterpart to push. The port is ILiveChannelPublisher (MMCA.Common/Source/Core/MMCA.Common.Application/Interfaces/Infrastructure/ILiveChannelPublisher.cs:9): one method, PublishAsync(channelKey, eventName, payloadJson, ...) (:17), whose doc comment states the contract plainly, channel events are not persisted, so clients not connected and subscribed at publish time never see them. The framework default is the no-op NullLiveChannelPublisher (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/NullLiveChannelPublisher.cs:11), swapped for the SignalR implementation by AddPushNotifications in any host that maps the hub.

The interesting part is who publishes and where the hub is. In ADC the producers are the Engagement module's conference-day live-layer command handlers (poll open/vote/close, question submit/upvote/moderate, group 22): each calls liveChannelPublisher.PublishAsync(...) after it commits, wrapped in a swallow-everything try/catch so a broker or transport hiccup never fails the business command (for example CloseLivePollHandler SaveChangesAsync at line 64 then the best-effort publish at line 68/85). But Engagement runs in its own process, and the SignalR hub, hence the only IHubContext that can actually reach connected clients, lives in the Notification service. The seam that bridges the gap is a pair of gRPC types that both wear the same ILiveChannelPublisher abstraction:

  • On the publisher side, LiveChannelPublisherGrpcAdapter (MMCA.ADC/Source/Services/MMCA.ADC.Notification.Contracts/LiveChannelPublisherGrpcAdapter.cs:20) implements the port on top of the generated gRPC client. Every call uses a tight 2-second deadline (:26) and logs-and-swallows every failure (CA1031 at :43-48), best-effort by contract (ADR-039). Engagement's composition root registers it via AddNotificationLiveChannelClient (MMCA.ADC/Source/Services/MMCA.ADC.Notification.Contracts/DependencyInjection.cs:40), which Replaces the framework's null default (:46) so the abstraction resolves to "forward over gRPC".
  • On the receiving side, LiveChannelGrpcService (MMCA.ADC/Source/Services/MMCA.ADC.Notification.Service/Grpc/LiveChannelGrpcService.cs:19) is the gRPC server: its PushToChannel RPC (:23) simply delegates to the local ILiveChannelPublisher (:30-31), which in that host resolves to SignalRLiveChannelPublisher. It carries no [Authorize] on purpose (:14-17): like the other internal gRPC surfaces it is reachable only on the internal service network, never routed by the Gateway.

So a live event travels Engagement handler -> ILiveChannelPublisher (gRPC adapter) -> Notification's LiveChannelGrpcService -> ILiveChannelPublisher (SignalR) -> Clients.Group(channelKey) -> every browser that called JoinChannel. The same interface appears three times with three different implementations, which is the port-and-adapter idea taken to its conclusion [Rubric §7, Microservices Readiness] [Rubric §1, SOLID].

The in-app inbox channel (read side)

The inbox is pure CQRS read/write over the UserNotification aggregate, exposed by InboxController (MMCA.Common/Source/Presentation/MMCA.Common.API/Controllers/Notifications/NotificationInboxController.cs:31) to any authenticated user, unlike the organizer-only send/history endpoints. Four slices back it:

  • GetMyNotificationsHandler (MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/UserNotifications/UseCases/GetInbox/GetMyNotificationsHandler.cs:16) joins UserNotification to PushNotification in LINQ query syntax (the title/body live on the push aggregate, the read-state on the per-user row), orders newest-first, and projects a UserNotificationDTO server-side (:29-42), paginated into a PagedCollectionResult<T> with the page size clamped to 500 (:25,44-51). This cross-aggregate read is done in the query, not via a navigation populator, because the two aggregates reference each other by ID only (ADR-006), which matters because in ADC UserNotification lives in its own ADC_Notification database.
  • GetUnreadNotificationCountHandler is a single filtered CountAsync for the bell badge.
  • MarkNotificationReadHandler loads the row filtered by both NotificationId and the caller's UserId (an ownership check) before flipping it; the domain method UserNotification.MarkAsRead(...) is itself idempotent (MMCA.Common/Source/Core/MMCA.Common.Domain/Notifications/UserNotifications/UserNotification.cs:58-67), so a double UI click is harmless.
  • MarkAllNotificationsReadHandler bulk-marks the caller's unread rows.

[Rubric §11, Security], every inbox read and mark scopes by the caller's UserId (from ICurrentUserService, threaded explicitly into the query/command records), so one user cannot read or mutate another's inbox, and an id owned by someone else yields the same NotFound as a missing id (no existence leak).

The email channel

Email is the smallest channel: a single port IEmailSender with one SMTP adapter, SmtpEmailSender (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/SmtpEmailSender.cs:12), which constructs and disposes a fresh System.Net.Mail.SmtpClient per send using an ISmtpSettings-bound host/port/credentials (:14-40) and offers a default-recipient overload (:51-52) for admin/system mail. It is independent of the push/inbox flow, nothing in the send handler calls it, and is wired separately. Locally it points at the Aspire MailDev container (SMTP localhost:1025, web inbox http://localhost:1080) for visual inspection. There is no NullEmailSender analogue in this group: email is registered only where a host opts in.

Feature gating, authorization, and the API surface

The entire push/inbox REST surface is governed by one feature flag, NotificationFeatures.PushNotifications (= "Notification.PushNotifications", MMCA.Common/Source/Core/MMCA.Common.Shared/Notifications/NotificationFeatures.cs:9). Both controllers carry [FeatureGate(NotificationFeatures.PushNotifications)] (NotificationsController.cs:29, NotificationInboxController.cs:29), so the channel can be switched off per-environment without code changes via Microsoft.FeatureManagement [Rubric §10, Cross-Cutting]. Authorization splits cleanly along the two controllers: NotificationsController (send + history) requires the organizer policy (:30), while InboxController requires only authenticated (NotificationInboxController.cs:30), the asymmetry that "anyone reads their own inbox, only organizers broadcast" [Rubric §9, API & Contract Design]. Both inherit ApiControllerBase and funnel failures through HandleFailure, so a Result.Failure becomes RFC-9457 Problem Details with the right status. The SignalR hub endpoint is mapped by MapNotificationHub() (see group 12), which reads the hub path from settings and no-ops gracefully when push settings are absent.

Composition: how a host assembles the subsystem

The wiring follows the framework's layered DI sequence. The ADC NotificationModule (MMCA.ADC/Source/Modules/Notification/MMCA.ADC.Notification.API/NotificationModule.cs:14) is an IModule discovered by the ModuleLoader in topological order; it declares Dependencies = ["Identity"] with RequiresDependencies = true (:20-23, it needs the attendee query) and delegates to AddNotificationModule(...) (MMCA.ADC/Source/Modules/Notification/MMCA.ADC.Notification.API/DependencyInjection.cs:21), which calls, in order: AddModuleNotificationApplication (registers AttendeeNotificationRecipientProvider with a plain AddScoped before the Common AddNotificationApplicationServices, whose TryAdd* registrations then leave the ADC override in place, MMCA.ADC/Source/Modules/Notification/MMCA.ADC.Notification.Application/DependencyInjection.cs:23-26), AddNotificationInfrastructure (registers the EF configs for the two aggregates), and AddPushNotifications (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/DependencyInjection.cs:243). The Common-side AddNotificationApplicationServices (MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/DependencyInjection.cs:36) registers all six handlers (:49-62), the DTO mapper, the FluentValidation validator (:65), an EntityQueryService for history, a NullNavigationPopulator<TEntity>, and the null recipient fallback (:68), all via TryAdd* so consumers can override any one of them. AddPushNotifications then wires AddSignalR() (:252), adds the Redis backplane for SignalR scale-out across replicas when a redis connection string is present (:254-258), and swaps in both transient adapters, SignalRPushNotificationSender for push and SignalRLiveChannelPublisher for live channels (:261-262), plus the ClaimBasedUserIdProvider (:263). A host that never calls AddPushNotifications keeps the two no-op publishers, so the pipelines still resolve and run without any real-time transport [Rubric §1, SOLID] (DIP: handlers depend on the ports; the adapters are chosen at the composition root).

In ADC's extracted topology this whole group runs as its own service host, MMCA.ADC.Notification.Service, the first module lifted out of the monolith. It is a thin host (API + Application only; no domain logic of its own beyond the two Common aggregates) and it runs the ADR-012 mixed-endpoint profile: the default Kestrel endpoints stay Http1AndHttp2 (MMCA.ADC/Source/Services/MMCA.ADC.Notification.Service/Program.cs:66-67) because the SignalR WebSocket Upgrade handshake needs HTTP/1.1, while a dedicated Http2-only "grpc" endpoint (port 8081 in the container, 55996 locally, per the Kestrel comment at Program.cs:55-65) serves the h2c LiveChannelPushService ingress that peers resolve by the named service-discovery entry http://_grpc.notification (Notification.Contracts/DependencyInjection.cs:25-30). The host maps the hub with MapNotificationHub() (Program.cs:242) and the gRPC ingress with MapGrpcService<LiveChannelGrpcService>() (Program.cs:247); the YARP Gateway forwards the /hubs/* and /Notifications/* prefixes to it (ADR-008, group 14). Because Identity is disabled in this host, AttendeeNotificationRecipientProvider's dependency on IAttendeeQueryService is satisfied by a gRPC client (AddIdentityAttendeeClient, Program.cs:185). [Rubric §7, Microservices Readiness], the same application code that ran in-process now runs as a standalone service with no rewrite, because every collaboration (recipients, auth, delivery) is behind an abstraction resolved at the edge.

A note on what this group does not do

The PushNotificationCreated domain event is raised by PushNotification.Create (PushNotification.cs:75), but in the current code it is not the trigger for delivery: delivery happens synchronously inside the send handler after the inbox rows are written, and a search confirms no IDomainEventHandler<PushNotificationCreated> exists (the type is referenced only by the aggregate that raises it and its unit tests). The event still flows through the outbox like any other domain event (ADR-003), so it is a published, persistable record of the creation and a seam for future event-driven consumers; it is simply not wired to one today. Treat the send path described above as the live mechanism, the live-channel gRPC path as the transient mechanism, and the event as the extension point.

GetMyNotificationsQuery

MMCA.Common.Application · MMCA.Common.Application.Notifications.UserNotifications.UseCases.GetInbox · MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/UserNotifications/UseCases/GetInbox/GetMyNotificationsQuery.cs:7 · Level 0 · record (sealed)

  • What it is: the read request for one user's in-app notification inbox, a paginated list of the notifications delivered to them. It carries the user's id plus a 1-based page number and page size.
  • Depends on: nothing first-party. The UserId is typed as the UserIdentifierType alias (= int; see the identifier-alias convention in primer §2).
  • Concept introduced, the CQRS query record. [Rubric §6, CQRS & Event-Driven] assesses whether side-effect-free reads are modelled separately from state-changing writes; this is a pure query, a record that names exactly what the caller wants to read and nothing more. [Rubric §9, API & Contract Design] (a small, explicit request contract): the positional record makes the three inputs a stable, immutable shape. The two paging parameters default (PageNumber = 1, PageSize = 20, GetMyNotificationsQuery.cs:9-10) so a caller may ask for just the first page by id alone. Note the doc comment (GetMyNotificationsQuery.cs:6) advertises a max 500 ceiling, that clamp is enforced in the handler, not the record (see GetMyNotificationsHandler).
  • Walkthrough: a single positional declaration (GetMyNotificationsQuery.cs:7-10): UserId (required, first positional), then the two defaulted paging ints. sealed record gives value equality and immutability for free.
  • Why it's built this way: keeping the query a plain immutable record lets the CQRS decorator pipeline (logging, caching) wrap its handler generically without the request carrying any behavior. The UserId travels in the query rather than being read from ambient state, so the handler stays a pure function of its input and is trivially testable.
  • Where it's used: dispatched by a controller action to GetMyNotificationsHandler, which projects the result into a PagedCollectionResult<T> of UserNotificationDTO.

GetNotificationHistoryQuery

MMCA.Common.Application · MMCA.Common.Application.Notifications.PushNotifications.UseCases.GetHistory · MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/PushNotifications/UseCases/GetHistory/GetNotificationHistoryQuery.cs:6 · Level 0 · record (sealed)

  • What it is: the read request for the push-notification history, the log of push notifications that were sent (reverse-chronological), independent of any single recipient's read state.
  • Depends on: nothing first-party.
  • Concept: another CQRS query record (the pattern is introduced on GetMyNotificationsQuery). The distinction worth noting is the domain it reads: a PushNotification is the sent message (one row per broadcast), whereas a UserNotification is the per-recipient copy. This query reads the former, so it takes no UserId, only paging. [Rubric §6, CQRS & Event-Driven]: read models are shaped to the question being asked, not forced through one entity.
  • Walkthrough: two defaulted positional ints, PageNumber = 1 and PageSize = 10 (GetNotificationHistoryQuery.cs:6-8). The default page size (10) is deliberately smaller than the inbox query's (20); like the inbox query, the doc comment advertises a max 500 cap that the handler enforces.
  • Why it's built this way: history is an admin/diagnostic read over the outbound message log, so it needs no per-user ownership input, keeping the contract minimal.
  • Where it's used: handled by GetNotificationHistoryHandler, which maps PushNotification rows to PushNotificationDTO via PushNotificationDTOMapper.

MMCA.Common.Application · MMCA.Common.Application.Notifications.UserNotifications.UseCases.GetUnreadCount · MMCA.Common/Source/Core/MMCA.Common.Application/Notifications/UserNotifications/UseCases/GetUnreadCount/GetUnreadNotificationCountQuery.cs:5 · Level 0 · record (sealed)

  • Concept: the leanest of the three read queries, one positional parameter, no paging, because it returns a scalar count rather than a page. [Rubric §12, Performance & Scalability]: a dedicated count query lets the UI poll a cheap COUNT(*) instead of fetching and counting the whole inbox.

IEmailSender

MMCA.Common.Application · MMCA.Common.Application.Interfaces.Infrastructure · MMCA.Common/Source/Core/MMCA.Common.Application/Interfaces/Infrastructure/IEmailSender.cs:6 · Level 0 · interface

  • What it is: the Application-layer abstraction for sending email. Two SendAsync overloads: one to an explicit recipient, one to a configured default/system recipient (e.g. admin notifications). Infrastructure supplies the concrete transport (SMTP, SendGrid, or a no-op).
  • Depends on: BCL only (Task, CancellationToken).
  • Concept introduced, transport abstraction at the Application boundary. [Rubric §3, Clean Architecture] assesses whether the application depends on abstractions rather than concrete I/O; the what (send this email) lives here in Application, and the how (which SMTP host, which API key) lives in Infrastructure behind this interface, so a handler can send mail without referencing any mail library. [Rubric §1, SOLID] (dependency inversion): handlers depend on IEmailSender, not a concrete client, so the transport is swappable and mockable.
  • Walkthrough
    • SendAsync(string to, string subject, string body, bool isHtml = false, CancellationToken = default) (IEmailSender.cs:15): the explicit-recipient overload; isHtml defaults false so a caller sends plain text unless it opts into HTML.
    • SendAsync(string subject, string body, bool isHtml = false, CancellationToken = default) (IEmailSender.cs:23): the same, minus the recipient, targeting the configured default/system mailbox (the doc comment, IEmailSender.cs:17, names admin notifications as the use case).
  • Why it's built this way: two overloads cover the two real shapes (mail to a user vs. mail to the operators) without a settings lookup leaking into the caller; both return a bare Task (fire-and-await) because delivery success is the transport's concern, not a domain Result.
  • Where it's used: notification/command handlers that send email on a domain event (registration confirmation, admin alerts). The concrete sender is registered in Infrastructure.
  • Caveats / not-in-source: the default-recipient overload's target mailbox is resolved by the Infrastructure implementation, not visible on this contract. Not determinable from this interface which settings key supplies it.

ILiveChannelPublisher

MMCA.Common.Application · MMCA.Common.Application.Interfaces.Infrastructure · MMCA.Common/Source/Core/MMCA.Common.Application/Interfaces/Infrastructure/ILiveChannelPublisher.cs:9 · Level 0 · interface

  • What it is: the Application-layer abstraction for publishing an ephemeral live event to a channel of connected clients (e.g. event:1 or session:123). Unlike a push notification, a channel event is not persisted: clients not connected and subscribed at publish time never see it.
  • Depends on: BCL only (Task, CancellationToken).
  • Concept introduced, ephemeral fan-out vs. durable notification. [Rubric §6, CQRS & Event-Driven] and [Rubric §12, Performance & Scalability]: this is the "live, best-effort" channel (think a poll's results changing on screen) as opposed to the durable, inbox-backed push path. The doc comment (ILiveChannelPublisher.cs:3-8) draws the contrast explicitly against IPushNotificationSender: channel events have no store-and-forward, so late subscribers miss them, which is the correct trade for high-frequency, transient UI updates that would be noise in a durable inbox. [Rubric §3, Clean Architecture]: the interface names channels and payloads as plain strings, so the Application layer stays ignorant of the SignalR groups (or other fan-out) that back it.
  • Walkthrough: one method, PublishAsync(string channelKey, string eventName, string payloadJson, CancellationToken = default) (ILiveChannelPublisher.cs:17). channelKey names the group (e.g. event:1), eventName is the application-defined event (e.g. poll.results-changed), and payloadJson is a pre-serialized JSON string, so the transport does no serialization of its own.
  • Why it's built this way: passing the payload as an already-serialized JSON string keeps the publisher a dumb pipe, the caller owns the contract shape, and the transport only needs to fan out bytes to subscribers of channelKey.
  • Where it's used: application handlers that push transient, per-channel UI updates (live event/session channels); implemented in Infrastructure over SignalR groups.

INotificationRecipientProvider

MMCA.Common.Application · MMCA.Common.Application.Interfaces.Infrastructure · MMCA.Common/Source/Core/MMCA.Common.Application/Interfaces/Infrastructure/INotificationRecipientProvider.cs:8 · Level 0 · interface

  • What it is: a single-method seam that resolves who should receive a notification, returning the user ids of all eligible recipients. Consuming apps implement it against their own domain model (all attendees, users in a role, subscribers to a topic).
  • Depends on: BCL only; returns a list of the UserIdentifierType alias. Works alongside IPushNotificationSender.
  • Concept introduced, separating "who to notify" from "how to notify". [Rubric §7, Microservices Readiness] assesses whether the framework stays independent of any one app's domain; the notification infrastructure knows how to deliver (IPushNotificationSender) but must not know who the recipients are (that is app-specific, e.g. ADC's attendee model).
