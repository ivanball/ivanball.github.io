24. ADC Application Host, UI Shell & Cross-Module Composition

What this chapter covers. Everything in the earlier ADC modules, Conference, Engagement, Identity, Notification, is consumed somewhere. This chapter is that somewhere: the client tier of ADC, the code that turns the shared, per-module Razor Class Libraries (MMCA.ADC.{Module}.UI) and the framework UI package (MMCA.Common.UI) into actually running applications and composes every conference module into one coherent shell. Two application shapes are built from the same components: a Blazor Web app (Server + WebAssembly) and a .NET MAUI native app for Android, iOS, macOS, and Windows. The types this group owns are deliberately thin: the ADCHome landing page and its page-local carrier records, three per-host FormFactor implementations, the MAUI MauiTokenStorageService, the ADCHomePageContent home-content adapter, and the MAUI bootstrap chain (MauiProgram, App, MainPage, and the per-OS entry points MainActivity/MainApplication/AppDelegate/ Program/the WinUI App). The heavy lifting lives below them in the modules and in MMCA.Common.UI; this chapter is about wiring and hosting, the half of "UI architecture" the modules cannot do for themselves.

Three hosts, one shared component set. The central idea, taught in primer §2, "Write-once UI, render everywhere", is that a page is authored once as a Razor component in a module's UI library and then rendered by every host without per-platform reimplementation. There are three host projects under MMCA.ADC/Source/Hosts/UI/: MMCA.ADC.UI.Web (the Blazor Server host, SSR prerender plus the interactive Server circuit), MMCA.ADC.UI.Web.Client (the Blazor WebAssembly client, compiled to run in the browser), and MMCA.ADC.UI (the .NET MAUI host, which packages the same components into a native app that hosts them in a BlazorWebView). [Rubric §18, UI Architecture] assesses cohesive, composable components and a clean host/shell split; [Rubric §22, Responsive & Cross-Browser/Device] assesses that one UI renders correctly across browsers and devices. Both are embodied by this single-component-set, multi-host design: adding a platform is "add a host that references the shared libraries," not "fork the UI."

The shared-vs-host seam: interfaces in Common, implementations per host. The reason one component can run in three places is that everything platform-specific hides behind interfaces declared in MMCA.Common.UI. A component never asks "am I on MAUI?"; it asks an injected abstraction, and each host supplies the adapter. [Rubric §1, SOLID] (Dependency Inversion) and [Rubric §3, Clean Architecture] are visible right here: the shared library defines the ports, the host defines the adapters, and the framework depends on nothing host-specific. This group holds three of those adapters. Form factor: IFormFactor lets a component report its host and OS without #ifdef, and each host registers its own FormFactor (the Server one returns the literal "Web", MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI.Web/Services/FormFactor.cs:12; the WASM one returns "WebAssembly", MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI.Web.Client/Services/FormFactor.cs:12; the MAUI one returns the live DeviceInfo.Idiom/DeviceInfo.Platform, MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Services/FormFactor.cs:12,15). Home-page content: IHomePageContent lets the shared / route render an app-specific landing page via DynamicComponent, and ADCHomePageContent points it at ADC's ADCHome (MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI.Web.Client/Pages/ADCHomePageContent.cs:10-12). Token storage: ITokenStorageService abstracts where JWTs live; this group supplies the MAUI implementation MauiTokenStorageService, while the browser hosts get ServerTokenStorageService and WasmTokenStorageService from G15. OAuth provider availability (IOAuthUISettings) and the culture/theme plumbing also sit behind Common interfaces, described in Group 15.

The landing page: a two-tier content model. ADCHome is the conference landing page: a live countdown, the keynote, a track grid, sponsor tiers, and venue/directions. It is authored twice, once for the web hosts (MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI.Web.Client/Pages/ADCHome.razor.cs:12, shared by Server prerender and WASM interactive) and once for MAUI (MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Pages/ADCHome.razor.cs:12), because each host tree compiles the component into its own render pipeline; the duplication is structural, not a design preference. Its data splits into two tiers, and the page-local records model that split. Dynamic data (the published event's name, dates, venue) is fetched from the public events API into ADCEventInfo (MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI.Web.Client/Pages/ADCHome.razor.cs:163), wrapped by the ADCCollectionResult envelope (ADCHome.razor.cs:161), with EventPhase (Upcoming/Live/Ended, ADCHome.razor.cs:28) driving the countdown state machine. Static data that changes once per conference year, the KeynoteSpeakerInfo (ADCHome.razor.cs:269), the twelve-entry ConferenceTrackInfo grid (ADCHome.razor.cs:270), and the SponsorInfo/SponsorTierInfo lists (ADCHome.razor.cs:271-272), is baked into the binary as static readonly arrays (ADCHome.razor.cs:176,188,217). The payoff is resilience: keynote, tracks, and sponsors render immediately from memory even when the event-date API is slow or cold, and a catch (HttpRequestException) falls back to hard-coded defaults (ADCHome.razor.cs:80-83) rather than failing the page. [Rubric §18, UI Architecture] and [Rubric §19, State Management & Data Flow] are both at play: the page owns minimal state, separates fetched from authored content, and degrades gracefully.

Prerender safety and the two ADCHome copies. The web copy of ADCHome guards its initialization with if (!RendererInfo.IsInteractive) (ADCHome.razor.cs:53): during SSR prerender it skips both the backend fetch and the ticking countdown timer, because an un-timed server-side call to a cold or unreachable backend would otherwise block the prerender (and therefore the page load and the post-login NavigateTo("/")) indefinitely. The static fallback renders immediately; the interactive pass loads the real event and starts the one-second timer (ADCHome.razor.cs:60-65). The MAUI copy has no such guard (MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Pages/ADCHome.razor.cs:44-53) because the BlazorWebView is always interactive and never prerenders. This is a real, verifiable difference between the two files, and the reason the "one component, two hosts" story is not literally one source file here.

Internationalization of the shell (ADR-027). The landing page is a small, honest case study in where localization applies and where it deliberately does not. UI chrome (the fallback event description, the hero date format, the "Platinum Sponsors" heading and its singular/plural word order) resolves through the IStringLocalizer L[...] indexer keyed by resource strings, so it follows the selected culture (ADCHome.razor.cs:37,148,266), matching the multi-locale en-US + es stance of ADR-027 (see primer §6's §27 note). Editorial content, the conference brand name "Atlanta Cloud + AI Conference" (ADCHome.razor.cs:35), the postal venue address (ADCHome.razor.cs:39), and the keynote bio, track catalog, and sponsor names (ADCHome.razor.cs:174-243), stays English and is annotated in source with explicit // i18n: allow comments explaining the exemption (this is the same English-only content the API would otherwise serve). [Rubric §27, Internationalization & Localization] assesses externalized strings and culture-aware formatting; this page shows the framework's rule in miniature: localize the chrome, exempt the branded/editorial data on purpose, and mark the exemption in source.

The MAUI host: bootstrap, WebView, and the back button. The MAUI side is a thin shell around the shared components. Every platform entry point does nothing but call MauiProgram.CreateMauiApp(): MainActivity launches the Android activity (MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Platforms/Android/MainActivity.cs:13) and MainApplication forwards to CreateMauiApp() (.../Platforms/Android/MainApplication.cs:17); AppDelegate (.../Platforms/iOS/AppDelegate.cs:11) and Program (.../Platforms/iOS/Program.cs:10-11) do the same on iOS; the WinUI App (.../Platforms/Windows/App.xaml.cs:16) on Windows. MauiProgram.CreateMauiApp (.../MMCA.ADC.UI/MauiProgram.cs:33) builds the entire DI/configuration graph: because MAUI does not auto-load appsettings.json from disk, it reads it from an embedded resource (MauiProgram.cs:42), then registers MudBlazor, the shared UI (AddUIShared, MauiProgram.cs:53), the ADCHomePageContent home content (MauiProgram.cs:54), the conditionally-enabled module UIs (gated by UIModuleConfiguration so a build can ship a subset of modules, MauiProgram.cs:57-67), and the MAUI flavors of the token, refresh, and form-factor services (MauiTokenStorageService at MauiProgram.cs:70, DirectApiTokenRefresher at MauiProgram.cs:71, FormFactor at MauiProgram.cs:76). The cross-platform App (.../MMCA.ADC.UI/App.xaml.cs:11) creates a window hosting MainPage, and MainPage (.../MMCA.ADC.UI/MainPage.xaml.cs:13) hosts the BlazorWebView and intercepts the hardware/gesture back button (OnBackButtonPressed, MainPage.xaml.cs:18), forwarding it to the WebView's own history via MauiBackNavigationBridge (MainPage.xaml.cs:48) and quitting the app only when there is nowhere left to go back to (MainPage.xaml.cs:49-52). [Rubric §25, Navigation, Routing & IA] shows up in that bridge: native back must map onto in-app navigation, not OS app-switching, or the native experience feels broken. MainActivity's ConfigurationChanges attribute (MainActivity.cs:9-12) is the other native subtlety: it stops Android from destroying the activity (and with it the Blazor circuit and all component state) on rotation or dark-mode toggle.

Host security: platform-appropriate token handling. The three token-storage implementations are a compact study in secret handling matched to the threat model. On the browser hosts the high-value refresh token is never exposed to JavaScript; it stays in an HttpOnly cookie and is exchanged through the same-origin proxy (see G15's ServerTokenStorageService and WasmTokenStorageService). On MAUI, which has no DOM and therefore no XSS surface, MauiTokenStorageService stores both tokens in OS SecureStorage, the platform secure enclave (Android Keystore, iOS Keychain, Windows DPAPI, MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Services/MauiTokenStorageService.cs:5-9) under fixed key names (MauiTokenStorageService.cs:11-12). [Rubric §11, Security] (server-side and at-rest secret handling) and [Rubric §26, Front-End Security] (no token reachable from page JS) are both directly embodied; the deeper design note is MMCA.ADC/TokenStorageDesignNote.md. One deliberate development-only relaxation lives in MauiProgram: a #if DEBUG block installs a SocketsHttpHandler that bypasses SSL certificate validation (MauiProgram.cs:88-97) so a MAUI device on the LAN can reach the WebAPI over the ASP.NET dev cert; it is scoped to DEBUG and annotated inline, not a production path. The Blazor host's Content-Security-Policy is provided by BlazorCspPolicyProvider, fed to the framework's SecurityHeadersMiddleware over the ICspPolicyProvider seam (ADR-023).

How it all fits at runtime. A request to the Blazor Web host renders the shared layout from MMCA.Common.UI; the navbar is composed from each enabled module's UI descriptor (Conference, Engagement, Identity, Notification), and / renders ADCHome through ADCHomePageContent. The page fetches the event for its countdown and renders static content immediately; the interactive Server circuit or the WASM runtime takes over after prerender, auth state flows through JwtAuthenticationStateProvider reading whichever ITokenStorageService the host registered, and outbound API calls carry the access token with proactive refresh. On MAUI the same component tree runs inside a BlazorWebView with SecureStorage-backed tokens and the native back-button bridge. In every case the application talks to the backend only through the YARP Gateway: the same seam that makes the modules independently extractable (ADR-007 gRPC extraction, ADR-008 service-extraction topology) also makes the UI host-agnostic. The client points at one origin, and the Gateway routes to whichever service owns the endpoint. That is the unifying theme of this chapter: thin hosts over shared components, talking to one gateway, with every platform difference pushed behind a Common interface.

ADCEventInfo, EventPhase, KeynoteSpeakerInfo, ConferenceTrackInfo, SponsorInfo

MMCA.ADC.UI · MMCA.ADC.UI.Pages · MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Pages/ADCHome.razor.cs · Level 0 · enum + records (all private inner types of ADCHome)

These five Level-0 types are private inner types declared inside the ADCHome landing-page component. They structure the page's two content tiers: a small dynamic projection deserialized from the public events API, and a set of static records baked into the binary for the annual conference cycle (keynote bio, twelve tracks, sponsor list). Grouping them is appropriate, they are structurally identical small carriers with no behavior, but each is cited individually below.

Type File:Line Kind What it carries

ADCEventInfo …/Pages/ADCHome.razor.cs:150 record (sealed, private) API projection: Id, Name, Description?, StartDate, EndDate, TimeZone, VenueAddress?, VenueMapUrl?

  • What they are: value-shaped data carriers for the landing page. EventPhase is the three-state enum (ADCHome.razor.cs:28-33) that drives the countdown timer; ADCEventInfo (:150-158) is the deserialization-only projection of one published event; the remaining three are immutable records that model the hand-authored keynote, track, and sponsor content.
  • Depends on: nothing first-party. ADCEventInfo is materialized from JSON via the BCL (System.Net.Http.Json / System.Text.Json); the static records reference only string/string[] and MudBlazor icon string constants (Icons.Material.Filled.*).
  • Concept introduced, the page-local two-tier content model. [Rubric §18, UI Architecture & Component Design] (assesses cohesive, well-bounded component design) and [Rubric §19, State Management & Data Flow] (assesses where component data originates and how it flows): the page keeps dynamic state (the event's dates/venue, fetched at load) separate from static state (the keynote, tracks, sponsors, compiled into private static readonly arrays). Splitting the two means the keynote section, track grid, and sponsor logos render immediately from in-memory data even while the event-date API call is still in flight or has failed, the page degrades gracefully to its fallback constants. Keeping all five types private to ADCHome is deliberate scoping: no other page can accidentally couple to this projection; a page that needs event data fetches its own.
    • This also touches [Rubric §27, Internationalization & Localization], but not the way an older edition claimed. The page is localized (ADR-027, en-US + es): the UI chrome around this content resolves through an IStringLocalizer (L[...]), e.g. EventDescription => _event?.Description ?? L["Fallback.EventDescription"] (:37) and the tier headings (see SponsorTierInfo). What is deliberately not translated is the editorial content these static records carry (the keynote bio, the track catalog, the sponsor names), the same English-only copy the API would serve for a regional conference; each such literal carries an explicit // i18n: allow justification comment (ADCHome.razor.cs:161-162 for the static block, plus :35 for the conference brand name and :39 for the postal venue address). So the split is "externalize the chrome, keep the editorial body as English constants", not a blanket single-locale stance (see primer §6's §27 note and ADR-027).
  • Walkthrough,
    • EventPhase (:28-33): computed by UpdateCountdown (:93-127). Upcoming → render the countdown and tick once per second; Live (now between start and end UTC) → "happening now"; Ended → _countdownTimer?.Change(-1, -1) stops ticking after the final render (:89).
    • ADCEventInfo (:150-158): pulled out of the API envelope (see ADCCollectionResult) in LoadEventAsync (:55-76). Every property has a Razor-side fallback (e.g. EventName => _event?.Name ?? "Atlanta Cloud + AI Conference", :35), so a null/missing event still renders. TimeZone drives the IANA-zone conversion in UpdateCountdown (:99-113), which falls back to treating local time as UTC if the zone id is unknown (catch (TimeZoneNotFoundException), :109).
    • KeynoteSpeakerInfo.BioParagraphs (:256) is a string[] so the template can emit one <p> per paragraph; the single instance is the Keynote constant (:163-173).
    • ConferenceTrackInfo.Icon (:257) holds a MudBlazor Icons.Material.* constant passed to <MudIcon>; the twelve-track array is the Tracks constant (:175-201).
    • SponsorInfo (:259): the SponsorTiers array (:204-230, see SponsorTierInfo) carries [SuppressMessage("…","S1075…")] (:203) because the hard-coded sponsor logo/link URLs are intentional external constants, justified inline ([Rubric §15, Best Practices], suppressions are scoped and explained, not blanket-disabled).
  • Why it's built this way: content that changes at most once per conference year does not warrant a CMS or database table; a recompile is the cheapest "publish". The dynamic/static split is what keeps the page resilient to a slow or unreachable API.
  • Where they're used: ADCHome.razor / ADCHome.razor.cs only. An identical copy of all five exists in MMCA.ADC.UI.Web.Client/Pages/ADCHome.razor.cs (the WebAssembly client), see the caveat.
  • Caveats / not-in-source: these private types are duplicated verbatim in the WASM client's own ADCHome.razor.cs (same names, same declaration order). A page code-behind cannot be shared as one file across the Blazor-Server (MAUI host) and WebAssembly render pipelines because each compiles it independently, so content and localization-key edits must be applied to both files in lockstep.

MainActivity

MMCA.ADC.UI · MMCA.ADC.UI · MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Platforms/Android/MainActivity.cs:13 · Level 0 · class

  • What it is: the Android main activity for the MAUI host. Its entire purpose is the [Activity(…)] attribute that declares which configuration changes the activity handles in-process, so Android does not destroy and recreate it, and therefore does not tear down the BlazorWebView and its live Blazor circuit, on rotation, dark-mode toggle, or density changes.
  • Depends on: MauiAppCompatActivity (Microsoft.Maui). No first-party types.
  • Concept introduced, Android ConfigurationChanges and Blazor-circuit preservation. [Rubric §18, UI Architecture] and [Rubric §22, Responsive & Cross-Browser/Device] (assesses correct behavior across device orientations and form factors). By default Android destroys and recreates the activity on a configuration change; for a BlazorWebView that destruction kills the whole Blazor circuit and all component state. Listing ConfigurationChanges = ScreenSize | Orientation | UiMode | ScreenLayout | SmallestScreenSize | Density (MainActivity.cs:12) tells Android the activity will handle those events itself (MAUI routes them through OnConfigurationChanged transparently), so no recreation, and no lost circuit, occurs.
  • Walkthrough: the class body is empty (:13-15); MauiAppCompatActivity does all MAUI initialization. The attribute is the whole behavioral surface: Theme = "@style/Maui.SplashTheme", MainLauncher = true, plus the ConfigurationChanges flags (:9-12).
  • Why it's built this way: generated by the MAUI template; the developer-relevant insight is that removing any flag silently regresses to activity-restart for that change, dropping the circuit, a defect that only surfaces on a physical device when a user rotates or switches theme.
  • Where it's used: the Android platform entry point; referenced by no other code.

ADCCollectionResult

MMCA.ADC.UI · MMCA.ADC.UI.Pages · MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Pages/ADCHome.razor.cs:148 · Level 1 · record (sealed, private)

  • Caveats / not-in-source: in an earlier edition this was documented as a generic ADCCollectionResult<T>; the current source is non-generic (record ADCCollectionResult(List<ADCEventInfo>? Items)), hardwired to ADCEventInfo. Verified at ADCHome.razor.cs:148.

  • What it is: a private record grouping sponsors into a named, colored tier: SponsorTierInfo(string Name, string Color, IReadOnlyList<SponsorInfo> Sponsors) (ADCHome.razor.cs:258).
  • Depends on: SponsorInfo (Level 0), holding a list of it lifts this to Level 1.

ADCHome

MMCA.ADC.UI · MMCA.ADC.UI.Pages · MMCA.ADC/Source/Hosts/UI/MMCA.ADC.UI/Pages/ADCHome.razor.cs:12 · Level 2 · class (sealed partial, Blazor component)

  • What it is: the ADC conference landing page component: it renders the countdown, keynote, track grid, sponsor tiers, and venue/location, fetching the default published event from the API to drive a live countdown timer.
  • Depends on: its own Level-0/1 inner types (ADCEventInfo, EventPhase, KeynoteSpeakerInfo, ConferenceTrackInfo, SponsorInfo, SponsorTierInfo, ADCCollectionResult). External: IHttpClientFactory, System.Threading.Timer, MudBlazor, Microsoft.AspNetCore.Components, and an IStringLocalizer (the L[...] indexer inherited from the shared localized page base).
  • Concept introduced, the self-contained Blazor page with a disposable timer. [Rubric §18, UI Architecture] and [Rubric §23, Front-End Performance & Rendering] (assesses render efficiency and resource cleanup). The component implements IDisposable (:12) and manages two unmanaged-ish resources: a CancellationTokenSource for the in-flight fetch and a System.Threading.Timer for the per-second countdown. The fetch is cancellation-aware and swallows OperationCanceledException / HttpRequestException to fall back to constants (:63-70); the timer ticks via InvokeAsync(StateHasChanged) (:84) to marshal renders back onto the Blazor sync context, and stops itself once EventPhase.Ended (:86-90). Dispose (:138-145) cancels and disposes the CTS and stops+disposes the timer, the discipline that prevents a leaked timer from re-rendering a disposed component.
  • Walkthrough,
    • Fields (:14-26): ApiJsonOptions (web defaults), EventStartTime (08:00 local), the injected HttpClientFactory, plus mutable countdown state (_cts, _countdownTimer, _timeRemaining, _phase, _event, _isLoading, _disposed).
    • OnInitializedAsync (:44-53): creates the CTS, awaits LoadEventAsync, then starts the 1-second timer only if the event hasn't already ended.
    • LoadEventAsync (:55-76): resolves the named "APIClient" HttpClient, deserializes the events endpoint into ADCCollectionResult, takes the first item, and in a finally clears _isLoading and recomputes the countdown.
    • UpdateCountdown (:93-127): converts the event's local start/end to UTC via the configured IANA zone (default "America/New_York", :99; default date 2026-05-30, :95), sets _timeRemaining, and derives _phase.
    • Presentation helpers: FormatEventDate (:129-136) formats the hero date with CultureInfo.CurrentCulture and a localized format resource L["Hero.DateFormat"] (:135), so the date layout and month names follow the selected language (ADR-027), this is deliberately culture- aware, unlike the culture-invariant identifier parsing in DomainHelper. MapSearchUrl (:41-42) builds a Google-Maps deep link from the (escaped) venue address.
  • Why it's built this way: a landing page that must stay live (a ticking countdown) and resilient (renders from constants when the API is slow/down) needs exactly this fetch-then-tick-then-dispose shape; the per-component IDisposable is the correct lifetime hook for the timer and CTS.

    • Self-stopping timer. A 1-second Timer (:64) ticks OnTimerTick (:91-104), which re-renders via InvokeAsync(StateHasChanged) (:97) and, once EventPhase.Ended, calls _countdownTimer?.Change(-1, -1) to stop firing (:99-103), no point re-rendering a finished countdown.
    • Deterministic disposal. Implementing IDisposable.Dispose (:151-158) sets a _disposed flag, cancels and disposes the CancellationTokenSource, and stops/disposes the timer, preventing callbacks from firing into a torn-down component. [Rubric §16, Maintainability].

On Windows the head is a WinUI 3 Application; the framework calls its constructor before anything else (the class comment calls it "the logical equivalent of main() or WinMain()", App.xaml.cs:11-13).
