12. API Hosting, Middleware, Idempotency & DTO/Contract Mapping

This chapter is the ASP.NET Core edge of an MMCA application, everything the framework ships so a consumer host (ADC's per-module API services, the gateway, the Store API) becomes a working HTTP surface with a handful of registration calls. It is the layer where a Result.Failure becomes a 404, where a POST retried after a flaky network does not create a duplicate, where every log line carries a correlation ID, and where a domain entity is reshaped into the DTO that crosses the wire. Almost none of it is application-specific: the controller bases are generic over <TEntity, TEntityDTO, TIdentifierType>, the middleware pipeline is one method, and the consumer apps inherit a fully-formed REST contract for free. Everything here lives in MMCA.Common.API (with two small Application/Infrastructure helpers, ICorrelationContext/CorrelationContext and JwtForwardingDelegatingHandler), and it sits at the top of the Clean-Architecture stack (primer §1), it depends inward on Application and Shared, never the reverse. Read the Result pattern, Error, and ErrorType sections first: this whole chapter is, in one sentence, the place the Result pattern meets HTTP. The central rubric column is [Rubric §9, API & Contract Design] (consistent, versioned, standardized contracts and error shapes), with heavy supporting roles for [Rubric §10, Cross-Cutting Concerns], [Rubric §11, Security], [Rubric §13, Observability & Operability], [Rubric §7, Microservices Readiness], and (since ADR-027) [Rubric §27, Internationalization], because the edge is also where a culture-agnostic Result error becomes a locale-specific message.

Where Result meets HTTP, the error-mapping spine

The single most-reused idea in this chapter is that expected failures are values, not exceptions, and the edge's job is to translate those values into RFC 9457 Problem Details responses with the right status code. ApiControllerBase (MMCA.Common/Source/Presentation/MMCA.Common.API/Controllers/ApiControllerBase.cs:16) is the abstract base every controller in the system ultimately derives from; its one job is HandleFailure(IEnumerable<Error>) (ApiControllerBase.cs:25), which takes the Errors out of a failed Result and produces an ObjectResult whose status is driven by the first error's ErrorType (ApiControllerBase.cs:38). The actual ErrorType → status table is not in the controller, it lives once in the internal static ErrorHttpMapping (.../Middleware/ErrorHttpMapping.cs:15), a FrozenDictionary keyed by the eight error types (Validation/Invariant → 400, NotFound → 404, Conflict → 409, Unauthorized → 401, Forbidden → 403, UnprocessableEntity → 422, Failure → 400). Centralizing that map is what keeps every endpoint's error shape identical, the §9 "standardized error responses" story. The same map is reused by UnhandledResultFailureFilter, the safety net described below.

Localizing error messages at the edge

The error-mapping spine decides the status code; a parallel seam decides the message text, and it is the framework's headline internationalization story. The discipline is that everything below the edge stays culture-agnostic: an Error carries a stable machine Code (for example "PhoneNumber.Empty") plus an English Message, and the domain, the handlers, and the whole Result pipeline never speak a culture. Only the HTTP edge localizes, and it does so keyed by that Code. IErrorLocalizer (MMCA.Common/Source/Presentation/MMCA.Common.API/Localization/IErrorLocalizer.cs:9) is the one-method contract Localize(code, fallbackMessage); its default ErrorLocalizer (.../Localization/ErrorLocalizer.cs:11) walks an ordered set of ErrorResourceSources (.../Localization/ErrorResourceSource.cs:12), Common's own first, then each module's, and returns the first key match, falling back to the caller's original English Message when the code is empty or unknown to every source (ErrorLocalizer.cs:18-32). That graceful degradation means an untranslated code never throws and never yields an empty string. The keys live in .resx siblings anchored by ErrorResources (.../Resources/ErrorResources.cs:9, with ErrorResources.resx and ErrorResources.es.resx), and modules add their own translations additively through AddErrorResources<TResource>() (MMCA.Common/Source/Presentation/MMCA.Common.API/DependencyInjection.cs:103).

The translation is applied in exactly one place so there is no second copy to drift: ErrorHttpMapping.BuildErrorsExtension (.../Middleware/ErrorHttpMapping.cs:48) projects each Error into the ProblemDetails errors array and replaces only the human-readable Message, leaving Code/Type/Source/Target verbatim so clients can still branch on them (ErrorHttpMapping.cs:51-55). Both failure paths hand it the request's IErrorLocalizer: ApiControllerBase.HandleFailure (ApiControllerBase.cs:47-48) for returned Result failures, and UnhandledResultFailureFilter (UnhandledResultFailureFilter.cs:46) for the accidental-Ok safety net. A null localizer leaves the English text unchanged, so the seam is a no-op until registered, and it is registered automatically by AddAPI via AddErrorLocalization() (DependencyInjection.cs:77,88).

Which culture a request runs under is set by UseCommonRequestLocalization() (.../Startup/WebApplicationExtensions.cs:126), wired into the pipeline immediately after correlation (WebApplicationExtensions.cs:53). It registers ASP.NET's RequestLocalizationOptions over the framework allowlist SupportedCultures (MMCA.Common/Source/Core/MMCA.Common.Shared/Globalization/SupportedCultures.cs:9, en-US default plus es), resolving culture from the query string, then the ASP.NET culture cookie, then Accept-Language. One culture cookie is the source of truth. The Blazor UI's switcher writes that cookie via the /culture/set endpoint mapped by MapCultureEndpoint() (WebApplicationExtensions.cs:160), but because the cross-origin gateway does not carry the cookie through to the backend services, the UI's CultureDelegatingHandler re-projects the active CultureInfo.CurrentUICulture onto every outbound API call as an Accept-Language header, so a service localizes its messages to match the user's chosen locale even across a service hop. [Rubric §27, Internationalization] assesses whether the system can serve more than one locale without greenfield rework; keying translation on the stable error Code (rather than re-deriving it from English prose) is exactly that, and it is ADR-027 (multi-locale i18n), which supersedes ADR-011's former single-locale non-goal. [Rubric §9, API & Contract Design] is the companion: the contract keeps Code stable for machines while only Message follows the caller's locale.

The controller hierarchy, a generic REST contract you inherit

Above ApiControllerBase sit two generic controllers that give a module a full CRUD surface for free. EntityControllerBase<TEntity, TEntityDTO, TIdentifierType> (.../Controllers/EntityControllerBase.cs:28) supplies the read side, GET all (capped at MaxPageSize, which falls back to 500), GET /paged (filter/sort/page with pagination metadata emitted in the X-Pagination response header, EntityControllerBase.cs:144), GET /lookup (id+name pairs for dropdowns, returned as BaseLookup<TIdentifierType>), and GET /{id}, every action delegating to the IEntityQueryService<TEntity, TEntityDTO, TIdentifierType> of G03 and folding any failure back through HandleFailure (which it overrides to also log the first error at Warning level, EntityControllerBase.cs:216). AggregateRootEntityControllerBase<TEntity, TEntityDTO, TIdentifierType, TCreateRequest> (.../Controllers/AggregateRootEntityControllerBase.cs:27) adds the write side, POST (dispatched to an ICommandHandler<in TCommand, TResult> instantiated as ICommandHandler<TCreateRequest, Result<TEntityDTO>>, returning 201 Created with a Location header via CreatedAtRoute) and DELETE (dispatched to the reusable DeleteEntityCommand<TEntity, TIdentifierType> handler, returning 204). The two interface contracts IEntityControllerBase<TEntityDTO, TIdentifierType> and IAggregateRootEntityControllerBase<TEntityDTO, TIdentifierType, TCreateRequest> pin the action shapes. A third base, AuthControllerBase (.../Controllers/AuthControllerBase.cs:16), is the HTTP front for the auth subsystem of G08, login/register/refresh/revoke, each delegating to IAuthenticationService and mapping Result failures through the same HandleFailure; OAuthControllerBase (.../Controllers/OAuthControllerBase.cs:32) adds the external-provider (Google/GitHub) OAuth2 flow, minting a single-use exchange code so access/refresh tokens never ride in the redirect URL, and ServiceInfoControllerBase (.../Controllers/ServiceInfoControllerBase.cs:30) exposes an anonymous /ServiceInfo discovery endpoint served by two API versions (returning ServiceInfoResponse for v1.0 and ServiceInfoV2Response for v2.0) that proves the versioning machinery works past a single version. The net effect is that a module's concrete controller is often a near-empty class that supplies route/version attributes and the type arguments. [Rubric §1, SOLID] and [Rubric §16, Maintainability] both reward this: the CRUD logic is written once and the per-entity controller has almost no reason to change. This pairing of generic entity controllers with a dynamic query contract (filter/sort/field-selection plus the X-Pagination metadata header, described under model binding below) is ADR-034 (generic entity controllers with a dynamic query contract).

The middleware pipeline, one ordered method

Hosts do not assemble the request pipeline by hand. WebApplicationExtensions's UseCommonMiddlewarePipeline() (.../Startup/WebApplicationExtensions.cs:45) lays down the entire ordering in one place, and ordering is the whole point of a middleware chain. The sequence is: exception handler → CorrelationIdMiddleware → UseCommonRequestLocalization (sets the request's CurrentUICulture so edge error-localization runs under the caller's locale, ADR-027) → a tiny shim that captures the pre-forwarded scheme/host → UseForwardedHeaders → conditional HTTPS redirect (skipped for application/grpc so h2c calls survive) → response compression → routing → CORS → UseAuthentication → rate limiter → SoftDeletedUserMiddleware → UseAuthorization → output cache → the always-mapped JWKS and OIDC discovery endpoints → MapControllers (WebApplicationExtensions.cs:47-114). Authentication runs before the rate limiter on purpose (WebApplicationExtensions.cs:96,101 and the comment at :97-100): the global limiter partitions by the authenticated principal, so HttpContext.User must already be populated or every request would look anonymous and the per-user cap would never engage (ADR-019). Two of those middlewares are this group's own. CorrelationIdMiddleware (.../Middleware/CorrelationIdMiddleware.cs:15) reads the inbound X-Correlation-ID header (falling back to the current W3C Activity trace-id, then TraceIdentifier), stamps it onto the scoped ICorrelationContext (whose implementation CorrelationContext defaults to a fresh GUID until set), and echoes it back on the response, that ID is the same one the CQRS logging decorators (G05) attach to every log scope, which is how a single request is traceable end-to-end ([Rubric §13, Observability & Operability]). SoftDeletedUserMiddleware (.../Middleware/SoftDeletedUserMiddleware.cs:15) runs after authentication and rejects (401) any authenticated request whose user has been soft-deleted (BR-133), caching the lookup for 30 seconds (SoftDeletedUserMiddleware.cs:17) and lazily resolving the Identity-owned ISoftDeletedUserValidator so the middleware no-ops in extracted services that do not host Identity (ADR-005 soft-delete; [Rubric §11, Security], [Rubric §7, Microservices Readiness]).

Exception handling, specific-first, ProblemDetails throughout

Not every failure is a Result, some are thrown exceptions, and the framework converts those into the same Problem Details shape. The API's DependencyInjection registers an ordered chain of IExceptionHandlers (via AddCommonExceptionHandlers(), DependencyInjection.cs:116), evaluated most-specific-first: OperationCanceledExceptionHandler (client disconnect → 499 Client Closed Request, so monitoring can tell abandonment from a server error), DomainExceptionHandler (business-rule violation → 400), DbUpdateExceptionHandler (EF concurrency/constraint failure → 409, with a deliberately generic message so schema names do not leak), ValidationExceptionHandler (FluentValidation → 400 with a per-property errors dictionary), and finally GlobalExceptionHandler as the catch-all 500 (DependencyInjection.cs:121-125). Alongside them, UnhandledResultFailureFilter (.../Middleware/UnhandledResultFailureFilter.cs:21) is an IAlwaysRunResultFilter registered globally (DependencyInjection.cs:47) that catches the programmer mistake of a controller Ok(result)-ing a failed Result, without it that would serialize as a misleading 200 OK; the filter rewrites it to the correct error status using the same ErrorHttpMapping table and logs a warning. Disabled feature gates get their own consistent shape via DisabledFeatureHandler (a [FeatureGate] on a disabled flag returns 404, not 403, ProblemDetails, so a turned-off capability is indistinguishable from one that never existed and is never advertised by a 403; ADR-031 config-driven feature flags). Together these guarantee one §9 invariant: every error leaving the API, thrown or returned, expected or accidental, is an RFC 9457 ProblemDetails with a sensible status.

Idempotency, making writes safe to retry

Because clients (and resilient HTTP handlers, ADR-009) retry, a POST that creates a resource must be safe to repeat. IdempotentAttribute (.../Idempotency/IdempotentAttribute.cs:16), a ServiceFilterAttribute so it resolves from DI, marks an action (the generic CreateAsync carries it by default, AggregateRootEntityControllerBase.cs:59) and activates IdempotencyFilter (.../Idempotency/IdempotencyFilter.cs:34). The filter keys off a client-supplied Idempotency-Key header: the first response is serialized into an IdempotencyRecord (status + JSON body) and stored in ICacheService for a window configured by IdempotencySettings (default 24h, IdempotencySettings.cs:16); subsequent requests with the same key replay the cached response with an X-Idempotent-Replay: true header without re-running the command. The implementation is a careful double-check: a lock-free fast-path cache read, then a per-key SemaphoreSlim (from a ConcurrentDictionary, cleaned up when no waiters remain) and a re-check, so a burst of concurrent duplicates collapses into a single execution rather than racing. No Idempotency-Key header means the action simply runs normally. [Rubric §9, API & Contract Design] and [Rubric §29, Resilience & Business Continuity] both live here: idempotent writes are the contract that makes client-side retries safe. This whole mechanism is ADR-017 (request idempotency), the inbound-HTTP-edge counterpart to ADR-003's outbox/handler idempotency.

Model binding, JSON shaping, and DTO/contract mapping

The edge also reshapes data on the way in and out. QueryFilterModelBinder (.../ModelBinders/QueryFilterModelBinder.cs:24) parses the structured ?filters[Name].operator=contains&filters[Name].value=shirt query syntax into a Dictionary<string,(Operator,Value)> that the paged-read endpoint forwards to the query service's dynamic filtering, silently discarding incomplete pairs and matching property names case-insensitively. This binder, together with the generic controller bases above and the paged read's fields/sort/pagination contract surfaced through the X-Pagination response header, is the dynamic query half of ADR-034. CurrencyJsonConverter (.../JsonConverters/CurrencyJsonConverter.cs:13, registered in AddAPI) serializes the Currency value object as its ISO-4217 code and, on read, validates through Currency.FromCode, an invalid or non-string token throws JsonException, which the framework turns into a 400, so a bad currency never reaches a handler. The manual DTO-mapping discipline (ADR-001, no reflection-based AutoMapper) is contracted here by two Application-layer interfaces that this group owns: IEntityDTOMapper<TEntity, TEntityDTO, TIdentifierType> maps a domain entity (or collection) to its outbound DTO, and IEntityRequestMapper<TEntity, TCreateRequest, TIdentifierType> maps an inbound create request to a domain entity via its factory, returning a Result<TEntity> so request-level validation (uniqueness, etc.) flows as a value. Both are auto-registered by Scrutor's module scan and implemented per-entity, often with source-generated Mapperly mappers (primer §3). The DTO contracts they target descend from IBaseDTO<TIdentifierType> (the universal Id marker) and, where optimistic concurrency matters, IConcurrencyAware (a round-tripped RowVersion, so a stale update maps to a 409 instead of silently clobbering a concurrent edit). [Rubric §9, API & Contract Design] is the through-line: DTOs are decoupled from entities and the mapping is explicit, compile-time, and testable.

Startup wiring, module-aware controllers, and the host edge

The whole edge is assembled by extension(IServiceCollection) / extension(WebApplication) members (primer §4). DependencyInjection's AddAPI(modulesSettings, configuration) (DependencyInjection.cs:42) registers controllers with the CurrencyJsonConverter, the global UnhandledResultFailureFilter, idempotency settings + filter, feature management + DisabledFeatureHandler, edge error-localization, and, when module settings are supplied, the ModuleControllerFeatureProvider that removes controllers belonging to disabled modules from MVC discovery (so a disabled module's transitively-referenced controllers do not 500 on every call). WebApplicationBuilderExtensions supplies the cross-cutting builder-side setup, header-based API versioning (v1.0 default), a per-authenticated-user global rate limiter (anonymous, /health, /alive, /.well-known/*, and application/grpc traffic deliberately bypassed, WebApplicationBuilderExtensions.cs:35-69), Brotli/Gzip compression, OpenAPI, CORS, and the two JWT-bearer registration paths (AddCommonAuthentication for in-process HS256/RS256, AddForwardedJwtBearer for JWKS-validating extracted services, ADR-004/007), each pinning ValidAlgorithms as defense against an algorithm-confusion swap. External OAuth providers are registered by ExternalAuthExtensions (.../Authentication/ExternalAuthExtensions.cs:23), inert until an OAuth:<Provider>:ClientId is configured. The companion endpoint extensions (JwksEndpointExtensions, OidcDiscoveryEndpointExtensions, OpenApiEndpointExtensions, SignalRExtensions, MiniProfilerExtensions) map the well-known and developer endpoints (the OpenAPI/Scalar/MiniProfiler ones only outside Production). OutputCacheOptionsExtensions registers named policies backed by PublicEndpointOutputCachePolicy (.../Caching/PublicEndpointOutputCachePolicy.cs:35), which keeps a public GET cacheable even when the UI attaches a bearer token (the built-in default policy would bypass the cache for any Authorization-bearing request), with an optional bypassRoles escape so a privileged caller who gets an elevated payload always reads fresh (ADR-026 output-cache edge tier; [Rubric §12, Performance & Scalability]). DatabaseInitializationExtensions's InitializeDatabaseAsync (.../Startup/DatabaseInitializationExtensions.cs:27) runs at boot, iterating every physical data source the host uses (ADR-006 database-per-service) and applying the configured strategy, Migrate, EnsureCreated, or a production None that throws on any pending migration, before seeding enabled modules, so each service is its own sole migrator that self-applies its EF migrations at startup (ADR-030). Finally, JwtForwardingDelegatingHandler (.../Infrastructure/Http/JwtForwardingDelegatingHandler.cs:17) is the outbound counterpart: it copies the caller's inbound Authorization header onto every outgoing HTTP call so distributed authorization works across service hops (mirroring the gRPC interceptor of G13). The two per-package markers AssemblyReference and ClassReference (.../AssemblyReference.cs:8,20) are the anchors the module scan and architecture tests pin against.

The end-to-end request flow

Putting it together, a typical authenticated POST /sessions flows like this. The request enters UseCommonMiddlewarePipeline: the exception handler wraps everything, CorrelationIdMiddleware assigns the trace ID, forwarded-header and HTTPS handling run, CORS gates the origin, UseAuthentication validates the bearer JWT (HS256 in the monolith, RS256 + JWKS across services), the per-authenticated-user rate limiter then throttles the now-identified caller (auth-first so the cap partitions by principal, ADR-019), and SoftDeletedUserMiddleware confirms the user is still active. Routing dispatches to a concrete controller derived from AggregateRootEntityControllerBase; the [Idempotent] attribute's IdempotencyFilter checks the Idempotency-Key cache and either replays or proceeds. The CreateAsync action binds the JSON body (with CurrencyJsonConverter shaping any Currency), then calls the create ICommandHandler, which runs the full decorator pipeline of G05, invoking the IEntityRequestMapper to build the entity and the IEntityDTOMapper to shape the response. On success the controller returns 201 Created, the response is cached against the idempotency key, and the correlation ID is echoed back. On a Result.Failure, HandleFailure maps the first Error's type to a status via ErrorHttpMapping and returns ProblemDetails; on a thrown exception, the ordered exception handlers do the equivalent. The same shapes apply whether the host is the ADC monolith or an extracted module service behind the YARP gateway, that transport-agnostic edge is exactly what makes service extraction reversible (ADR-007/008; [Rubric §7, Microservices Readiness]).

PublicEndpointOutputCachePolicy

MMCA.Common.API · MMCA.Common.API.Caching · MMCA.Common/Source/Presentation/MMCA.Common.API/Caching/PublicEndpointOutputCachePolicy.cs:35 · Level 0 · class (sealed)

  • What it is: a custom ASP.NET Core IOutputCachePolicy for public, user-independent GET/HEAD endpoints that must stay cacheable even when the request carries an Authorization header. It exists to fix a specific interaction: the framework UI attaches a Bearer token to every outgoing API call, including reads of [AllowAnonymous] endpoints whose body is identical for every caller, so the built-in default output-cache policy (which refuses to serve or store a cached response for any authenticated request) causes logged-in users to bypass the cache entirely and hit the database on every read (doc comment, PublicEndpointOutputCachePolicy.cs:8-18).
  • Depends on: Microsoft.AspNetCore.OutputCaching (IOutputCachePolicy, OutputCacheContext, CacheVaryByRules), Microsoft.AspNetCore.Http (HttpMethods, StatusCodes), System.Security.Claims.ClaimsPrincipal, Microsoft.Extensions.Primitives.StringValues.   • Concept introduced, caching public reads across the authenticated/anonymous divide. [Rubric §12, Performance & Scalability] (assesses caching, load-shedding, and read throughput) and [Rubric §11, Security] (assesses that a shared cache never leaks identity-scoped data). §12 is the reason this class exists: replacing the default policy on identity-independent reads means a logged-in caller's request is served from the output cache instead of round-tripping to the database, so the cache actually earns its keep for the common (authenticated) case. §11 is the guardrail: because a cached response is served verbatim to every subsequent caller, the class docs warn to apply it ONLY where the payload does not depend on the caller (PublicEndpointOutputCachePolicy.cs:24-33), and it offers a bypassRoles escape hatch for endpoints that return an elevated payload to a privileged role (e.g. organizers see unpublished rows), those callers skip the cache entirely so their identity-dependent response is never stored.
