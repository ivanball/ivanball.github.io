26. Testing & Quality Infrastructure

What this chapter covers. This is the largest group in the guide by type count, more than a thousand types (1,124 in the group membership): the great majority are per-project test classes rolled up by project in the Per-project test rollup (1,003 of them), joined by the reusable test infrastructure sectioned in full, because everything that proves the system works lives here. That includes the shipped, reusable test-base packages; the shared architecture-fitness rule library and the per-repo thin suites that turn the layering rules into executable governance; the bUnit component tests and the Playwright end-to-end suites with their page objects; and the thousands of [Fact]s that pin domain invariants, CQRS handlers, validators, DTO mappers, EF configurations, and controllers. Four of the framework's fourteen NuGet packages are test infrastructure, not test code, MMCA.Common.Testing, MMCA.Common.Testing.E2E, MMCA.Common.Testing.UI, and MMCA.Common.Testing.Architecture, all under MMCA.Common/Source/Hosting/ (see primer §5), consumed by downstream apps as the foundation for their own tests. The bulk of the group, the per-[Fact] test classes, is rolled up by project in the Per-project test rollup; the reusable bases, the shared fitness library and its thin subclasses, and the gallery harness get full per-type sections in the other units of this chapter. This overview teaches how the whole quality apparatus fits together: the test-pyramid layers, the seams the framework ships so consumers do not re-invent them, and the runtime mechanics of the trickier tiers (Blazor-aware E2E, in-process gallery hosting, NetArchTest fitness functions consumed through a per-repo IArchitectureMap).

This group is where [Rubric §14, Testability & Test Strategy] (assesses whether the codebase has a deliberate, layered test strategy, fast unit tests at the base, integration above, a thin E2E cap, and whether testing is designed for, not bolted on) is most directly visible. The presence of shipped test-base packages, two independent fitness gates (compile-time + NetArchTest), and a backend-less accessibility gallery is strong evidence that testability is a first-class architectural concern, not an afterthought.

The test pyramid, layer by layer. The suites map onto a classic pyramid. At the base are pure unit tests with no I/O, domain entity factories and invariants, value objects, the Result/Error primitives, CQRS handlers with mocked repositories, FluentValidation validators, and DTO/request mappers. These dominate the type count and run in-memory in milliseconds. Above them sit persistence and EF-configuration tests, many of which spin up SQLite in-memory to exercise real EF model building (entity configurations, the outbox, repositories, the multi-data-source resolver) without a SQL Server. Above those are integration tests that boot a whole service host through WebApplicationFactory<Program> and hit it over HTTP with a real (but resettable) database. Capping the pyramid are the component tests (bUnit, rendering Blazor components in isolation) and the E2E tests (Playwright, driving a real browser against the running stack). A separate, orthogonal tier, the architecture-fitness tests, sits outside the pyramid: they assert structural rules over compiled assemblies and need neither a database nor a browser.

The shipped reusable bases (the part downstream apps depend on). The integration tier is anchored by IntegrationTestBase<TFixture> and its companion abstraction IIntegrationTestFixture. The base supplies a configured HttpClient, bearer-token management, typed GetAsync<T>/PostAsync<T>/PutAsync<T>/DeleteAsync helpers, per-test database reset (InitializeAsync calls Fixture.ResetDatabaseAsync()), and a thread-safe NextId() counter; downstream fixtures implement IIntegrationTestFixture to own the WebApplicationFactory lifecycle and, critically under database-per-service (ADR-006), reset every physical data source, which the interface's own doc comment spells out. The framework also ships a concrete fixture scaffold, SqlServerIntegrationTestFixtureBase<TEntryPoint>, which boots a service host in-process against a throwaway SQL Server database (LocalDB by default, a CI SQL container via an env-named base connection string), applies the module's migrations on first start, resets data between tests with Respawn, and drops the database on disposal, pushing all host-specific settings through process environment variables it restores on teardown. Authentication for these tests comes from JwtTokenGenerator, which signs RS256 tokens with an embedded dev keypair so tests exercise the same JWKS/RS256 validation path production runs (ADR-004); each consumer extends it with role-specific helpers (AdminToken, OrganizerToken). Test data is assembled through fluent builders descending from EntityBuilderBase<TBuilder, TEntity>, and feature flags are toggled in tests via FeatureManagementTestExtensions (ConfigureTestFeatureFlags overrides FeatureManagement:* through an in-memory configuration). This "ship the test base as a package" decision is what lets ADC and Store write integration tests with a few lines of fixture glue instead of re-deriving HTTP/auth/reset plumbing, a concrete [Rubric §16, Maintainability & Evolvability] win (one base evolves, all consumers inherit the fix).

The component tier, bUnit against MudBlazor. BunitComponentTestBase (in MMCA.Common.Testing.UI) is the shared base for rendering Blazor components without a browser. It registers MudBlazor's services, puts JSInterop in loose mode so MudBlazor components that probe JS during render do not throw, and wires permissive-but-real auth doubles (a mutable AuthenticationStateProvider plus an IsAuthenticated-only IAuthorizationService) so both <AuthorizeView> cascades and pages that inject AuthenticationStateProvider directly resolve; it also registers AddLocalization() so components that inject IStringLocalizer<T> render localized markup (ADR-027) with no per-test setup. Tests render anonymous by default and pass a ClaimsPrincipal to exercise the authorized branch; a RenderMudProviders() helper renders the popover/dialog/snackbar providers so components that open a dialog or raise a toast have somewhere to render. The base is deliberately written against bUnit v2 (the line compatible with xUnit v3 / Microsoft Testing Platform) and isolates the two version-specific symbols so a downgrade would touch only this one file. Alongside behavioral assertions, the tier ships a dependency-free golden-markup regression helper, MarkupSnapshot (returning a MarkupSnapshotResult), which normalizes the per-render GUIDs MudBlazor injects into element ids and ARIA associations, then compares a component's rendered markup to a committed baseline under a Snapshots/ folder so an unintended structural change to a shared primitive fails the build; UPDATE_SNAPSHOTS=1 rewrites baselines for review-and-commit, and a missing baseline is written and reported as a non-match so a regression can never slip through on an absent snapshot. This tier squarely serves [Rubric §28, Front-End Testing] (assesses whether UI logic is tested below the slow E2E layer).

The E2E tier, Blazor-aware Playwright. The MMCA.Common.Testing.E2E package is the most mechanically interesting tier because Blazor InteractiveAuto breaks naive browser automation: a page renders as static prerendered HTML before the WASM/SignalR runtime hydrates, so a click or fill issued too early is silently dropped. PlaywrightFixture (an xUnit collection fixture) launches one browser per run, selecting the engine from E2E_BROWSER (chromium/firefox/webkit) for the cross-browser matrix ([Rubric §22, Responsive/Cross-Browser]). E2ETestBase gives each test a fresh IBrowserContext/IPage, optional trace capture (per-test traces written only for failed tests when E2E_TRACE names a directory), and LoginAsAdminAsync/LoginAsUserAsync/RegisterNewUserAsync flows. The hydration-race fixes live in PageExtensions: GotoAndWaitForBlazorAsync waits for window.Blazor._internal then flushes two animation frames plus a short settle delay; BlazorNavigateAsync does client-side SPA navigation for already-authenticated pages (avoiding a prerender that lacks the in-browser JWT); and FillAndVerifyAsync fills a field then auto-waits on ToHaveValueAsync, re-typing character-by-character if re-hydration wiped the value, the single shared helper that replaced a swarm of fixed-delay retry loops (each of these has its own war story in the project's memory). The configuration surface (E2ETestConfiguration and its AdminCredentials/UserCredentials) is env-overridable so the same suite runs locally and in CI, with a tunable AuthTimeout/AuthGraceTimeout for the contended-runner auth-detection race. The package also ships abstract workflow bases, UserLoginTestsBase, UserRegistrationTestsBase, ProfileManagementTestsBase, LogoutTestsBase, and Identity page objects (LoginPage, RegisterPage, ProfilePage) so a consumer gets the entire Login/Register/Profile/Logout journey, including its accessibility assertions, by subclassing.

Accessibility as a gate, not a checkbox. [Rubric §21, Accessibility] is enforced in the test run. PageExtensions.AssertNoAccessibilityViolationsAsync runs an axe-core scan and throws AccessibilityViolationException (with a per-node, compacted-HTML summary pinpointing the offending element) if any violation is found; AxeOptions scopes every scan to the documented target, WCAG 2.1 AA (wcag2a/wcag2aa/wcag21a/wcag21aa tags), deliberately excluding axe's advisory "best-practice" rules so the gate fails only on real conformance violations. A single documented exception, AxeOptions.Wcag21AaExceptMudPagerCombobox, disables the one aria-input-field-name rule that MudBlazor 9.6.0's unlabeled MudTablePager "rows per page" select trips (accepted upstream limitation, not reachable from app markup). The shipped workflow bases call this on Login/Register/Profile, and new pages are expected to add an equivalent check.

The accessibility gallery, testing the framework's own UI. A subtlety: MMCA.Common ships UI components (MMCA.Common.UI) but is not a runnable app, so how does its own CI a11y-scan its shared Login/Register pages? Answer: GalleryHost builds a backend-less Blazor host that renders the real shared auth pages (/login, /register) and a primitives + notifications showcase with stubbed consumer seams (no-op auth, anonymous auth-state, null token storage, canned notification services), discovered through a one-line GalleryUIModule whose assembly the shared Router scans. The gallery E2E collection fixture self-hosts that host in-process on an ephemeral Kestrel port (a real StartAsync() on a bound port, not WebApplicationFactory's in-memory TestServer, which Playwright cannot reach over the wire), deliberately avoiding the separate-dotnet run + health-poll cold-start that made ADC's E2E job fragile. It even enables the qps-Ploc pseudo-locale unconditionally so a pseudo-localization round-trip runs as a CI gate ([Rubric §27, Internationalization]). The gallery and its E2E project are intentionally outside MMCA.Common.slnx so the unit-test run stays fast; they run only in CI's dedicated ui-e2e job. This is the framework eating its own dog food: the shared UI is held to the same WCAG 2.1 AA gate it imposes on consumers.

Front-end performance, measured on the live page. The E2E package also ships WebVitalsCollector, which installs browser PerformanceObserver hooks before first paint and reads back Core Web Vitals (LCP, CLS, FCP, TTFB) plus a single-interaction INP sample from the real page, with no third-party JS or network egress ([Rubric §12, Performance & Scalability] and [Rubric §23, Front-End Performance]). LCP/CLS are Chromium-only, so on Firefox/WebKit those fields stay zero and the budget assertions pass; consumers keep their own budget-asserting tests, this class is only the measurement infrastructure (the client-side analogue of a backend load test).

Architecture-fitness tests, executable governance shipped as a package. Distinct from the pyramid, the *.Architecture.Tests projects assert structural rules over compiled assemblies using NetArchTest.eNhancedEdition. They are the runtime half of a doubled enforcement scheme (the compile-time half is the LayerEnforcement.targets MSBuild guard described in primer §4). The defining design of this chapter is that those rules live once in a shared NuGet package, MMCA.Common.Testing.Architecture (ADR-015), rather than being copy-pasted per repo. It defines the reusable rule bodies once, ArchitectureRules (a partial class), the ArchitectureAssert diagnostics, the RuleHelpers reflection toolbox, and twenty-five abstract *TestsBase classes (e.g. LayerDependencyTestsBase, DomainPurityTestsBase, MicroserviceExtractionTestsBase, PiiConventionTestsBase) full of [Fact]s, most parameterized by a single seam, IArchitectureMap (the exact method-and-base tallies are computed from source and CI-gated in MMCA.Common's FACTS.md, so this chapter does not restate the running totals). The library reaches beyond pure layering. A resx-parity gate (LocalizationResourceTestsBase, subclassed by Common's LocalizationResourceTests and joined by ADC's own TranslationCompletenessTests) fails the build when an English .resx key ships without its required-culture sibling, turning translation completeness into a build gate (ADR-027, [Rubric §27, Internationalization]). A vertical-slice cohesion gate (SliceCohesionTestsBase into SliceCohesionTests) fails when a handler or validator is stranded out of its command/query namespace, keeping a feature a cohesive unit rather than a smear across horizontal folders ([Rubric §5, Vertical Slice]). And a non-vacuous PII data-governance contract (PiiErasureContractFitnessTests) forces a representative [Pii]-carrying data subject through redaction, log-masking, and in-place IAnonymizable erasure end to end, closing the structurally-vacuous gap the framework-side PiiConventionTestsBase scan leaves (no data subject lives in Common's Domain), ADR-005, [Rubric §30, Compliance/Privacy/Data Governance]. Each repo supplies one map implementation declaring its layer/module assemblies (CommonArchitectureMap, AdcArchitectureMap), and its arch-test classes shrink to thin sealed subclasses (= new XArchitectureMap();, three to ten lines) that inherit the actual [Fact]s. So the same compiled rule, "Domain must not depend on Application", "[Pii] implies IAnonymizable", "Application/Domain/Shared must never reference MassTransit" (depend on IMessageBus instead, ADR-007/008), runs identically across MMCA.Common, MMCA.ADC, and MMCA.Store, eliminating the drift the old per-repo duplication invited. The framework-only checks that do not generalize (the MMCA.Common.Grpc boundary, IMessageBus/IJwksProvider placement) stay in FrameworkSanityTests; DependencyVersionTests (a thin subclass of DependencyVersionTestsBase) parses Directory.Packages.props to fail the build if MassTransit's major reaches 9 (the v8-pin policy, v9 needs a commercial license; only Common pins it, so only Common subclasses the base); and ADC adds the document-consistency DataResidencyTests. This whole apparatus is the clearest expression in the codebase of [Rubric §34, Architecture Governance & Documentation] (architecture rules automatically enforced, not merely documented) crossed with [Rubric §14, Testability & Test Strategy] (the cheapest, fastest tier, pure reflection, no DB or browser) and [Rubric §7, Microservices Readiness] (the extraction-coupling rules keep the modules liftable); shipping the rules as a versioned package is itself a §16/§34 maintainability win (one rule evolves, every consumer inherits it on the next bump).

Operational and host-hardening fitness. A cluster of host-level checks proves the runtime posture the production deploy leans on, distinct from both the pyramid and the structural rules. In MMCA.Common.Aspire.Tests, WarmupReadinessGateTests and WarmupHostedServiceTests pin the startup warm-up contract ([Rubric §29, Resilience & Business Continuity]): the readiness gate stays closed until every IWarmupTask has run, then latches open idempotently and thread-safely, and crucially it opens even when a warm-up task throws, so a transient dependency outage can never wedge a replica permanently out of rotation (ADR-025). SecurityHeadersMiddlewareTests (and the shipped, host-agnostic SecurityHeadersTestsBase that any host can subclass to probe its own edge) assert the hardened baseline response headers are always emitted (X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Referrer-Policy, Permissions-Policy), HSTS is environment-gated, and the Content-Security-Policy follows the injected ICspPolicyProvider, a [Rubric §26, Front-End Security] concern verified at the host edge. ADC's Gateway suite carries the sibling GracefulShutdownTests: it boots the real Gateway host and asserts a requested stop drains and fires ApplicationStopping then ApplicationStopped within the bounded shutdown timeout, so a hosted service that refused to drain (and would silently wedge a rolling deploy) fails the build instead of latently in prod.

Performance smoke, measured not assumed. A dedicated BenchmarkDotNet project, MMCA.Common.Benchmarks ([Rubric §12, Performance & Scalability], which assesses whether hot paths are measured rather than assumed), rounds out the quantitative tiers. SpecificationBenchmarks profiles the Specification hot path: it contrasts the production path (one SampleItem-typed spec instance reused, so its compiled expression stays cached) against the anti-pattern of a fresh spec per evaluation that recompiles each call, and measures the cost of composing an And/Or EF-translatable criteria tree, all DB-free under [MemoryDiagnoser]. Like the gallery and its E2E suite, the benchmark project sits outside MMCA.Common.slnx and runs on demand (dotnet run -c Release --project Tests/Performance/MMCA.Common.Benchmarks, with a fast --job Dry correctness smoke), so it never slows the unit-test job.

Test-runner mechanics you will meet immediately. All projects run on Microsoft Testing Platform (MTP), not VSTest, set by global.json, with xUnit v3, AwesomeAssertions, and Moq. Run one class or method by targeting the project and passing an MTP filter after -- (-- -class "*FooTests*" / -- -method "*Pattern*"); see primer §5. Every test project must contain at least one test or MTP exits 8, CI uses --minimum-expected-tests 1 to catch silent discovery failures. Test files relax naming/complexity analyzer rules via the [Tests/**/*.cs] .editorconfig section. A practical wrinkle worth internalizing: in the headless authoring environment there is no reachable SQL Server, so the SQL-backed integration tiers (including SqlServerIntegrationTestFixtureBase<TEntryPoint>) build but can only be run-verified under Aspire/Docker or a CI SQL service container; the SQLite-in-memory persistence tests, the pure unit tests, and the NetArchTest fitness suites all run anywhere.

EntityBuilderBase<TBuilder, TEntity>

MMCA.Common.Testing · MMCA.Common.Testing.Builders · MMCA.Common/Source/Hosting/MMCA.Common.Testing/Builders/EntityBuilderBase.cs:9 · Level 0 · class (abstract)

  • What it is: the abstract seed of the test data builder pattern. A concrete builder subclasses it, wires sensible defaults, and exposes fluent WithX(...) setters so a test spells out only the properties it actually cares about (EntityBuilderBase.cs:3-8).
  • Depends on: no first-party types. It is a two-parameter generic constrained by a self-referential type bound, where TBuilder : EntityBuilderBase<TBuilder, TEntity> (:9-11), the curiously-recurring template pattern (CRTP) that lets each WithX on a subclass return the concrete TBuilder for chaining rather than the base type. The intended Build() implementation calls the entity's own Result<T>-returning factory, so builders sit conceptually atop AuditableAggregateRootEntity<TIdentifierType>'s Create(...) idiom and the Result type, though the base itself references neither.
  • Concept introduced, the test data builder. [Rubric §14, Testability & Test Strategy] (assesses whether the codebase makes tests cheap to write and resistant to churn) and [Rubric §33, Developer Experience] (assesses the ergonomics of the day-to-day authoring loop). Entities here can only be born through a private-constructor + static Create factory that returns a Result<T> (see primer §2), so a test that news up a valid aggregate by hand must repeat every required field on every call. A builder centralizes those defaults: the test overrides one property, calls Build(), and the builder feeds the full set through the factory, throwing if the factory returns a failure (the class doc records exactly that contract, :12-16). The self-typed generic is what keeps the fluent chain returning the derived builder, so new EventBuilder().WithTitle("X").WithCapacity(50).Build() stays strongly typed at every hop.
  • Walkthrough: the base is deliberately almost empty, one abstract member: TBuilder Build() (:17) is the terminal operation each concrete builder implements to run the domain factory and return the constructed TEntity. There are no fields and no constructor: the base contributes only the type shape (the CRTP bound) and the Build() obligation; defaults and WithX setters live on the subclass.
  • Why it's built this way: pushing the shared shape (self-typed generic, Build() contract) into one abstract base means every module's builders read identically and a reviewer knows where to look; keeping it dependency-free means the builder base ships in the test package without dragging Domain in. Throwing on a factory failure inside Build() is intentional: a builder is for the happy-path fixtures a test takes as given, so a misconfigured default should fail loudly at construction rather than hand back a half-built object.
  • Where it's used: subclassed by the per-entity builders in the downstream apps' test suites (for example an EventBuilder / SessionBuilder in the ADC integration tests) and consumed by tests built on IntegrationTestBase<TFixture>.
  • Caveats / not-in-source: the base declares only Build(); the defaulting and fluent setters, and the throw-on-failure behavior the doc comment describes, are the responsibility of each concrete subclass and are not visible in this file.

IIntegrationTestFixture

MMCA.Common.Testing · MMCA.Common.Testing · MMCA.Common/Source/Hosting/MMCA.Common.Testing/IIntegrationTestFixture.cs:8 · Level 0 · interface

  • What it is: the minimal contract an integration-test fixture satisfies: hand a test an HttpClient against an in-process host, and reset the database to a clean slate between tests (IIntegrationTestFixture.cs:3-7).
  • Depends on: BCL only, HttpClient and Task. No first-party types: it is the abstraction that IntegrationTestBase<TFixture> is generic over, so the test base couples to this interface rather than to any concrete host-boot mechanism.
  • Concept introduced, the fixture / test-base split for in-process integration tests. [Rubric §14, Testability & Test Strategy] (the integration tier boots the real ASP.NET pipeline in memory, one rung up the pyramid from the reflection-only fitness tests) and [Rubric §7, Microservices Readiness] (assesses whether the design survives being split into per-service databases). An integration test needs two lifetimes: an expensive host + database that is stood up once per test class (the fixture), and a cheap reset that runs before each test so tests do not see each other's rows. This interface names exactly those two operations and nothing else, leaving how the host boots and how the database resets to the concrete fixture. The ResetDatabaseAsync doc (:13-18) is where the database-per-microservice reality surfaces: a host that owns multiple physical data sources must reset every relational source, and the doc points implementers at resolving IEntityDataSourceRegistry and IDataSourceResolver from the host to enumerate the connection strings (see ADR-006, database-per-service).
  • Walkthrough: two members:
    • HttpClient CreateClient() (:11), produces a client wired to the in-memory test server, the surface a test drives requests through.
    • Task ResetDatabaseAsync() (:19), returns the database to a known-clean state between tests, typically via Respawn, and (per the doc) across all relational sources when the host is multi-source.
  • Why it's built this way: keeping the contract to two methods means any host, a single-database monolith or a multi-source extracted service, can satisfy it, and the shared test base stays host-agnostic. It is the interface segregation that lets one IntegrationTestBase<TFixture> serve every repo.
  • Where it's used: the TFixture constraint on IntegrationTestBase<TFixture>, and implemented by SqlServerIntegrationTestFixtureBase<TEntryPoint> (the shared SQL Server fixture) and by downstream per-service fixtures.

Per-project test rollup

This guide treats tests as grouped, not sectioned per [Fact] (the logged exception in the charter): the reusable test bases, the shared architecture-fitness library and its per-repo thin subclasses, and the component Gallery harness each get their own ### treatment in the earlier parts of this chapter, but the bulk of the suite, 1,003 individual test types across 36 projects, is rolled up here. Each row below names a test project (assembly), the count of test types it contributes to the 1,003, what it covers, and its style (unit / integration / component / E2E / performance-smoke). Counts reconcile exactly to the unit input.

A few cross-cutting facts hold for every row, so they are stated once here rather than repeated:

  • Stack. Every project is xUnit v3 run under the Microsoft Testing Platform (not VSTest, global.json sets "runner": "Microsoft.Testing.Platform"), with AwesomeAssertions for fluent asserts, Moq for test doubles, and coverlet for coverage. The lone exception is MMCA.Common.Benchmarks, a BenchmarkDotNet executable (not a test project). See primer §3 for the platform/runner externals.
  • Layering mirror. The ADC module suites repeat the same seven-project shape per module ({Module}.{Shared,Domain,Application,Infrastructure,API,UI}.Tests + a per-service {Module}.IntegrationTests), so once you understand the Conference column you understand Engagement and Identity: they differ only in volume, not in kind.
  • Fitness tests and shared bases live elsewhere. MMCA.Common.Architecture.Tests and MMCA.ADC.Architecture.Tests (the NetArchTest layer/purity/extraction suites, thin subclasses of the shared ArchitectureRules rule library) are not in this table: they are covered as first-class sections earlier in this chapter. The same is true of the shared test bases (IntegrationTestBase<TFixture>, BunitComponentTestBase, the Playwright fixtures) and the MMCA.Common.UI.Gallery harness.
  • Two integration tiers, deliberately split. Each service has a per-service *.IntegrationTests project that boots one host through WebApplicationFactory<Program> with cross-service gRPC edges faked and no broker (these gate deploy via the integration-tests CI job and need a real SQL Server named by ADC_TEST_SQL_BASE). Separately, MMCA.ADC.CrossService.IntegrationTests boots all three REST hosts against Testcontainers SQL Server + RabbitMQ to prove the genuine broker and gRPC round-trips (a non-gating nightly tier that needs Docker). [Rubric §14, Testability] (assesses how thoroughly and at what cost the system can be verified): the count and spread below, heavy at the inner Application/Domain layers, thinner at the edges, with a dedicated integration + E2E tier, is the classic healthy test pyramid, and the fact that the volume concentrates in fast in-memory unit layers keeps the feedback loop cheap.
