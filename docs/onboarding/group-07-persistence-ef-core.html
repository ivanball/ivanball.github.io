7. Persistence & EF Core

What this chapter covers. This group is the codebase's storage engine room: everything that turns a tracked domain entity into rows in the right database, and back. It answers four questions that the higher layers take for granted. Which database does this entity live in, and how is that decided without the domain knowing anything about persistence? How does one EF Core DbContext class serve several physically-separate databases at once, and degrade gracefully when a relationship crosses a database boundary? How do audit stamping, soft-delete, optimistic concurrency, and domain-event capture happen automatically on every save without leaking into any handler? And how do dotnet ef migrations work when the schema is split across one database per module? The answers are the database-per-service strategy (ADR-006) made concrete: a configuration-driven data-source router, an eager entity to database registry, a per-source context factory, a unit of work that fans saves and transactions across sources, two SaveChanges interceptors, a small library of model conventions and value converters, and the engine-specific EF configuration base classes that the primer called "the per-entity half of database-per-service."

If you have not yet internalized the entity contracts (IAuditableEntity, AuditableBaseEntity<TIdentifierType>, AuditableAggregateRootEntity<TIdentifierType>), soft-delete + audit, strongly-typed identifier aliases, or the Result pattern, skim primer §2 first. This chapter is where those promises are kept. It also sits underneath two neighboring chapters that lean on it heavily: the read engine in G03, Querying & Specifications consumes the repositories and the data-source service defined here, and the event spine in G04, Events & Outbox is written into the very save pipeline this chapter describes.

The central design tension: one DbContext class, many databases

Start with the most important and most counter-intuitive fact, because every other design choice flows from it. There is exactly one concrete relational context class, SQLServerDbContext, over the abstract base ApplicationDbContext (with sibling CosmosDbContext and SqliteDbContext for the two non-production engines). The codebase deliberately does not split this into a context-per-module (ConferenceDbContext, EngagementDbContext, and so on); that is a standing design decision, and the PendingModelChangesWarning suppression in SQLServerDbContext.cs:47-58 exists precisely to support it. Yet ADC runs four separate SQL databases, one per module, database-per-service (ADR-006). How can one context class serve four databases?

The trick is that ApplicationDbContext is instantiated once per physical database, each instance carrying a different PhysicalDataSource (connection string + migrations assembly) and building a different model that contains only its own database's entities. The mechanism that keeps EF from reusing the first model for every database is DataSourceModelCacheKeyFactory, installed in OnConfiguring (ApplicationDbContext.cs:101): it keys EF's internal model cache by (context type, physical source name) rather than by context type alone. Without it, the first model built would silently be reused for all four databases, a subtle, catastrophic bug the comment at ApplicationDbContext.cs:98-100 calls out explicitly. So the singular SQLServerDbContext class becomes four runtime models, one per database, each filtered to its own entities by ApplicationDbContext.ApplyConfigurationsForEntitiesInContext (ApplicationDbContext.cs:241-268), which asks the registry which entities belong to this source and applies only their configurations.

Deciding where an entity lives: the routing chain

An entity's home database is never declared on the entity, it is derived, eagerly, at startup, from its EF configuration class. Three cooperating types form the routing chain, and it is worth holding them apart:

  • DataSource is the engine enum (SQLServer, Sqlite, CosmosDB), and DataSourceKey is the (engine, database-name) record struct that uniquely identifies one physical database. PhysicalDataSource pairs a key with its resolved connection string, migrations assembly, and Cosmos database name.
  • DataSourceResolver (singleton, IDataSourceResolver) is the configuration-to-physical mapper. At construction it reads the ConnectionStrings (the Default source) and the DataSources appsettings section, and builds a logical-name to physical-key map per engine. Its crucial behavior is collapse: a logical source with no connection string, or one whose connection string equals the top-level default, collapses onto the shared Default source, so a host with no DataSources config behaves exactly like a single-database monolith (one context, one change tracker, FKs intact). Two logical names sharing a connection string collapse to one canonical physical source, and conflicting SQLServerMigrationsAssembly declarations on a collapsed source fail fast at startup (DataSourceResolver.cs:243-245). This collapse property is the single feature that makes the monolith and the distributed deployment the same code.
  • EntityDataSourceRegistry (singleton, behind IEntityDataSourceRegistry) is the entity-to-physical mapper. It reflects over the configuration assemblies once, finds each IEntityTypeConfigurationBase<TEntity, TIdentifierType> implementation, derives its engine from [UseDataSource] (carried by the engine base class) and its database name from [UseDatabase], then the entity's module namespace, then Default (EntityDataSourceRegistry.cs:164-177), resolves the logical name through the resolver, and stores the answer in a frozen dictionary Snapshot. It is built lazily and rescanned once on a lookup miss to pick up module assemblies loaded after first access, and it fails fast if two configurations claim one entity for different databases (EntityDataSourceRegistry.cs:138-141), an entity must live in exactly one database.

DataSourceService (behind IDataSourceService) is a thin facade over the registry that the rest of the codebase actually depends on. Its most-used method is HaveIncludeSupport(a, b), can EF .Include() from entity A to entity B?, which returns true only when both resolve to the same physical relational database (DataSourceService.cs:31-32). That single predicate is what G03's NavigationMetadataProvider consults to decide whether a navigation is a server-side JOIN or must fall back to a cross-source batch loader (INavigationPopulator<in TEntity>, ADR-002). The data-source layer thus quietly governs the read side of the whole application. This is dense [Rubric §8, Data Architecture] (assesses deliberate persistence design, multi-database ownership, routing) and [Rubric §7, Microservices Readiness] (each module's data is independently addressable, so a module can be lifted to its own service by repointing one config entry).

Cross-source relationships degrade automatically

Because EF cannot enforce a foreign key into another database, a relationship whose two ends resolve to different physical sources must be re-shaped. That is the job of CrossDataSourceDegradeConvention, a model-finalizing convention added in ApplicationDbContext.ConfigureConventions (ApplicationDbContext.cs:116). For each model being built it removes the cross-source FK constraint (keeping the scalar FK column plus a compensating index so the value is still queryable), ignores CLR navigation members that point at foreign entities, and removes the foreign entity types from this model entirely (CrossDataSourceDegradeConvention.cs:63-89). Runtime navigation across the boundary then flows through the INavigationPopulator batch loader, and consistency across databases is the outbox's job, not a transaction's. Critically, when every entity collapses onto one database (the monolith case) nothing is foreign and the convention is a structural no-op (CrossDataSourceDegradeConvention.cs:53-55), the model is byte-for-byte the single-database model. This is the model-building embodiment of [Rubric §7]: the seam exists and is tested, but it costs nothing when unused.

Building and caching contexts: the factory layers

Three factory surfaces turn a DataSourceKey into a live context, and they stack.

PhysicalDbContextFactory (behind IPhysicalDbContextFactory, singleton, never pooled because each instance carries per-source PhysicalDataSource state) is the bottom layer: Create(key) switches on key.Engine and constructs the matching concrete context (SQLServerDbContext, SqliteDbContext, or CosmosDbContext) directly, passing it that source's resolved PhysicalDataSource (PhysicalDbContextFactory.cs:34-45). It returns a raw, uncached context: engine selects the class, source name selects the database.

Above it, DbContextFactory (behind IDbContextFactory, scoped) is the per-request coordinator that the unit of work uses. It caches one context per DataSourceKey within a scope so every repository for the same database shares one change tracker (DbContextFactory.cs:34,52-67), and it owns the cross-source SaveChanges/transaction/migration/disposal coordination. Note the honesty in its transaction handling: with multiple sources, ExecuteInTransactionAsync opens a transaction per source and commits them sequentially, best-effort, there is no two-phase commit (DbContextFactory.cs:258-304); the outbox is the cross-source consistency mechanism (ADR-006). It also handles the gnarly SET IDENTITY_INSERT dance for importing entities with explicit primary keys (DbContextFactory.cs:132-224, grouping by table because SQL Server allows only one table's IDENTITY_INSERT on at a time) and wraps transactions in EF's execution strategy so they cooperate with the retry-on-failure policy SQLServerDbContext enables (SQLServerDbContext.cs:42-45).

Off to the side sits a third surface that exists purely to preserve EF Core's own IDbContextFactory<TContext> DI shape for consumers that predate the multi-database model (health checks and the like). DefaultSqlServerDbContextFactory, DefaultSqliteDbContextFactory, and DefaultCosmosDbContextFactory are thin adapters that each return their engine's Default-source context by delegating to PhysicalDbContextFactory (DefaultEngineDbContextFactories.cs:13-37), and ApplicationDbContextEFFactory picks among them by the DefaultDataSource/DataSource config key (ApplicationDbContextEFFactory.cs:35-41). These are not on the routing path a repository takes, they are a compatibility surface pinned to the Default database; entity-aware routing always goes through DbContextFactory.

The repository and unit-of-work layer

Application and query handlers never touch a DbContext directly; they go through repositories. The contracts live in the Application layer (so the domain stays persistence-free) and are deliberately ISP-segregated: IEntityReader<TEntity, TIdentifierType> (single-entity lookups), IEntityQuerier<TEntity, TIdentifierType> (collection queries, projections, counts), and IWriteRepository<TEntity, TIdentifierType> (adds, updates, concurrency-token application, bulk delete, save) compose up into IReadRepository<TEntity, TIdentifierType> and the full IRepository<TEntity, TIdentifierType>. A handler that only needs GetByIdAsync can depend on IEntityReader and signal its minimal data access, a textbook [Rubric §1, SOLID] Interface Segregation arrangement spelled out in the interface doc comments. The EF implementations are EFReadRepository<TEntity, TIdentifierType> and EFRepository<TEntity, TIdentifierType>, each optionally wrapped by a MiniProfiler timing decorator (EFReadRepositoryDecorator<TEntity, TIdentifierType> / EFRepositoryDecorator<TEntity, TIdentifierType>) chosen by RepositoryFactory only when UseMiniProfiler is configured (RepositoryFactory.cs:33-37); ReadRepositoryExtensions adds convenience helpers on top.

UnitOfWork (behind IUnitOfWork) is the seam that ties entity to database to context to repository together. GetRepository<TEntity, TId>() asks IDataSourceService for the entity's DataSourceKey, asks DbContextFactory for that database's context, and asks RepositoryFactory to build a repository over it, caching the result per entity type so all access in a scope shares one repository and one change tracker (UnitOfWork.cs:33-46). Its SaveChangesAsync, transaction, and identity-insert methods all delegate to the DbContextFactory, so "save the unit of work" really means "save every context this scope touched, each stamping its own audit fields." This is [Rubric §2, Design Patterns] (Repository + Unit of Work, idiomatic and earning their keep across multiple databases) and the data side of [Rubric §3, Clean Architecture].

The save pipeline: where cross-cutting concerns are enforced

Everything that makes a save "correct" happens in two EF SaveChanges interceptors, registered on every context in OnConfiguring (ApplicationDbContext.cs:94-96), so no handler ever writes audit or event code by hand.

  • AuditSaveChangesInterceptor stamps CreatedOn/By on Added entities and LastModifiedOn/By on Modified ones, reading the current user from the context's CurrentSaveUserId (set by DbContextFactory from ICurrentUserService just before the save). It explicitly marks CreatedOn/By un-modified on updates (AuditSaveChangesInterceptor.cs:54-55) so the creation stamp is immutable, and a null current user falls back to default as the system-operation sentinel (AuditSaveChangesInterceptor.cs:41). This realizes the primer's audit-fields promise.
  • DomainEventSaveChangesInterceptor is the producer side of the outbox. On SavingChanges it scans the change tracker for aggregate roots with buffered domain events, serializes each into an OutboxMessage row, and Adds it to the same context (guarded by the context's SupportsOutbox flag, which Cosmos turns off), so the outbox row and the business change commit in one atomic transaction (DomainEventSaveChangesInterceptor.cs:79-108). On SavedChanges it dispatches those events in-process via IDomainEventDispatcher and marks the rows processed; if in-process dispatch throws, it signals the OutboxProcessor to retry from the durable row (DomainEventSaveChangesInterceptor.cs:114-143). It keeps per-context state in a ConditionalWeakTable (DomainEventSaveChangesInterceptor.cs:31) so nothing leaks when a context is disposed. This is the heart of ADR-003 (dual dispatch); the full treatment lives in G04.

Soft-delete and optimistic concurrency are applied in OnModelCreating, not per save: ApplicationDbContext.ApplySoftDeleteFilters (ApplicationDbContext.cs:149-163) adds a global query filter e => !e.IsDeleted to every non-owned IAuditableEntity via expression trees, and ConfigureConcurrencyTokens (ApplicationDbContext.cs:176-196) maps RowVersion as a SQL Server rowversion (or a plain application-managed concurrency token on SQLite, which has no server-generated equivalent). The same OnModelCreating also configures the OutboxMessages and InboxMessages tables (InboxMessage, ADR-021) for every relational provider (ApplicationDbContext.cs:135-139). Together these touch [Rubric §8, Data Architecture] (transactions, soft-delete, audit, concurrency control, all centralized, all automatic) and the lifecycle half of [Rubric §30, Compliance & Data Governance] (soft-delete preserves rows for audit/undelete; ADR-005 covers true erasure).

Engine-agnostic entity configuration and the supporting cast

The per-entity half of database-per-service is the configuration hierarchy. EntityTypeConfigurationBase<TEntity, TIdentifierType> (over IEntityTypeConfigurationBase<TEntity, TIdentifierType>) handles the one cross-cutting concern every config needs, ignoring the in-memory DomainEvents collection; and (since ADR-018) the single engine-aware EntityTypeConfiguration<TEntity, TIdentifierType> base carries all the table/container/schema/key mapping logic, reading the [UseDataSource(<engine>)] attribute to pick the right conventions (EntityTypeConfiguration.cs:44-100). The three engine bases EntityTypeConfigurationSQLServer<TEntity, TIdentifierType>, EntityTypeConfigurationCosmos<TEntity, TIdentifierType>, and EntityTypeConfigurationSqlite<TEntity, TIdentifierType> (with matching IEntityTypeConfigurationSQLServer, IEntityTypeConfigurationCosmos, and IEntityTypeConfigurationSqlite markers) are now thin shims that just carry that attribute, so moving an entity between engines is a one-token base-class (or attribute) change with no configuration-body edits. As the primer notes, this is a real, tested capability: in production all current ADC entity configs use the …SQLServer base, but ADC Conference's Session to Cosmos / Room to SQLite move (ADR-018) is the staged first adoption (Common plumbing + portability tests in place), so treat Cosmos/SQLite as supported, exercised seams rather than dormant ones. ModelBuilderExtensions is the ApplyAllConfigurations helper that discovers and applies these per assembly, and NamespaceConventions derives the module name (the database name and SQL schema) from an entity's namespace (NamespaceConventions.cs:16-22). The notification entities ship their own configs, PushNotificationConfiguration and UserNotificationConfiguration.

A handful of smaller types round out the chapter. EncryptedStringConverter is an EF value converter giving AES-256-GCM authenticated encryption of a string column at rest, requiring a 32-byte key (EncryptedStringConverter.cs:40-52), which is [Rubric §11, Security] and [Rubric §30] (field-level encryption for sensitive data, the storage-side companion to IAnonymizable). CosmosIntIdValueGenerator generates integer keys for the Cosmos seam (CosmosIntIdValueGenerator.cs:16-26). ProfilingHelper and the keyless ValReturn<T> (ApplicationDbContext.cs:51, for raw-SQL scalar results) are small infrastructure utilities. IQueryableExecutor and EFQueryableExecutor abstract async IQueryable materialization behind a guard that also works over plain in-memory queryables (EFQueryableExecutor.cs:43), so application-layer code can stay EF-free and testable ([Rubric §14, Testability]). IDbSeeder / DbSeeder, the EnsureCreated/Migrate methods on IDbContextFactory, EntityConfigurationOptions, IEntityConfigurationAssemblyProvider and DefaultEntityConfigurationAssemblyProvider handle startup wiring of which assemblies hold configurations.

Design-time: one migrations project per database

Because each database has its own model and its own migrations assembly, dotnet ef must be told which database to target. DesignTimeDbContextHelper builds a SQLServerDbContext without the application's DI container for design-time commands: a downstream migrations project implements EF's IDesignTimeDbContextFactory<SQLServerDbContext> in a few lines, supplies its connection settings and configuration assemblies via DesignTimeDbContextOptions, and the helper stands up just enough of the routing chain (ExplicitAssemblyProvider, a fresh DataSourceResolver

  • EntityDataSourceRegistry, and the NullDomainEventDispatcher no-op) to produce a context whose model contains only the selected source's entities. The source is chosen by -- --datasource <Name> on the command line (DesignTimeDbContextHelper.cs:86-104). This is [Rubric §17, DevOps] and [Rubric §33, Developer Experience]: per-database migrations stay a one-command operation despite the multi-database topology.

Design-time is where migrations are authored; applying them on deploy is a separate decision, recorded in ADR-030 (startup sole-migrator). In Azure Container Apps every service host runs with DatabaseInitStrategy = Migrate and minReplicas: 1, so each service self-applies its own pending EF migrations at boot and is the sole migrator of its database, before the new revision serves traffic (the fan-out is DbContextFactory.MigrateAsync, DbContextFactory.cs:307-311). There is deliberately no deploy-step sqlcmd (or dotnet ef database update) backstop: that choice overrides the framework's documented None-for-production default, a divergence adopted after a startup-race incident in which an external migration step collided with the container's own boot migration on a fresh per-service database and wedged the deploy. The build-time dotnet ef migrations has-pending-model-changes gate only detects model drift; the container is what does the applying. This is the deploy-side [Rubric §29, Resilience & Business Continuity] counterpart to the design-time [Rubric §17] story above; see ADR-030 for the full rationale and trade-offs (and ADR-025 for the readiness gating that keeps traffic off a still-migrating replica).

End-to-end: one write, start to finish

To see the whole engine room at once, follow a single command that creates an aggregate. A handler asks IUnitOfWork for GetRepository<Session, int>(). The unit of work asks IDataSourceService which database Session lives in (the registry already knows: SQL Server, Conference), asks DbContextFactory for that database's cached SQLServerDbContext (built once via PhysicalDbContextFactory with the Conference connection string, its model filtered to Conference entities), and wraps it in an EFRepository. The handler calls AddAsync(session) and the command's TransactionalCommandDecorator (G05) calls IUnitOfWork.SaveChangesAsync. That fans to DbContextFactory.SaveChangesAsync, which sets the current user on the context and calls base.SaveChangesAsync. The interceptor pipeline fires: AuditSaveChangesInterceptor stamps the audit fields; DomainEventSaveChangesInterceptor serializes the session's domain events into OutboxMessage rows in the same context. EF commits the entity, its outbox rows, and the audit stamps in one transaction against the Conference database. On success the event interceptor dispatches in-process and marks the rows processed; the background OutboxProcessor is the safety net. The handler never saw a connection string, a DbContext, an audit field, or an outbox row, which is exactly the point. For the why behind each choice, ADR-006 (database-per-service), ADR-003 (outbox dual-dispatch), ADR-002 (navigation populators), and ADR-005 (soft-delete vs. erasure) are the primary references.

  • What it is: a three-value enum (CosmosDB, Sqlite, SQLServer) naming which database engine persists a given entity type. It shares a file with IDataSourceService, the service that reads it.
  • Depends on: nothing first-party (BCL only).
  • Concept introduced, database-per-service routing at the entity level. Crucially the members encode a capability, not just a name: the doc comments say Cosmos DB has "no cross-container JOINs" (IDataSourceService.cs:8), SQLite "supports JOINs within a single database file" (:11), and SQL Server has "full relational JOIN support" (:14).   • Walkthrough: three members declared in order with no explicit numeric values (IDataSourceService.cs:9,12,15), so they take their declaration ordinals (CosmosDB = 0, Sqlite = 1, SQLServer = 2). Each carries a one-line doc comment recording its JOIN capability.
  • Why it's built this way: putting the capability difference in the enum means IDataSourceService.HaveIncludeSupport can answer "can these two entities be JOINed?" with a lookup plus an equality check rather than scattered engine if chains.
  • Where it's used: paired with a database name inside DataSourceKey; resolved by IDataSourceService and its concrete DataSourceService; consumed by NavigationMetadataProvider to classify each navigation as an EF include or a manual load.
  • Caveats / not-in-source: because the ordinals are implicit, reordering these members would change the numeric value of each. Whether any site persists the enum by ordinal (which would make a reorder breaking) is Not determinable from source: I did not trace a serialization call site for it.

The [UseDatabase("Notification")] routing and the ToTable(..., "Notification") schema override are identical to the sibling above (see PushNotificationConfiguration for the full teaching of that shape); this section focuses on what differs, the indexing. Every entity in this framework is soft-deleted (rows set IsDeleted = true, never physically removed), so a plain unique index on (UserId, PushNotificationId) would forbid a user from ever re-receiving a notification whose prior inbox row was soft-deleted. A filtered index (HasFilter("[IsDeleted] = 0")) sidesteps that: the uniqueness constraint applies only to live rows, and the second, non-unique filtered index narrows the physical index to just the non-deleted rows the unread-count query actually scans. [Rubric §8, Data Architecture] covers indexing strategy as a first-class storage concern; the deliberate composite keys here map directly to the two access patterns (one-inbox-row-per-user-per-notification, and unread lookups). [Rubric §12, Performance & Scalability] assesses whether hot read paths are supported by targeted indexes; the (UserId, IsRead) filtered index is sized for the notification-badge count that every authenticated page issues. [Rubric §30, Compliance, Privacy & Data Governance] covers how governance rules like soft-delete are honored at the storage layer; the IsDeleted = 0 filter keeps soft-deleted inbox history out of both the uniqueness rule and the query index without hard-deleting it.
