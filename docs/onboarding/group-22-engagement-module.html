22. ADC Engagement Module (Session Bookmarks)

What this chapter covers. The Engagement module is ADC's smallest bookmarking bounded context, and it is the cleanest place in the whole codebase to watch a single aggregate travel through every Clean Architecture layer end to end: domain entity, use-case handlers, EF persistence, REST controller, the cross-service gRPC seam, and the Blazor UI. Its one persisted aggregate is the UserSessionBookmark: a row that says "this user has starred this session for their personal schedule." Around that one entity the module wires up the full create/list/delete lifecycle, a uniqueness invariant, a reactivate-instead-of-recreate twist, a cross-module count service exposed over gRPC, and a star-toggle UI surface that the Conference session pages reuse. The same UI library also hosts the attendee feedback pages (event/session question answers), but, as we will see, those pages are thin HTTP clients over the Conference module's feedback endpoints: the feedback domain lives in Conference, not here.

One scope note up front. The Engagement service host (MMCA.ADC.Engagement.Service) has grown a second capability, the conference-day live layer (live polls plus session Q&A, the "Happening Now" / session Live / presenter surfaces). That layer is a separate bounded concern with its own chapter (Group 23); this chapter stays on the bookmark aggregate and the feedback pages. Because the two concerns share the same module (one DbContext, one feature-flag class, one permission set, one UI module descriptor), a handful of the shared plumbing types in this chapter's membership are wired for both, and those are called out honestly where they appear rather than re-explained.

This is also the module that best demonstrates the modular-monolith to extractable-service thesis the codebase commits to (primer §2; ADR-007 and ADR-008). Engagement logically depends on Conference (you cannot bookmark a session that does not exist, or one that has been cancelled), and Conference logically depends on Engagement (a session detail page wants to show "42 people bookmarked this"). That bidirectional coupling is satisfied in-process when both modules live in one host and over gRPC when each runs as its own service, with the C# interface unchanged on both sides. Engagement is the textbook example of how that swap happens at the composition root and nowhere else.

The aggregate and its invariants, what gets persisted

The heart of the module is UserSessionBookmark (MMCA.ADC/Source/Modules/Engagement/MMCA.ADC.Engagement.Domain/UserSessionBookmarks/UserSessionBookmark.cs:17), a sealed AuditableAggregateRootEntity<TIdentifierType> holding just two scalars: a UserId and a SessionId, both with private setters (:20, :23). Both are cross-module foreign keys carried as plain values, not navigation properties: Identity owns the user, Conference owns the session, and under database-per-service (ADR-006) Engagement's database cannot hold a real FK to either. The entity is built through the UserSessionBookmark.Create(userId, sessionId) factory (:41) that returns Result<UserSessionBookmark> after Result.Combine-ing UserSessionBookmarkInvariants (both IDs must be non-default, UserSessionBookmarkInvariants.cs:11-15), so an invalid bookmark can never be constructed: the Result-pattern / factory discipline taught in the primer (§2) and group 02.

A design choice worth calling out early: instead of separate Created/Deleted domain events, every state change raises one event, UserSessionBookmarkChanged, carrying a DomainEntityState discriminator (Added / Deleted) per business rule BR-60 (UserSessionBookmark.cs:56 raises Added from the factory, :87 raises Deleted from the overridden Delete()). And rather than hard-deleting then re-inserting when a user re-stars a session they previously un-starred, the aggregate exposes a Reactivate() method (:67) that Undelete()s the soft-deleted row and re-raises the Added event (BR-135), preventing an unbounded pile-up of tombstoned rows for the same (user, session) pair. Soft-delete, audit, and domain-event collection all come from the Common base class; the module only adds the two FKs and the lifecycle nuances.

The use cases, a vertical slice per operation

Each operation is a self-contained slice under Application/UserSessionBookmarks/UseCases/{Operation}/ (vertical-slice architecture, primer §2):

  • Create, CreateBookmarkHandler (CreateBookmarkHandler.cs:17) is the most interesting handler in the module. It first calls Conference's ISessionBookmarkValidationService (ValidateSessionForBookmarkAsync, :30) to confirm the session is bookmarkable (not declined/cancelled per BR-49, not a service session per BR-91), a cross-module call that is in-process or gRPC depending on deployment. It then checks BR-21 (no duplicate active bookmark, an ExistsAsync, :37), looks for a soft-deleted row to revive (BR-135, a GetAllAsync with ignoreQueryFilters: true to see past the soft-delete filter, :50-56), and delegates the create-or-reactivate decision to the pure BookmarkManagementDomainService (:59). Its input is the CreateBookmarkRequest record, pre-screened by CreateBookmarkRequestValidator (FluentValidation) in the pipeline.
  • List, GetUserBookmarksHandler (GetUserBookmarksHandler.cs:17) returns a newest-first (OrderByDescending(b => b.CreatedOn), :57) PagedCollectionResult<T> with optional EventId filtering (BR-58, again resolving session IDs via the Conference validation service's GetSessionIdsByEventAsync, :39) and a server-side page-size cap of 500 (BR-11, Math.Min(query.PageSize, 500), :29). It pushes counting, ordering, and Skip/Take down to SQL via the repository's IQueryable, not into memory (:51-60).
  • Lookup, GetBookmarkedSessionIdsHandler (GetBookmarkedSessionIdsHandler.cs:12) returns a SessionId to BookmarkId dictionary in one projected read over the read-only repository (GetProjectedAsync, :22), which is exactly what the unified Sessions grid needs to render a filled/empty star per row without N round-trips.
  • Delete: there is no bespoke delete handler; the controller wires the generic DeleteEntityHandler<TEntity, TIdentifierType> via a DeleteEntityCommand<TEntity, TIdentifierType>, with ownership enforced before dispatch in the controller.

Reads also flow through the generic Common machinery: the module registers EntityQueryService<TEntity, TEntityDTO, TIdentifierType> for UserSessionBookmark (Application/DependencyInjection.cs:38) with no bespoke authorization specification, own-data scoping is applied imperatively at the controller edge instead (below). Entity-to-DTO conversion is a compile-time Mapperly mapper, UserSessionBookmarkDTOMapper ([Mapper], UserSessionBookmarkDTOMapper.cs:11, ADR-001), producing the UserSessionBookmarkDTO that crosses the wire. Because bookmarks have nothing to eager-load, the slice registers a NullNavigationPopulator<TEntity> (ADR-002, Application/DependencyInjection.cs:37), a deliberate no-op that satisfies the populator contract.

Persistence, one entity, one config, one DbContext

Engagement owns its own database (ADC_Engagement) under database-per-service. The bookmark EF configuration, UserSessionBookmarkConfiguration (UserSessionBookmarkConfiguration.cs:17), derives from EntityTypeConfigurationSQLServer<TEntity, TIdentifierType>, the base-class choice that pins the entity to SQL Server (primer §2, the engine-agnostic-entities note). Its one notable rule is a filtered unique index on (UserId, SessionId) with HasFilter("[IsDeleted] = 0") (:32-34), which enforces BR-21 at the database level while still permitting a soft-deleted historical row to coexist with a live one (the reason Reactivate() exists rather than re-insert). Note the config's own remark (:12-16): the Bookmark to Session FK is deliberately not declared in this configuration but deferred to the shared DbContext level, to avoid a cross-module Infrastructure to Domain dependency; and under database-per-service the two entities in fact live in separate databases (ADC_Engagement versus ADC_Conference), so no physical cross-database FK exists at all, consistent with module isolation.

The DbSet itself is declared on ModuleApplicationDbContext (ModuleApplicationDbContext.cs:16), an abstract context that adds Engagement's tables to the Common ApplicationDbContext and inherits auditing, soft-delete filters, and domain-event dispatch via interceptors. Its UserSessionBookmarks set is declared at :24. This is the first shared-plumbing seam: alongside the bookmark set, the same context also declares the live-layer DbSets (LivePolls, LivePollOptions, LivePollVotes, SessionQuestions, SessionQuestionUpvotes, :26-39), and the two live-layer EF configs that appear in this chapter's membership, SessionQuestionConfiguration and SessionQuestionUpvoteConfiguration, are auto-discovered from the same Infrastructure assembly. Those entities and their configs belong to the conference-day live layer (Group 23); they ride in this DbContext only because a module owns exactly one context (ADR-006), not because they are part of the bookmark aggregate.

The API surface and authorization

BookmarksController (BookmarksController.cs:32) exposes the REST surface (POST / GET list / GET session-ids / DELETE) on top of an ApiControllerBase that maps Error categories to HTTP status codes. The whole controller is gated two ways (:30-31): [FeatureGate(EngagementFeatures.SessionBookmarks)] (EngagementFeatures, value "Engagement.SessionBookmarks") that makes the entire feature vanish (404) when toggled off, and [Authorize(Policy = AuthorizationPolicies.RequireAuthenticated)] (BR-42, no anonymous bookmarking; the policy constant is AuthorizationPolicies). Authorization is resource-scoped, and the two read endpoints and the delete endpoint reach that scoping two different ways:

  • The list and session-ids GETs each carry [ServiceFilter(typeof(OwnerOrAdminFilter))] (:61, :82). Common's shared OwnerOrAdminFilter (ADR-033, drift D9) compares the userId query argument against the caller's user_id claim and rejects a mismatch unless the caller holds the Organizer bypass role. Its vocabulary is configured for ADC via OwnerOrAdminFilterOptions in AddModuleEngagementAPI (API/DependencyInjection.cs:44-49: OwnerClaimType = "user_id", BypassRole = RoleNames.Organizer, OwnerParameterName = "userId").
  • The DELETE keeps a bespoke inline check instead (:106-125): it asks ICurrentUserService whether the caller is an Organizer, and if not runs an ownership ExistsAsync over the query service, deliberately returning 404 rather than 403 so the API never reveals that a bookmark it will not let you touch exists. This is the same per-mutation inline pattern Store's OrdersController keeps.

The same AddModuleEngagementAPI block also declares the module's role-to-permission grants (:51-55): EngagementPermissions (a single engagement:live:manage capability, in this chapter's membership) is granted wholesale to the Organizer and Admin roles. That permission backs the live layer's [HasPermission(...)] endpoints (Group 23); the bookmark endpoints stay on RequireAuthenticated. It is documented here because the grant lives in the Engagement API registration this chapter covers.

The cross-service seam, gRPC extraction in miniature

Conference needs to display per-session bookmark counts, so Engagement offers a tiny outbound port: IBookmarkCountService (IBookmarkCountService.cs:8, in .Shared, the only project both modules can reference; its one method is GetBookmarkCountForSessionAsync). It has three implementations, and which one is resolved tells you the deployment topology:

  1. In-process, BookmarkCountService (BookmarkCountService.cs:11, Application layer) runs a CountAsync over the repository (:19-21). Used when Engagement and Conference share a host.
  2. Disabled stub, DisabledBookmarkCountService (DisabledBookmarkCountService.cs:7) returns 0, registered as a singleton by EngagementModule.RegisterDisabledStubs (EngagementModule.cs:29-30) so a Conference-only host still resolves the interface (returns "no bookmarks") instead of failing DI.
  3. gRPC adapter, BookmarkCountServiceGrpcAdapter (BookmarkCountServiceGrpcAdapter.cs:14, in .Contracts) implements the same interface by calling a generated gRPC client; on the server side, BookmarkCountsGrpcService (BookmarkCountsGrpcService.cs:12) bridges the wire call back to the in-process IBookmarkCountService.

The clever part is the swap, in the Contracts-project DependencyInjection (Source/Services/MMCA.ADC.Engagement.Contracts/DependencyInjection.cs): its AddEngagementBookmarkCountClient extension (:41) uses services.Replace(...) (:47), not TryAdd, so it overwrites whichever of the first two registrations is present, guaranteeing the gRPC adapter wins regardless of whether the real impl or the stub got there first (it must run after ModuleLoader.DiscoverAndRegister, as its own doc comment states, :34-37). The C# interface never changes; only the registration does. This is exactly the ADR-007 "*.Contracts + typed clients" pattern, and the symmetric inbound dependency, Engagement consuming Conference's ISessionBookmarkValidationService, is what makes the Conference-Engagement pair bidirectional (the AppHost deliberately omits a reciprocal WaitFor to avoid a startup deadlock; transient "peer not ready" errors self-heal via the Polly pipeline baked into AddTypedGrpcClient). EngagementModule itself implements IModule declaring Dependencies => ["Conference"] and RequiresDependencies => true (EngagementModule.cs:19,22), so the ModuleLoader registers it in topological order after Conference.

The eventing footprint

UserSessionBookmarkChanged is a BaseDomainEvent (UserSessionBookmarkChanged.cs:15-20) collected on the aggregate and dispatched after SaveChangesAsync. Within the module it carries the add/delete signal for any interested handler; because the count read is a live COUNT rather than a denormalized counter, there is no in-module projection that must consume it today, its presence is the seam for future integration consumers (for example a cross-service "popular sessions" projection over the outbox, ADR-003). The bookmark FKs being plain scalars (no cross-database FK) is precisely why consistency between Engagement and Conference flows through this event plus the gRPC validation call, never through the database.

The UI surface, star toggles, and feedback pages that are not really Engagement's

The MMCA.ADC.Engagement.UI Razor Class Library contributes two distinct kinds of bookmark-and-feedback UI (a third kind, the live-layer pages, belongs to Group 23). First, the bookmark clients: BookmarkService (BookmarkService.cs:14, the attendee's own personal-schedule page, implementing IBookmarkUIService) and SessionBookmarkUIService (SessionBookmarkUIService.cs:13, implementing the cross-module ISessionBookmarkUIService that the Conference session grid calls to render and toggle stars inline). Both extend Common's AuthenticatedServiceBase, attaching the bearer token and a Polly retry policy, and talk to the Gateway's bookmarks routes, meaning the UI goes back through the public REST contract, not a back-channel.

Second, and this is the part to not be misled by, the same library hosts the attendee feedback pages, EventFeedback (EventFeedback.razor.cs:16) and SessionFeedback (SessionFeedback.razor.cs:16), with their IEventFeedbackUIService / ISessionFeedbackUIService clients (both interfaces declared in Services/IFeedbackUIService.cs:21,43) and the question-loading QuestionLookupService (implementing IQuestionLookupService, Services/IFeedbackUIService.cs:10). These pages render dynamic Rating/Text/Email questions and submit per-question answers using a POST-as-upsert pattern (BR-107), with an inner AnswerState holder (EventFeedback.razor.cs:263) for two-way binding and a StateHasChanged()-before-navigate dance (:169-173) so the unsaved-changes guard sees the cleared dirty flag. But their services POST to Conference's eventquestionanswers / sessionquestionanswers endpoints (EventFeedbackService / SessionFeedbackService, EventFeedbackService.cs:17) and traffic in Conference's DTOs, resolving the parent event/session name through Conference's own lookup and entity services rather than any Engagement type; the feedback question/answer aggregates belong to the Conference bounded context (group 17). Engagement.UI is simply the convenient presentation home for these attendee-facing pages; treat them as a UI-only adjacency, not part of the bookmark domain.

EngagementUIModule (an IUIModule, EngagementUIModule.cs:14) is the module's Blazor descriptor and the second shared-plumbing seam. It contributes no bookmark or feedback nav item (those pages are reached from the Conference session/event pages), but it does contribute the live-layer "Happening Now" nav item (NavItems, :18-21) and the invisible LiveEventListener layout component (LayoutComponentTypes, :23), both live-layer concerns (Group 23), and it exposes the assembly for Blazor component discovery (:25). The per-host UI registration lives in the UI-layer DependencyInjection (AddEngagementUI, UI/DependencyInjection.cs:21), which AddScopeds the bookmark and feedback clients (:24-30) alongside the live-layer clients, with route constants (feedback and live) in EngagementRoutePaths (EngagementRoutePaths.cs:6).

How the layers register, and how to read the rest of this chapter

Each layer's wiring lives in its own DependencyInjection (Application/Infrastructure/API/UI/Contracts, several share the name, distinguished by namespace and file path), all using C# extension(IServiceCollection) members (primer §4). The API AddEngagementModule chains the three backend layers (API/DependencyInjection.cs:23-30); the Application registration TryAdds the domain service (:34), the query service (:38), the delete handler (:39), and the in-process count service (:42), then leans on convention scanning (ScanModuleApplicationServices, Application/DependencyInjection.cs:58) to pick up handlers, validators, and mappers; the AssemblyReference/ClassReference markers exist solely as scan anchors. The per-type sections that follow are ordered by dependency Level: the leaf records, markers, and feature/route/permission constants first; then validators, DTOs, and UI service interfaces; then the modules, pages, and services; then the aggregate, controller, domain service, persistence, handlers, and finally the gRPC contract types that close the extraction seam.
