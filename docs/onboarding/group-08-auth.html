8. Authentication & Authorization

This chapter covers how MMCA.Common answers two questions on every request: who are you (authentication) and are you allowed to do this (authorization). The framework ships the whole pipeline, JWT minting and validation, password hashing, brute-force protection, current-user/claims extraction, role-based policies, resource-ownership filters, and the browser-side HttpOnly session-cookie machinery that lets a Blazor Web App survive a hard refresh, and the two consumer apps (ADC, Store) wire it up with only a handful of registration calls. Everything here lives in MMCA.Common and is designed so a module can be lifted into its own microservice and still validate the monolith's tokens (ADR-004, ADR-007/008). Read the Result pattern and Error sections first, auth surfaces all of its expected failures (bad password, lockout, expired session) as Result.Failure, never exceptions.

The token lifecycle, minting, validating, refreshing

The center of the system is the JWT access token. ITokenService defines the contract and TokenService (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/TokenService.cs:23) implements it. On a successful login the Identity module asks GenerateAccessToken(userId, email, role, fullName) (TokenService.cs:67) for a short-lived signed JWT and GenerateRefreshToken() (TokenService.cs:104) for a 64-byte cryptographically-random opaque string; both, plus the access-token expiry, are returned to the caller as an AuthenticationResponse (MMCA.Common/Source/Core/MMCA.Common.Shared/Auth/AuthenticationResponse.cs:10). The access token carries the claims the rest of the system reads, sub/user_id (the numeric UserIdentifierType), name, email, and role, plus a jti and iat for traceability (TokenService.cs:78-84). The token's iat/nbf/exp timestamps are read from an injected TimeProvider (TokenService.cs:74) rather than DateTime.UtcNow, so token minting is deterministically testable. Because access tokens are short-lived, the refresh token is the long-lived credential: when an access token expires, the client posts the expired-token + refresh-token pair to auth/refresh and gets a rotated pair back.

The signing story is what makes microservice extraction possible. TokenService materializes its signing and validation keys once at construction (TokenService.cs:47-64) and supports two algorithms selected by IJwtSettings.SigningAlgorithm. In monolith mode the default is HS256 (symmetric HMAC-SHA256) reusing a Base64 secret (BuildHmacCredentials, TokenService.cs:160), the same process signs and validates, so a shared secret is fine. In microservice mode it switches to RS256 (asymmetric RSA-SHA256, BuildRsaCredentials, TokenService.cs:174): the issuer signs with its private key, and other services validate with the matching public key. That is the whole point of ADR-004's dual-fetch / JWKS design, see the next paragraph.

GetPrincipalFromExpiredToken (TokenService.cs:117-151) is the subtle method behind refresh. It validates issuer, audience, and signing key but deliberately skips lifetime validation (ValidateLifetime = false, TokenService.cs:125, with the CA5404 warning explicitly suppressed) because its job is to read claims out of an already-expired token. Critically, it pins ValidAlgorithms to the configured algorithm (TokenService.cs:130) and then re-checks the token's alg header against that algorithm (TokenService.cs:139-140) to block a token-substitution attack, an attacker cannot present an HS256 token signed with the (public) RSA key in place of a real RS256 token. TokenService is IDisposable and owns its RSA handles, releasing them on disposal (TokenService.cs:154-158).

Cross-service validation, JWKS (ADR-004)

When a module is extracted, it no longer shares the issuer's process or secret, so it needs the issuer's public signing key to validate incoming RS256 tokens. IJwksProvider and its implementation RsaJwksProvider (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Auth/RsaJwksProvider.cs:15) materialize a JsonWebKeySet from a PEM-encoded RSA public key (configured via JwksSettings, lazily built and cached, RsaJwksProvider.cs:17). The API layer maps that key set at /.well-known/jwks.json and an OIDC discovery document at /.well-known/openid-configuration; a downstream service points its JwtBearer handler at the issuer's authority (AddForwardedJwtBearer), fetches the discovery document, follows jwks_uri, and validates the signature against the published key, holding only the public half. When JWKS publishing is disabled (the default for a monolith) or no key is configured, RsaJwksProvider returns an empty key set so the endpoint stays queryable rather than erroring (RsaJwksProvider.cs:24-33). This is the charter's "dual-fetch": discover the key set from the issuer's discovery document, and degrade gracefully when it isn't there. Because only Identity ever holds signing key material, a compromised non-issuer service cannot forge tokens and key rotation is publish-once at the issuer, which is the group's [Rubric §7, Microservices Readiness] seam: the trust boundary lives at the transport edge, and the application code above it is identical in the monolith and as a service.

Passwords, hashing and verification

IPasswordHasher / PasswordHasher (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/PasswordHasher.cs:12) hash and verify passwords with PBKDF2-HMAC-SHA512 at 600,000 iterations (OWASP 2023 guidance), a 32-byte random salt, and a 64-byte output (PasswordHasher.cs:14-24). Verification is salt-length-discriminated for backward compatibility: a 128-byte salt means a legacy HMAC-SHA512 hash and is verified that way, otherwise the PBKDF2 path runs (PasswordHasher.cs:52-54). The final comparison uses CryptographicOperations.FixedTimeEquals to defeat timing side-channels (PasswordHasher.cs:58). The hasher only stores (byte[] Hash, byte[] Salt); the iteration count and algorithm live in the verifier, which is why legacy hashes can still be checked, and because HashPassword always writes a fresh 32-byte-salt PBKDF2 record (PasswordHasher.cs:34-42), a legacy 128-byte-salt account silently migrates to the modern format the next time its password is set. This salt-length-versioned, no-rehash-flag scheme is the subject of ADR-032. [Rubric §11, Security] assesses how well secrets, crypto, and attack surface are handled; the deliberate iteration count, constant-time compare, and explicit alg re-check above are concrete §11 strengths.

Brute-force and rate-limit protection

ILoginProtectionService / LoginProtectionService (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Auth/LoginProtectionService.cs:18) sits in front of the login and register flows, protecting exactly the anonymous auth surface that ADR-019's authenticated-only limiter exempts (this is ADR-029). Its state lives entirely in ICacheService (Redis in production, in-memory locally, never a table, per ADR-026) so the counters work across a scaled-out host. Failed logins are counted per email; once MaxFailedAttempts is exceeded it applies an exponential backoff lockout, 1 << excessAttempts seconds (the shift exponent clamped to 30 so a persistent attacker cannot wrap the lockout TTL negative via C#'s 5-bit shift masking), capped at MaxLockoutSeconds (LoginProtectionService.cs:50-60). A successful login clears the counter and lockout (ResetFailedAttemptsAsync, LoginProtectionService.cs:65). Registration is rate-limited per client IP within a configurable window (CheckRegistrationRateLimitAsync, LoginProtectionService.cs:72), which is why IAuthenticationService.RegisterAsync takes an optional ipAddress. All thresholds are config-bound via LoginProtectionSettings (defaults: 5 failed attempts, 300s lockout cap, 10 registrations per IP per hour, LoginProtectionSettings.cs:18-43). Lockouts and rate-limit hits return Result.Failure with Auth.TooManyAttempts / Auth.RegistrationRateLimitExceeded errors rather than throwing. This brute-force control is one layer of the layered rate-limiting design (ADR-019): the always-on global limiter (AddCommonRateLimiting, G12) caps only authenticated callers and exempts anonymous/infrastructure traffic, output caching absorbs anonymous public reads, and this LoginProtectionService handles login/registration brute-force, three controls shaped to three distinct threats rather than one blunt per-IP cap. [Rubric §29, Resilience & Business Continuity] covers this: distributed-cache-backed lockout and IP caps blunt credential-stuffing and registration-spam without a dedicated WAF.

The authentication workflows

The login, registration, refresh-rotation, and revocation flows are hoisted once into AuthenticationServiceBase<TUser> (MMCA.Common/Source/Core/MMCA.Common.Application/Auth/AuthenticationServiceBase.cs:34), a template-method base (constrained to an AuditableAggregateRootEntity<UserIdentifierType> that also implements IAuthUser) that owns the invariant workflow and defers everything app-specific to abstract or virtual hooks (FindUntrackedByEmailAsync, EmailExistsAsync, CreateUser, CreateAccessToken, and the optional login/refresh gates, AuthenticationServiceBase.cs:275-310). The public port is IAuthenticationService (MMCA.Common/Source/Core/MMCA.Common.Application/Auth/IAuthenticationService.cs:11), whose ExternalLoginAsync default member returns a not-supported failure (IAuthenticationService.cs:66) so OAuth account linking stays app-level. LoginAsync (AuthenticationServiceBase.cs:63) follows an untracked-then-tracked dual-fetch: an untracked read verifies credentials so failed logins never pay EF change-tracking cost (AuthenticationServiceBase.cs:86), then a tracked re-fetch (AuthenticationServiceBase.cs:110) stamps the rotated refresh token only once credentials pass. RefreshTokenAsync (AuthenticationServiceBase.cs:180) enforces BR-206 reuse detection: a presented refresh token that does not match the stored one is treated as possible theft, the stored token is revoked (AuthenticationServiceBase.cs:223) to force re-authentication. The IAuthUser contract (MMCA.Common/Source/Core/MMCA.Common.Domain/Auth/IAuthUser.cs:10) is deliberately minimal, password material plus refresh-token state and its UpdateRefreshToken/RevokeRefreshToken mutators (IAuthUser.cs:27,30), nothing about profile or claims. [Rubric §1, SOLID] and [Rubric §2, Design Patterns] both live here: the workflow is closed for modification but open to each app's User factory and claim set through hooks.

The request/response DTOs are thin record structs in MMCA.Common.Shared.Auth, LoginRequest, RegisterRequest, RefreshTokenRequest, ChangePasswordRequest, OAuthCodeExchangeRequest, the device-aware AuthenticationRequest for MAUI clients, and AuthenticationResponse, all in the Shared layer so both the API and the Blazor/WASM client reference the same shapes. The FluentValidation validators are bundled into a single parameter object, AuthenticationValidators (to keep the app service under its constructor-arity guardrail), wrapping LoginRequestValidator and RefreshTokenRequestValidator; the login validator is deliberately shallow (non-empty + email-format only) so the service can return a single generic credential error without leaking which field was wrong (LoginRequestValidator.cs:6-9). [Rubric §9, API & Contract Design] looks at how request/response contracts are modeled and shared; keeping these DTOs in Shared is the §9 pattern that lets one record type serve controller binding and the typed UI client.

Reading "who am I", current user & claims

Once a request is authenticated, the rest of the application reads identity through ICurrentUserService / CurrentUserService (MMCA.Common/Source/Core/MMCA.Common.Infrastructure/Services/CurrentUserService.cs:12), a scoped service over IHttpContextAccessor. It exposes the ClaimsPrincipal, the parsed UserId and Role, a generic GetClaimValue<T> for module-specific claims (e.g. speaker_id, customer_id) that keeps Common decoupled from any module's claim vocabulary, and a case-insensitive IsInRole. Both UserId and Role are computed once per request via Lazy<T> because the service is scoped (CurrentUserService.cs:17-24). The custom user_id claim it reads is exactly the one TokenService emits, the two are a matched pair. The SignalR side has its own tiny reader, ClaimBasedUserIdProvider, which pulls the same user_id claim so Clients.User(id) routes push notifications to the right connections. ISoftDeletedUserValidator is an application-layer port (implemented in the Identity module) that lets auth code ask "has this user been soft-deleted?" without a cross-module domain reference, the inversion keeps Common free of Identity's entities (ADR-005 soft-delete; [Rubric §3, Clean Architecture]).

Authorization, roles, policies, and ownership

Role names are canonicalized in RoleNames (Organizer/Attendee/ContentEditor for ADC, Admin/Customer for Store) so no layer hard-codes role strings, and RoleValue is the dependency-free base each app subclasses for a validated, case-insensitive role value object. AuthorizationPolicies declares the matching policy-name constants (RequireOrganizer, RequireAttendee, RequireAdmin, RequireAuthenticated), constants rather than an enum because [Authorize(Policy = ...)] needs compile-time strings, and AuthorizationExtensions's AddAuthorizationPolicies() (an extension(IServiceCollection) member) wires each constant to an ASP.NET Core policy via RequireRole/RequireAuthenticatedUser (AuthorizationExtensions.cs:24-32). For resource-level authorization, where role isn't enough because a Customer may only touch their own cart/order, OwnerOrAdminFilter is an IAsyncActionFilter (MMCA.Common/Source/Presentation/MMCA.Common.API/Authorization/OwnerOrAdminFilter.cs:20) that lets the bypass role through, otherwise compares the caller's owner claim against the resource identifier taken from the route or a model-bound argument and returns 403 on mismatch (OwnerOrAdminFilter.cs:25-53). Its vocabulary (owner claim, bypass role, parameter name) is host-configurable through OwnerOrAdminFilterOptions, whose defaults (customer_id / Admin / id) preserve the original hard-coded behavior, so a host with a different ownership vocabulary reconfigures rather than forks the filter (ADR-033). OwnershipHelper holds the shared IsAdmin bypass it reuses (OwnershipHelper.cs:17) plus the GetOwnershipSpecification helpers that scope collection queries to the caller's own rows. This row/resource-level ownership axis sits beside the ADR-020 RBAC/permission model rather than inside it: the filter is applied today by Store's ShoppingCartsController (MMCA.Store/Source/Modules/Sales/MMCA.Store.Sales.API/Controllers/ShoppingCartsController.cs:38) and CustomersController (MMCA.Store/Source/Modules/Identity/MMCA.Store.Identity.API/Controllers/CustomersController.cs:32). [Rubric §11, Security] again: this is defense-in-depth, coarse role policies at the edge, fine-grained ownership checks per resource.

Permission-based authorization, capabilities over role names

A newer layer sits beside the role policies: endpoints can require a fine-grained permission (a capability such as "sessions:manage") instead of a role name, via the HasPermissionAttribute ([HasPermission("...")], an AuthorizeAttribute whose policy name is perm: + the permission through PermissionPolicy). The point is decoupling: an endpoint declares the capability it needs, and which roles confer that capability is decided in one place, the IPermissionRegistry. Adding a role or re-shaping who-can-do-what becomes a registry change, not an edit scattered across every [Authorize(Roles = ...)] site. Each module declares its own role to permission grants by calling AddPermissions(...) (an AuthorizationExtensions member, AuthorizationExtensions.cs:54) and accumulating them into a shared PermissionRegistryBuilder, whose Grant is additive and unioning; grants from every module fold into one immutable PermissionRegistry. At request time, PermissionPolicyProvider materializes a "perm:<permission>" policy on demand (PermissionPolicyProvider.cs:31, no per-permission registration) and attaches a PermissionRequirement; PermissionAuthorizationHandler then grants it when the principal either carries the permission as an explicit AuthClaimTypes.Permission claim or holds a role that the registry maps to it (PermissionAuthorizationHandler.cs:29-30). The mechanism is backward-compatible and opt-in (ADR-020): AddAuthorizationPolicies() wires it for free, the named role policies keep working unchanged (the provider falls through to the default for any non-perm: policy name), and an app that never calls AddPermissions simply has no permission grants. This is the model that lets ADC introduce a granular ContentEditor role holding only the session-catalog-curation subset of an organizer's capabilities, a distinction the registry expresses centrally that per-endpoint role lists could not. [Rubric §11, Security] / [Rubric §10, Cross-Cutting Concerns]: capability-based authorization keeps the authz model auditable in one registry rather than drifting across controllers.

The HttpOnly session-cookie subsystem (Blazor Web App)

The largest moving part here exists to solve one specific browser problem: a Blazor Server + WebAssembly app keeps its access token in WASM memory, which is empty on a cold navigation (new tab, F5, deep link). During server-side prerender there is no token, so [Authorize] pages would bounce to /login even for a logged-in user. The session-cookie subsystem (ADR-022) fixes this by mirroring the tokens into two HttpOnly cookies, mmca_auth_access and mmca_auth_refresh, that the browser sends on every request including the SSR GET. The pieces:

  • SessionCookieEndpoints maps three endpoints (SessionCookieEndpoints.cs:15): POST/DELETE /auth/session-cookie let the WASM client seed/clear the cookies at login/logout, and POST /auth/session/token is the same-origin "validate-or-refresh" the browser polls to hydrate its in-memory access token, the refresh token is never serialized to JS. The cookie names live here as constants (SessionCookieEndpoints.cs:17-18); SessionCookieRequest is the seed payload.
  • SessionCookieJar centralizes writing/deleting the cookies with the right HttpOnly/SameSite/Secure flags (SessionCookieJar.cs:30, Secure relaxed in Development), so no path can set them inconsistently.
  • CookieTokenReader reads the access token during SSR, preferring a freshly-rotated token stashed on HttpContext.Items over the inbound cookie (CookieTokenReader.cs:19), so a refresh that happened earlier in the same request is visible to authentication.
  • SessionCookieAuthenticationHandler is an ASP.NET Core auth scheme (SchemeName = "SessionCookie", SessionCookieAuthenticationHandler.cs:32) that reads the cookie JWT, checks expiry (SessionCookieAuthenticationHandler.cs:55), and projects its claims into HttpContext.User for the prerender pass. It deliberately does not re-validate the signature (SessionCookieAuthenticationHandler.cs:18-23): the cookie was minted by the UI host from a token the API already validated, and the API still fully validates every real API call, this handler only surfaces identity for SSR. Its HandleChallengeAsync redirects to /login?returnUrl=…. SessionCookieAuthenticationExtensions registers it.
  • ICookieSessionRefresher / CookieSessionRefresher implement server-side "validate-or-refresh" (CookieSessionRefresher.cs:43): if the access cookie's JWT is still valid (with a 30s clock-skew margin) return it; otherwise call the API's auth/refresh server-to-server (so the refresh token never reaches browser JS), write the rotated pair back as cookies via the jar, and stash the new access token on HttpContext.Items for the current request. It is a singleton guarded by a SemaphoreSlim plus a 10-second rotation-grace IMemoryCache keyed by the old refresh token (CookieSessionRefresher.cs:50-53), so a thundering herd of concurrent cold navigations collapses into a single rotation (single-flight) instead of repeatedly rotating the token. SessionTokenResult and SessionTokenResponse are its internal/wire shapes, note the response carries the access token only.
  • CookieSessionRefreshMiddleware runs before UseAuthentication() and, only on GET navigations (Accept: text/html, CookieSessionRefreshMiddleware.cs:29-32), invokes the refresher so an expired-access/valid-refresh session is silently refreshed before SSR authorization runs, instead of bouncing to login. It is gated to navigations so it never fires on static assets or XHR/API calls. CookieSessionRefreshMiddlewareExtensions's UseCookieSessionRefresh() is the registration helper.

The /auth/session/token endpoint and the cookies are CSRF-hardened by defense-in-depth: HttpOnly stops JS from reading the tokens, SameSite=Lax blocks cross-site cookie attachment, and the endpoint additionally rejects requests whose Sec-Fetch-Site header is cross-site (SessionCookieEndpoints.cs:65-67). [Rubric §26, Front-End Security] assesses exactly this, token storage, XSS/CSRF exposure on the client; keeping refresh tokens out of JS entirely and layering SameSite + Sec-Fetch-Site is the §26 story for this app. [Rubric §19, State Management] is also in play: the in-memory-token-plus-rehydration design is how auth state survives a render-mode transition and a hard refresh.

How it all wires together at runtime

A logged-in interactive request flows: client attaches the bearer access token → ASP.NET Core's JwtBearer (HS256 locally, RS256 + JWKS across services) validates it → CurrentUserService exposes the parsed identity → role policies from AuthorizationExtensions, permission requirements via HasPermissionAttribute, and, where needed, OwnerOrAdminFilter gate the action. A cold full-page navigation to a protected Blazor page instead flows: GET arrives with HttpOnly cookies → CookieSessionRefreshMiddleware refreshes if the access cookie expired → SessionCookieAuthenticationHandler reads the (possibly just-refreshed) cookie via CookieTokenReader and populates HttpContext.User → SSR [Authorize] passes → once WASM boots it polls /auth/session/token to hydrate its in-memory token for subsequent API calls. The consumer apps' API hosts expose the actual login/register/refresh HTTP surface through AuthControllerBase (G12), which delegates to IAuthenticationService and maps Result failures to HTTP status codes via ApiControllerBase. The whole subsystem is transport-agnostic where it counts (the JWKS + RS256 seam) so a module's auth keeps working unchanged when it moves behind the gateway (ADR-004, ADR-007/008).

AuthorizationPolicies

  • Walkthrough
    • [SuppressMessage("SonarAnalyzer.CSharp", "S2339", ...)] (AuthorizationPolicies.cs:10): SonarAnalyzer flags public const in a non-abstract class; the suppression is justified inline because attribute arguments must be compile-time constants (static readonly would not compile at the attribute site), so const is the only option, a documented suppression, not a blanket disable ([Rubric §15, Best Practices & Code Quality]).
    • Four public const string members (AuthorizationPolicies.cs:14,17,20,23), each documented with the role/condition it requires.

  • Where it's used: [Authorize(Policy = ...)] on controller actions and minimal-API endpoints. OwnerOrAdminFilterOptions

MMCA.Common.API · MMCA.Common.API.Authorization · MMCA.Common/Source/Presentation/MMCA.Common.API/Authorization/OwnerOrAdminFilterOptions.cs:11 · Level 0 · class (sealed)

  • What it is: the host-configurable vocabulary that OwnerOrAdminFilter reads to decide which claim carries the caller's owner id, which role bypasses the check, and which request parameter names the resource being accessed. Three mutable string settings, all defaulted.
  • Depends on: nothing first-party (a plain options POCO). Bound by the ASP.NET Core options system and consumed by OwnerOrAdminFilter.
  • Concept introduced, options-driven authorization vocabulary. [Rubric §11, Security] (assesses whether a security control generalizes without being rewritten per host) and [Rubric §16, Maintainability & Evolvability] (a policy externalized to configuration rather than hard-coded). The same owner-or-admin mechanism must serve hosts with different claim and role vocabularies, Store reasons about a customer_id claim and an Admin bypass role, while ADC's Engagement module reasons about a user_id claim and an Organizer bypass role. Rather than fork the filter per host, the filter reads its vocabulary from this options object; the defaults reproduce the original hard-coded behavior exactly (OwnerOrAdminFilterOptions.cs:5-10), so existing hosts need no configuration and a differing host overrides via services.Configure<OwnerOrAdminFilterOptions>(...). This is the seam ADR-033 (ADRs/033-resource-ownership-authorization.md) adds so one filter covers both apps.
  • Walkthrough: three public string { get; set; } members, each with its default and doc comment.
    • OwnerClaimType (OwnerOrAdminFilterOptions.cs:14), default "customer_id", the claim carrying the caller's owner id.
    • BypassRole (OwnerOrAdminFilterOptions.cs:17), default "Admin", the role exempt from the ownership check.
    • OwnerParameterName (OwnerOrAdminFilterOptions.cs:24), default "id", the request parameter compared against the owner claim, either a route value (/customers/{id}) or a model-bound query/body argument (?userId=42).
  • Why it's built this way: mutable get; set; (not required init) is the ASP.NET Core options convention, so a host mutates only the settings it cares about and inherits the rest. Keeping the vocabulary in one bindable object means the security behavior is auditable in configuration and the filter code stays vocabulary-agnostic.
  • Where it's used: injected as IOptions<OwnerOrAdminFilterOptions> into OwnerOrAdminFilter; left at defaults by Store, overridden to user_id / Organizer by ADC's Engagement module (ADR-033).

PermissionPolicy

MMCA.Common.API · MMCA.Common.API.Authorization · MMCA.Common/Source/Presentation/MMCA.Common.API/Authorization/PermissionPolicy.cs:9 · Level 0 · class (static)

  • What it is: the naming convention that maps a permission string to an ASP.NET Core policy name. A permission such as "sessions:manage" is exposed to the authorization system as the policy name "perm:sessions:manage".
  • Depends on: nothing first-party (BCL only).
  • Concept introduced, encoding a capability inside a policy name. [Rubric §11, Security]. ASP.NET Core authorization is policy-name driven, but pre-registering one named policy per permission would be unmanageable. Instead a single prefix ("perm:") marks a name as a permission policy, and PermissionPolicyProvider recognizes the prefix and materializes the policy on demand. This static class is the one place the prefix and the encode rule live, so the attribute that writes the name (HasPermissionAttribute) and the provider that reads it can never disagree.
  • Walkthrough: const string Prefix = "perm:" (PermissionPolicy.cs:12) and static string NameFor(string permission) => Prefix + permission (PermissionPolicy.cs:17). The provider does the inverse with a range slice.
  • Why it's built this way: centralizing the convention removes the drift risk between the writer and the reader of the policy name, the same single-source-of-truth reasoning as AuthorizationPolicies. The perm:{permission} on-demand-policy convention is the load-bearing detail of ADR-020 (ADRs/020-permission-based-authorization.md): it removes the need for a per-permission named-policy registration.
  • Where it's used: by HasPermissionAttribute (to build the policy name it passes to the base [Authorize]) and PermissionPolicyProvider (to recognize and decode the name).

PermissionRequirement
