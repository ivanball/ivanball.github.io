21. ADC Conference - UI

What this chapter covers. This is the consumer half of the "write-once UI, render everywhere" story (primer §2): the Blazor pages and per-page HTTP services that turn the Conference REST surface (G20) into the screens an organizer, a speaker, or an anonymous attendee actually touches. Everything here lives in the per-module Razor Class Library MMCA.ADC.Conference.UI, which, like every consumer UI, assembles the reusable primitives taught in G15 (Common UI Framework) into concrete pages. There is almost no new infrastructure here; the value is in seeing how a real, nine-area feature surface (events, sessions, speakers, categories, questions, rooms, feedback, public browsing, session-selection) is composed from the framework's list-page base, typed HTTP service base, theme, and module system. The headline lens is [Rubric §18, UI Architecture & Component Design] (assesses component reuse, separation of presentation from data access, and a coherent composition model); the per-area pages are pure composition over a small set of bases. Because the very same Razor components compile into the Blazor Server, WebAssembly, and .NET MAUI hybrid hosts, this one library renders the conference across Web, Android, iOS, macOS, and Windows with no per-platform reimplementation. [Rubric §22, Responsive & Cross-Browser/Device].

The layering inside the UI: a page never touches HttpClient. Each page is a .razor + .razor.cs code-behind pair that depends only on a UI service interface, never on HttpClient, never on the API's internals. The CRUD-shaped entities (events, sessions, speakers, categories, questions, rooms) each get a service that derives from Common's EntityServiceBase<TEntityDTO, TIdentifierType> and exposes the IEntityService<TEntityDTO, TIdentifierType> contract: EventService, SessionService, SpeakerService, ConferenceCategoryService, CategoryItemService, QuestionService, and RoomService. They inherit GetAllAsync/GetPagedAsync/GetByIdAsync/AddAsync/UpdateAsync/DeleteAsync and only add the handful of bespoke verbs the conference needs, for example EventService layers PublishAsync/UnpublishAsync and RefreshFromSessionizeAsync onto the inherited CRUD (EventService.cs:16-52), each routed through the inherited SendRequestAsync helper so a back-end Result.Failure is unwrapped into a typed, displayable error via ServiceExceptionHelper before EnsureSuccessStatusCode can throw something contextless. [Rubric §3, Clean Architecture] and [Rubric §9, API & Contract Design]: the page binds to a DTO contract (EventDTO, SessionDTO, SpeakerDTO) and an interface, and the wire envelope is the uniform PagedCollectionResult<T> / CollectionResult<T> the API returns for every entity.

The list pages: derive from DataGridListPageBase<TDto>, get everything for free. Every list screen, organizer EventList, SessionList, SpeakerList, ConferenceCategoryList, QuestionList, RoomList, and the public PublicEventList / PublicSessionList / PublicSpeakerList, inherits DataGridListPageBase<TDto>. That base supplies server-side paging against MudDataGrid<T>, CancellationTokenSource lifecycle, loading state, filter/sort extraction from MudBlazor's GridState<T>, ISnackbar error surfacing, the SSR-prerender state persistence that stops the grid flashing empty on the InteractiveAuto transition, and, crucially, viewport-driven mobile rendering, switching to a card list below the sidebar-collapse breakpoint. A concrete page therefore reduces to overriding Title, GridRef, SaveFilters/RestoreFilters, and a LoadServerData delegate that calls its service's GetPagedAsync and folds in page-specific filters, EventList.razor.cs:48-66 is the canonical ~15-line example; the mobile path reuses the same fetch delegate through FetchMobilePage. [Rubric §23, Front-End Performance & Rendering] (avoiding redundant fetches/round-trips) and [Rubric §19, State Management & Data Flow] (paging/sort/filter persisted as the URL is the source of truth). This is the "compose, don't repeat" thesis of G15 made concrete across nine list pages.

The detail/create pages and the UI service contracts. The create and detail pages, EventCreate/EventDetail, SessionCreate/SessionDetail, SpeakerCreate/SpeakerDetail, and the category/question/room equivalents, are ordinary Blazor @page components that @inject one or more UI service interfaces, load on OnInitializedAsync, bind a MudBlazor form, and route every mutation back through the service layer (never a raw HTTP call), surfacing domain errors through the shared ErrorMessages helper. Each entity has its own per-feature interface, IEventUIService, ISessionUIService, ISpeakerUIService, IConferenceCategoryUIService, ICategoryItemUIService, IQuestionUIService, IRoomUIService, which extends the generic IEntityService and declares only the entity's extra verbs. The pure @page plumbing was vetted in the prior edition (organizer detail pages bind a DTO, load via the service, surface ErrorMessages); the per-type sections that follow carry it forward against current source.

Child-and-join entities: a thin POST/DELETE base. Sessions, speakers, and events own join relationships, a speaker is added to a session, a category item to a speaker, that the generic CRUD base can't model because the write carries a parent id. These get four near-identical services (EventSpeakerService, SessionSpeakerService, SessionCategoryItemService, SpeakerCategoryItemService) over the shared, purpose-built ChildEntityServiceBase, which was hoisted out of this file into MMCA.Common.UI since this chapter's first edition (MMCA.Common/Source/Presentation/MMCA.Common.UI/Services/ChildEntityServiceBase.cs:14) so every consumer module can reuse it. It exposes only PostAsync (add, ChildEntityServiceBase.cs:18) and DeleteByIdAsync (remove, which returns false on a 404 rather than throwing, ChildEntityServiceBase.cs:33) against the "APIClient" named client, again unwrapping domain failures through ServiceExceptionHelper (ChildEntityServiceBase.cs:25, 46). Each Conference join service now reduces to supplying its endpoint and adding typed AddAsync/DeleteAsync wrappers over those two verbs (ChildEntityServices.cs:13-72). Their interfaces (IEventSpeakerUIService, ISessionSpeakerUIService, ISessionCategoryItemUIService, ISpeakerCategoryItemUIService) live in IChildEntityUIService.cs. Note the team memory here: the generic delete path once sent only the child id while the controller bound a parentId from the query string, 404-ing the remove, the join services explicitly pass the parent (e.g. ?eventId={eventId} in the feedback delete) to avoid exactly that shape. [Rubric §24, Forms, Validation & UX Safety].

Display-enrichment lookups: the GetAll-vs-GetById populator gap, worked around in the UI. Because the API's list endpoints don't always populate every cross-entity navigation, several pages need a cheap id→name map to render speaker names beside a session or an event name beside a room. Three lookup services fill that role, SpeakerLookupService, EventLookupService, and CategoryItemLookupService (behind ISpeakerLookupService, IEventLookupService, ICategoryItemLookupService). Each does one large pageSize=10000 fetch and folds the result into a Dictionary of lightweight projection records, SpeakerInfo, EventInfo, CategoryItemInfo (SpeakerLookupService.cs:14-33). PublicSessionList, for instance, fetches the speaker lookup once in OnInitializedAsync and joins each session's SessionSpeakers against it to display names (PublicSessionList.razor.cs:110, 265-273). This is a deliberate client-side join over the navigation-populator (ADR-002) gap between the API's list and by-id read shapes, and the place to remember the known limitation that a GET /sessions?includeChildren can return sessionSpeakers=[], which is why the lookup-join exists rather than relying on the embedded list.

Three feature areas that go beyond CRUD. First, the speaker self-service dashboard: SpeakerDashboard is gated on the speaker_id JWT claim (read straight from AuthenticationStateProvider, SpeakerDashboard.razor.cs:56-60) and shows the linked speaker's sessions, per-session bookmark counts, and feedback, with inline profile editing (BR-214). It leans on SpeakerDashboardService (behind ISpeakerDashboardUIService), which derives from Common's AuthenticatedServiceBase so its calls carry the bearer token and the Polly retry policy. Second, organizer feedback moderation (BR-53): OrganizerEventFeedback / OrganizerSessionFeedback pages let organizers review and delete answers via OrganizerEventFeedbackService / OrganizerSessionFeedbackService (interfaces IOrganizerEventFeedbackUIService / IOrganizerSessionFeedbackUIService), organizers get the unscoped server-side view (the specification is null for organizer users) and the delete passes the parent id explicitly (OrganizerFeedbackService.cs:48). [Rubric §11, Security] (server-side authorization scoping, not a client-side hide).

Session-selection decision support, the asynchronous edge. The most behaviour-rich page is the organizer-only SessionSelectionDashboard, which renders category distribution, speaker overlap, and AI content-similarity scoring over an event's session pool via SessionSelectionService (behind ISessionSelectionUIService). Its GetDashboardAsync reads a SessionSelectionDashboardDTO; its ScoreSessionsAsync POSTs to the scoring endpoint and handles 202 Accepted explicitly: because AI scoring of every eligible session can take minutes, the API runs the ScoreEventSessionsCommand in a background scope and returns 202 immediately, so the UI service maps that to a sentinel ScoreEventSessionsResultDTO { SessionsScored = -1 } to signal "started in background" rather than a completed count (the 202 branch is SessionSelectionService.cs:42-45, inside ScoreSessionsAsync). [Rubric §6, CQRS & Event-Driven] and [Rubric §29, Resilience]: the fire-and-forget contract is honoured on both sides, and the dashboard read goes through the inherited retry policy so a transient blip self-heals.

Public vs. authenticated rendering of the same entities. A recurring [Rubric §11, Security] pattern: the same conference entity is exposed through two page families. The public family (PublicEventList/PublicEventDetail, PublicSessionList/PublicSessionDetail, PublicSpeakerList/PublicSpeakerDetail) is anonymous-readable and output-cached at the API; the organizer family exposes edit controls behind role gating. PublicSessionList shows the nuance well: it is read-only for anonymous users (BR-43), but an authenticated user gets inline bookmark stars and a "My Schedule" toggle, wired through the optional ISessionBookmarkUIService (injected nullable because the Engagement module may be disabled, PublicSessionList.razor.cs:34), a clean illustration of the modular monolith's cross-module-via-interface discipline degrading gracefully when a module is absent. [Rubric §7, Microservices Readiness].

Routes and navigation. All paths are centralized in ConferenceRoutePaths, a static catalogue of literal routes and id-parameterized builder methods (EventDetails(id), PublicSessionDetails(id), EventFeedbackOrganizer(id), …) typed against the module's identifier aliases, pages navigate with NavigationManager.NavigateTo(ConferenceRoutePaths.EventDetails(id)) rather than hand-building URL strings, so a route change happens in one file (ConferenceRoutePaths.cs:8-48). [Rubric §25, Navigation, Routing & Information Architecture]. User-facing strings are not inline English: every page resolves its labels and snackbar messages through an injected IStringLocalizer (the L["..."] calls in each code-behind, for example EventList.razor.cs:17-18 for the title and EventList.razor.cs:82 for a delete toast, or SpeakerDashboard.razor.cs:50-51, 102) over co-located .resx resources, so the conference surface follows the framework's two-locale (en-US default plus Spanish) i18n. [Rubric §27, Internationalization & Localization] assesses externalized strings and culture-aware formatting; this area now embodies it under ADR-027, which superseded the old single-locale ADR-011 (primer §6).

How it all plugs into the shell. Two registration types wire the area in. ConferenceUIModule implements Common's IUIModule (the front-end counterpart of the IModule back-end contract): it declares the module's NavItem list, whose labels are ADR-027 resource keys (Nav.Events, Nav.Dashboard, …) each carrying a TitleResource so the shared NavMenu localizes them at render time against the co-located ConferenceUIModule.resx pair: public entries for everyone, a speaker_id-claim-gated "Dashboard" (RequiredClaim: "speaker_id", ConferenceUIModule.cs:26), and an Organizer-role-gated admin group (Events/Sessions/Speakers/Categories/Questions/Rooms/Session Selection, ConferenceUIModule.cs:29-35), and it exposes its assembly so the host can discover the Razor routes (ConferenceUIModule.cs:18-38). The companion DependencyInjection extension AddConferenceUI() (a C# extension(IServiceCollection) member, primer §4) is the one call a host makes: it Scrutor-scans the assembly to register every IEntityService<,> implementation as scoped, then explicitly registers the child-entity, dashboard, feedback, session-selection, and lookup services, and finally registers ConferenceUIModule as a singleton IUIModule so the shell folds its nav items and routes in with no edit to the shell itself (DependencyInjection.cs:19-53). [Rubric §1, SOLID] (Open/Closed) and [Rubric §18, UI Architecture]. Read the per-type sections that follow for the mechanics of each page and service; the bUnit/Playwright tests that exercise this library live in the testing chapter (G25).

ConferenceRoutePaths

MMCA.ADC.Conference.UI · MMCA.ADC.Conference.UI · MMCA.ADC/Source/Modules/Conference/MMCA.ADC.Conference.UI/ConferenceRoutePaths.cs:6 · Level 0 · class (static)

  • What it is: the single source of truth for every Conference UI page URL: route-path constants plus typed …Details(id) factory methods, covering both the organizer-management routes (admin) and the public attendee-facing routes.
  • Depends on: no first-party types. The detail-factory methods take identifier-type aliases (EventIdentifierType, SessionIdentifierType, SpeakerIdentifierType, ConferenceCategoryIdentifierType, QuestionIdentifierType, RoomIdentifierType, each a solution-wide global using … = System.Guid;/= int; alias, see primer §2 "Strongly-typed identifier aliases").
  • Concept introduced, centralized route registry. [Rubric §25, Navigation, Routing & Information Architecture] assesses whether navigation is coherent, role-aware, and free of scattered magic strings; this class embodies it by hoisting every URL into one place. Two route namespaces encode the module's dual audience: organizers navigate to /events, /sessions, /speakers (:8-30); attendees navigate to the /conference/* prefix (PublicEvents, PublicSessions, PublicSpeakers, :33-38). Keeping these together means an @page directive and a NavigationManager.NavigateTo call can never drift apart, and the public/admin split is visible at a glance.
  • Walkthrough: the file is grouped by concern. Admin CRUD routes come in a regular triple per entity, a list (Events, :8), a create (EventCreate, :9), and a static detail factory that interpolates the id (EventDetails(EventIdentifierType id) => $"/events/{id}", :10); the same shape repeats for sessions (:12-14), speakers (:16-18), conference categories (:20-22), questions (:24-26), and rooms (:28-30). Public attendee routes mirror a subset under /conference/* (:33-38). The remainder are single-purpose: the claim-gated SpeakerDashboard (:41), organizer feedback-moderation routes EventFeedbackOrganizer(id)/SessionFeedbackOrganizer(id) (:44-45), and the SessionSelectionDashboard decision-support route (:48). List/create paths are static readonly string fields; detail paths are methods because they embed a runtime id.
  • Why it's built this way: if the admin prefix changes (e.g. /events → /admin/events) you edit one constant and every consumer follows, with no grep-and-replace and no risk of a stray hardcoded URL surviving the refactor.
  • Where it's used: every Conference UI Razor page's @page directive, every NavigationManager.NavigateTo call in Conference components, and ConferenceUIModule's NavItems list, which reads these constants directly rather than re-typing the strings.

ConferenceUIModule

MMCA.ADC.Conference.UI · MMCA.ADC.Conference.UI · MMCA.ADC/Source/Modules/Conference/MMCA.ADC.Conference.UI/ConferenceUIModule.cs:14 · Level 3 · class (sealed)

  • What it is: the Conference module's UI descriptor: it contributes the navigation items for conference management (Events, Sessions, Speakers, Categories, Questions, Rooms, Session Selection) and the public attendee items, and exposes its assembly so the shell can discover the module's routable Blazor components.

  • Concept introduced, the modular-UI descriptor, the front-end analogue of IModule. [Rubric §11, Security]: navigation is gated by RequiredClaim (e.g. "speaker_id" for the speaker Dashboard, :24) and role (RoleNames.Organizer for admin items), but this is UX-level hiding only; the server still enforces authorization on every request, so a hidden link is not a security boundary.

  • Why it's built this way: mirroring the backend IModule pattern on the UI side keeps the app extensible: adding a module adds its menu and its pages without touching the shell. The sealed keyword matches the codebase's default for non-base classes (analyzer-enforced).
  • Where it's used: registered as a singleton IUIModule by the Conference UI DependencyInjection (AddConferenceUI); discovered and aggregated by the Common UI shell's navigation builder (which sorts and renders items by Section/Group/role) and by the Blazor router's assembly discovery.

Externals: Microsoft.Extensions.DependencyInjection and Scrutor (services.Scan(...)).
  • Concept introduced: the extension(IServiceCollection) DI idiom and Scrutor assembly scanning are both taught in primer §3/§4 and first shown in tier-00; this is the UI-layer instance of the same pattern. [Rubric §1, SOLID] (DIP): every page depends on a UI interface, never a concrete service, and this method is the only place the concrete-to-interface bindings live. [Rubric §5, Vertical Slice]: registration sits inside the Conference UI library beside the features it serves, so enabling Conference UI is a single AddConferenceUI() call rather than edits scattered through the host's Program.cs. [Rubric §16, Maintainability]: the Scrutor scan means a new entity's IEntityService implementation is registered automatically, no DI edit needed, while the bespoke services are listed explicitly because each binds a single named interface.
  • Walkthrough: AddConferenceUI() (:19) runs four phases. (1) Convention scan (:22-26): services.Scan(...) from Scrutor walks the Conference UI assembly (FromAssemblyOf<ConferenceUIModule>()), finds every class AssignableTo(typeof(IEntityService<,>)), registers each AsImplementedInterfaces() with WithScopedLifetime(), this picks up the generated per-entity CRUD UI services in bulk. (2) Explicit scoped bindings for services that don't fit the open-generic convention: the four child-entity managers (:29-32), the speaker dashboard (:35), the two BR-53 organizer feedback-moderation services (:38-39), and the session-selection decision-support service (:42). (3) Cross-module lookups (:45-47), SpeakerLookupService, EventLookupService, CategoryItemLookupService, used by dropdowns/pickers that need data the page doesn't own.   • Why it's built this way: co-locating registration with the slice keeps the host thin and the module self-describing; mixing a Scrutor convention scan (for the repetitive entity services) with explicit AddScoped calls (for the one-off services) is the codebase's standard trade-off between zero-ceremony bulk registration and clarity for the special cases.
  • Where it's used: called from the UI host composition (the Web host's Program.cs and the MAUI MauiProgram, tier-04/tier-06), gated by the module-enabled check so Conference UI is only wired when the Conference module is on.
  • Caveats / not-in-source: the prior tier edition described this as merely "registers all UI service implementations against their interfaces"; that was accurate but incomplete, the current source additionally wires the Scrutor scan, the dashboard/feedback/session-selection services, and the three cross-module lookups, all verified above against DependencyInjection.cs:11-54.

The CategoryTitle is pre-joined by the lookup service (it stitches categories to items in one pass) so each item already knows its category's human label, no per-row N+1 lookup in the markup.
  • Where it's used: produced by CategoryItemLookupService.GetAllAsync as the value type of an IReadOnlyDictionary<CategoryItemIdentifierType, CategoryItemInfo>; consumed by Session and Speaker pages to label category tags.
