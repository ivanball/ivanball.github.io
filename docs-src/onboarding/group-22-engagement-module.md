# 22. ADC Engagement Module (Session Bookmarks)

**What this group covers.** Engagement is the ADC bounded context that lets an authenticated attendee build a personal schedule and give feedback. This chapter walks the part of that context that ships as the module's steady core: the **session-bookmark aggregate** and everything wrapped around it (use cases, persistence, REST API, cross-service gRPC contracts and adapters, and the feedback pages in the module's Blazor UI). The module also hosts the conference-day live layer (LivePolls and SessionQuestions), which is a large enough surface that it is documented on its own; here the live layer appears only where it touches the shared plumbing, for example the feature flags in [EngagementFeatures](#engagementfeatures) and the permission grants in [EngagementPermissions](#engagementpermissions). Engagement is one of four extracted service hosts in MMCA.ADC, so most of this chapter is a worked example of the framework patterns taught earlier (Result, CQRS, domain events, database-per-service, gRPC extraction) applied to one small aggregate. Read [Group 02](group-02-domain-building-blocks.md) for the entity primitives and [primer §2](00-primer.md#2-architectural-styles-this-codebase-commits-to) for the styles before you go deep here.

**The aggregate is deliberately tiny.** [UserSessionBookmark](#usersessionbookmark) (`MMCA.ADC.Engagement.Domain/UserSessionBookmarks/UserSessionBookmark.cs:17`) is a sealed [AuditableAggregateRootEntity<TIdentifierType>](group-02-domain-building-blocks.md#auditableaggregaterootentitytidentifiertype) that carries only two scalar foreign keys, `UserId` and `SessionId` (`UserSessionBookmark.cs:20`, `:23`), both stored as plain columns rather than cross-database relationships because the user lives in the Identity database and the session in the Conference database (ADR-006, database-per-service). A private constructor plus a static `Create` factory returning [Result<UserSessionBookmark>](group-01-result-error-handling.md#result) (`UserSessionBookmark.cs:41`) is the only way in: the factory runs `Result.Combine` over [UserSessionBookmarkInvariants](#usersessionbookmarkinvariants) (`UserSessionBookmark.cs:45`), which just delegate to `CommonInvariants.EnsureIdIsNotDefault` (`UserSessionBookmarkInvariants.cs:12`), so an invalid bookmark cannot be constructed. Interesting design decision: instead of separate Created and Deleted events, the aggregate raises one [UserSessionBookmarkChanged](#usersessionbookmarkchanged) event carrying a [DomainEntityState](group-02-domain-building-blocks.md#domainentitystate) discriminator (`UserSessionBookmark.cs:56`, `:72`, `:87`), documented in-code as BR-60. A `Reactivate()` method (`UserSessionBookmark.cs:67`) undeletes a soft-deleted row and re-raises the Added event, which is the domain half of the re-bookmarking story (BR-135).

**The write path.** [BookmarksController](#bookmarkscontroller) (`MMCA.ADC.Engagement.API/Controllers/BookmarksController.cs:32`) is the thin REST edge: every action injects a handler and delegates, mapping [Result](group-01-result-error-handling.md#result) failures to RFC 9457 Problem Details through `ApiControllerBase.HandleFailure`. A POST routes into [CreateBookmarkHandler](#createbookmarkhandler) (`MMCA.ADC.Engagement.Application/UserSessionBookmarks/UseCases/Create/CreateBookmarkHandler.cs:17`), which is the most instructive type in the module because it shows how a slice coordinates a cross-module call, a uniqueness check, and a create-or-reactivate decision. It first asks Conference's [ISessionBookmarkValidationService](group-17-conference-domain.md#isessionbookmarkvalidationservice) whether the session may be bookmarked (`CreateBookmarkHandler.cs:30`, enforcing BR-49/BR-91 across the service boundary), then checks for an existing active bookmark and returns `Error.Conflict` on a duplicate (`CreateBookmarkHandler.cs:37`, BR-21), then loads any soft-deleted row with `ignoreQueryFilters: true` (`CreateBookmarkHandler.cs:50`) and hands the create-or-reactivate choice to the pure [BookmarkManagementDomainService](#bookmarkmanagementdomainservice) behind the [IBookmarkManagementDomainService](#ibookmarkmanagementdomainservice) interface (`CreateBookmarkHandler.cs:59`). That domain service (`BookmarkManagementDomainService.cs:13`) either calls `Reactivate()` on the found row or `UserSessionBookmark.Create(...)`, so BR-135 lives in the domain, not smeared across the handler. Persistence and the domain-event dispatch happen at `unitOfWork.SaveChangesAsync` (`CreateBookmarkHandler.cs:69`), then the entity is projected to a [UserSessionBookmarkDTO](#usersessionbookmarkdto) through [UserSessionBookmarkDTOMapper](#usersessionbookmarkdtomapper).

**The read path and its authorization.** Two query slices back the personal-schedule views: [GetUserBookmarksHandler](#getuserbookmarkshandler) (`GetUserBookmarksHandler.cs:17`) returns a paged, `CreatedOn`-descending [PagedCollectionResult<T>](group-01-result-error-handling.md#pagedcollectionresultt) of DTOs with server-side count, ordering, and OFFSET/FETCH pagination capped at 500 (`GetUserBookmarksHandler.cs:29`, `:51`, `:54`), optionally narrowed to one event by resolving its session ids through Conference (`GetUserBookmarksHandler.cs:39`, BR-58); [GetBookmarkedSessionIdsHandler](#getbookmarkedsessionidshandler) returns the lightweight session-id set the session grid uses to paint filled or empty bookmark icons. Both list endpoints are authorization-scoped by the shared [OwnerOrAdminFilter](group-08-auth.md#owneroradminfilter) applied as a `[ServiceFilter]` (`BookmarksController.cs:61`, `:82`): the caller's `userId` query argument must equal their `user_id` claim unless they hold the Organizer bypass role. That vocabulary (owner claim, bypass role, parameter name) is not hard-coded in the filter; it is configured for ADC in `AddModuleEngagementAPI` through `OwnerOrAdminFilterOptions` (`MMCA.ADC.Engagement.API/DependencyInjection.cs:44`, drift D9, ADR-033). The DELETE keeps a separate inline ownership check that returns 404 rather than 403 so it never leaks whether a bookmark exists (`BookmarksController.cs:106`), then runs the generic [DeleteEntityCommand<TEntity, TIdentifierType>](group-05-cqrs-pipeline.md#deleteentitycommandtentity-tidentifiertype) which soft-deletes.

**Persistence.** Engagement's [ModuleApplicationDbContext](#moduleapplicationdbcontext) (`MMCA.ADC.Engagement.Infrastructure/Persistence/DbContexts/ModuleApplicationDbContext.cs:16`) is an abstract context that declares the module's `DbSet`s (bookmarks plus the live-layer entities) and inherits auditing, soft-delete query filters, and domain-event dispatch from the common [ApplicationDbContext](group-07-persistence-ef-core.md#applicationdbcontext); the concrete per-engine class (SQLServerDbContext today) sits under it, one instance per physical database (ADR-006). The bookmark table shape is pinned by [UserSessionBookmarkConfiguration](#usersessionbookmarkconfiguration), and the module owns its own `ADC_Engagement` database with its own `dbo.OutboxMessages`, so the [UserSessionBookmarkChanged](#usersessionbookmarkchanged) events serialized during `SaveChangesAsync` never race another service's outbox rows (ADR-003).

**How Engagement talks to its neighbors.** Engagement is both a client and a server on the gRPC mesh (ADR-007, ADR-008). As a **server**, its in-process [BookmarkCountService](#bookmarkcountservice) (`BookmarkCountService.cs:11`) answers "how many people bookmarked this session" for Conference's session cards; that interface is exposed over the wire by [BookmarkCountsGrpcService](#bookmarkcountsgrpcservice) (`BookmarkCountsGrpcService.cs:12`), and Conference consumes it through the matching client-side [BookmarkCountServiceGrpcAdapter](#bookmarkcountservicegrpcadapter) (`BookmarkCountServiceGrpcAdapter.cs:14`), which re-implements the same C# [IBookmarkCountService](#ibookmarkcountservice) interface so the calling code never knows it crossed a process. The identical bridge pattern carries the privacy export: [UserEngagementExportService](#userengagementexportservice) (`UserEngagementExportService.cs:14`) projects a user's bookmarks and submitted questions to id-and-date DTOs (soft-deleted rows excluded by the query filter), served by [UserEngagementExportGrpcService](#userengagementexportgrpcservice) (`UserEngagementExportGrpcService.cs:16`) and consumed by Identity through [UserEngagementExportServiceGrpcAdapter](#userengagementexportservicegrpcadapter) so Identity can assemble the cross-service data-subject export (PRIVACY.md §7, ADR-005). As a **client**, the create and query handlers call Conference's [ISessionBookmarkValidationService](group-17-conference-domain.md#isessionbookmarkvalidationservice) the same way.

**Module wiring and feature gating.** [EngagementModule](#engagementmodule) (`EngagementModule.cs:14`) is the [IModule](group-14-module-system-composition.md#imodule) entry point discovered by `ModuleLoader` and registered in topological order; it declares a dependency on Conference and `RequiresDependencies => true` (`EngagementModule.cs:20`, `:23`). When the module is disabled in a host, `RegisterDisabledStubs` swaps in [DisabledBookmarkCountService](#disabledbookmarkcountservice) and [DisabledUserEngagementExportService](#disableduserengagementexportservice) (`EngagementModule.cs:30`) so a Conference-only or Identity-only host still resolves those interfaces without a live Engagement backend. Every attendee-facing endpoint is behind a `[FeatureGate]` keyed to [EngagementFeatures](#engagementfeatures) (`EngagementFeatures.cs:8`), so the whole bookmark surface can be turned off and returns 404, and the live-layer endpoints are additionally guarded by role-to-permission grants declared in `AddModuleEngagementAPI` from [EngagementPermissions](#engagementpermissions) (`DependencyInjection.cs:51`).

**The feedback UI.** The module's Blazor UI contributes two feedback pages, [EventFeedback](#eventfeedback) (`EventFeedback.razor.cs:16`) and [SessionFeedback](#sessionfeedback) (`SessionFeedback.razor.cs:16`). They render a dynamic question form (Rating, Text, Email) whose definitions and answers actually belong to the Conference context: the pages inject Conference-facing UI services, [IQuestionLookupService](#iquestionlookupservice), [IEventFeedbackUIService](#ieventfeedbackuiservice), and [ISessionFeedbackUIService](#isessionfeedbackuiservice) (`IFeedbackUIService.cs:10`, `:21`, `:43`), and submit one answer per question using a POST-as-upsert convention (BR-107) so re-visiting the page edits existing answers rather than duplicating them. The page tracks per-question [AnswerState](#answerstate) and existing answer ids to switch its button between Submit and Update (`EventFeedback.razor.cs:33`). These pages sit in Engagement.UI for navigation reasons; the persisted answers are Conference-owned.

**Reminders and navigation.** For the MAUI client, [SessionReminderPlanner](#sessionreminderplanner) (`SessionReminderPlanner.cs:29`) is a pure, clock-injected planner (ADR-042 Wave 2) that turns bookmarked sessions into [SessionReminder](#sessionreminder) records (`SessionReminderPlanner.cs:13`): it converts each session's event-local wall-clock start to UTC, handling spring-forward gaps and fall-back ambiguity explicitly (`SessionReminderPlanner.cs:91`), skips past sessions, and derives a stable notification id from the session id so rescheduling replaces rather than duplicates a reminder (`SessionReminderPlanner.cs:37`). The [EngagementUIModule](#engagementuimodule) descriptor (`EngagementUIModule.cs:14`) registers the module's Blazor assembly, its Happening Now nav item (a resource key, ADR-027), and an invisible live-channel listener component.

**Rubric lens.** This module is a compact end-to-end illustration of most of Part A. The aggregate and its invariants are the [Rubric §4, Domain-Driven Design] story (rules enforced inside a factory, a single state-carrying domain event); the controller-handler-domain-service split is [Rubric §5, Vertical Slice] and [Rubric §6, CQRS & Event-Driven]; the gRPC adapters and disabled stubs are [Rubric §7, Microservices Readiness] and [Rubric §9, API & Contract Design]; the per-service database and its own outbox are [Rubric §8, Data Architecture]; the `OwnerOrAdminFilter` scoping and the 404-not-403 delete are [Rubric §11, Security]; and the export projection plus soft-delete exclusion are [Rubric §30, Compliance and Privacy]. The feedback pages and reminder planner extend the story into Part B ([Rubric §18, UI Architecture] and [Rubric §24, Forms/Validation/UX Safety]). Relevant ADRs, in the order they surface above: ADR-003 (outbox), ADR-005 (soft-delete vs erasure), ADR-006 (database-per-service), ADR-007 and ADR-008 (gRPC extraction and topology), ADR-033 (owner-or-admin filter), and ADR-042 (MAUI reminder dispatch).
